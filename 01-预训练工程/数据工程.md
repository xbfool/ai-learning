# 数据工程

> 数据质量决定模型质量上限。本文梳理从原始网页到高质量预训练 Token 的完整流水线。

## 一、数据处理流水线

```
原始网页 (Common Crawl)
  │
  ├── 1. URL 过滤        → 移除低质量域名
  ├── 2. 文本提取        → HTML → 纯文本 (trafilatura, resiliparse)
  ├── 3. 语言识别        → fastText 语言分类器
  ├── 4. 去重 (Dedup)    → MinHash / SimHash / Exact Dedup
  ├── 5. 质量过滤        → 基于模型的分类器 (perplexity, 教育性)
  ├── 6. PII 清洗        → 移除个人身份信息
  ├── 7. 安全过滤        → 移除有毒/违规内容
  └── 8. 数据混合        → 按比例混合不同数据源
      │
      → 高质量预训练 Token (5T-30T)
```

## 二、核心开源数据集

### FineWeb (HuggingFace, 2024)
- **规模**：15T tokens
- **论文**：[arXiv:2406.17557](https://arxiv.org/abs/2406.17557)
- **核心创新**：
  - 教育内容分类器 (educational quality classifier) 过滤高质量数据
  - 基于 MinHash 的大规模去重
  - FineWeb-Edu 子集：专注高教育价值内容
- **关键发现**：数据质量 > 数据数量；经过良好过滤的 1.3T tokens 可以匹配未过滤的 15T tokens

### DCLM (DataComp for Language Models, 2024)
- **论文**：[arXiv:2406.11794](https://arxiv.org/abs/2406.11794)
- **核心创新**：
  - 开放数据筛选竞赛/基准
  - 基于 fastText 的质量分类器，用高质量参考数据（如 Wikipedia、教科书）训练
  - 证明 **model-based filtering** 是最有效的数据筛选策略
- **关键发现**：用小型分类器（fastText）做数据筛选，性价比远高于用大模型

### Dolma 3 (AI2, 2025)
- 用于 OLMo 3 训练
- 起始规模：~9.3T tokens 的网络抓取
- 最终混合：~5.9T tokens，重点加码代码和数学
- **完全公开**：数据处理代码 + 原始数据全部开源

## 三、合成数据 (Synthetic Data)

### 动机
- 高质量自然数据越来越稀缺
- 合成数据可以针对性补充特定领域（数学推理、代码、指令遵循）

### 方法
1. **Rephrase/Reformulate**：用 LLM 改写已有数据（Kimi K2 采用此策略）
2. **Self-Instruct**：LLM 自动生成指令-响应对
3. **Evol-Instruct**：逐步提高指令复杂度（WizardLM）
4. **数学/代码验证**：生成问题 + 用规则验证答案正确性
5. **蒸馏**：从大模型生成数据训练小模型（DeepSeek-R1 蒸馏到 1.5B-70B）

### Kimi K2 的合成数据策略
- 设计了精细的 **rephrasing pipeline**
- 在不引入显著过拟合的情况下增加高质量 token 数量
- 显著提升了 token 利用效率

## 四、数据混合 (Data Mixing)

### 典型混合比例
| 数据类型 | 占比 | 来源 |
|---------|------|------|
| 网页文本 | 50-70% | Common Crawl 过滤后 |
| 代码 | 10-20% | GitHub, Stack |
| 数学/科学 | 5-10% | arXiv, 教科书, 合成 |
| 书籍 | 5-10% | Project Gutenberg 等 |
| 百科 | 3-5% | Wikipedia |
| 对话/论坛 | 2-5% | Reddit, StackOverflow |

### 多阶段训练中的数据混合
- **阶段 1 (主训练)**：通用混合，大规模训练
- **阶段 2 (中训练/退火)**：提高高质量数据占比（代码、数学、推理）
- **阶段 3 (长上下文)**：长文档数据占主导
- OLMo 3 明确采用三阶段策略：主训练 → 数学/代码强化 → 长上下文扩展

## 五、关键经验

1. **去重极其重要**：重复数据会导致 memorization，降低泛化能力
2. **质量过滤决定上限**：一个好的数据分类器价值 > 10x 更多的数据
3. **数据混合是门艺术**：需要大量消融实验确定最优比例
4. **合成数据是双刃剑**：能补充稀缺领域，但过多会导致 model collapse
5. **数据溯源和版权**：AI2 坚持完全开放数据链条，是行业标杆

### 6.1 FineWeb：大规模蒸馏网络以获取最优文本数据

**论文**：*The FineWeb Datasets: Decanting the Web for the Finest Text Data at Scale* (NeurIPS 2024 Datasets & Benchmarks Track)
**机构**：HuggingFace
**链接**：[arXiv:2406.17557](https://arxiv.org/abs/2406.17557)

#### 6.1.1 论文定位与核心贡献

FineWeb 是目前规模最大的完全公开的高质量网页预训练数据集，包含 **15 万亿 (15T) tokens**，来源于 96 个 Common Crawl 快照。论文的核心贡献不仅在于数据集本身，更在于**对数据处理流水线每一步进行了系统性的消融实验 (ablation study)**，为社区提供了可复现的最佳实践。

FineWeb 在多项基准测试（包括 ARC、HellaSwag、MMLU、OpenBookQA、PIQA、Winogrande）的聚合得分上，显著优于此前所有开源数据集，包括 C4、RefinedWeb、Dolma v1.6、The Pile、SlimPajama 和 RedPajama2。

#### 6.1.2 文本提取：trafilatura vs resiliparse

数据工程的第一步是从 HTML 中提取纯文本。Common Crawl 默认提供的 WET 文件（纯文本抽取）质量较差，保留了大量模板内容（boilerplate）和导航菜单。FineWeb 论文对比了两种主流提取工具：

| 工具 | 特点 | 保留率 | 聚合得分 |
|------|------|--------|---------|
| **trafilatura** | 元数据感知提取，对多种页面类型鲁棒性强 | 43.25% | 0.4429 |
| **resiliparse** | 速度优先，对畸形 HTML 容忍度高，比 trafilatura 快数倍 | 19.44% | 0.4402 |
| jusText | 传统方法 | 49.96% | 0.4474 |
| WET (默认) | Common Crawl 默认提取 | - | 0.4057 |

**关键发现**：trafilatura 和 resiliparse 在下游性能上非常接近，但 resiliparse 速度快数倍。FineWeb 最终选择 trafilatura 是因为其提取质量略优，且 RefinedWeb 也采用了相同的工具。DCLM 则选择了 resiliparse，以追求更高的处理速度。

**核心教训**：使用任何专用提取器都远优于直接使用 Common Crawl 默认的 WET 文件。WET 文件保留了过多的模板文本（导航栏、页脚、广告残留等），对模型训练产生显著的负面影响。

#### 6.1.3 URL 级别去重

FineWeb 的第一步去重在 URL 层面进行：

- **方法**：对 URL 进行小写化 (lowercase) 归一化后，每个唯一 URL 只保留一个文档
- **去除比例**：移除了 **71.5%** 的 tokens，从原始数据缩减到约 5.6T tokens
- **原因**：Common Crawl 的多次抓取导致同一网页被反复收录，URL 去重是成本最低的第一道屏障

#### 6.1.4 MinHash 去重：参数选择与消融

FineWeb 采用基于 MinHash 的近似去重，这是处理 Web 级别数据的标准方法。其具体参数设定：

**MinHash 参数**：
- **n-gram 大小**：5-grams（将文档分割为 5 个连续 token 的片段）
- **哈希函数数量**：112 个哈希函数
- **分桶策略**：112 个 minhash 签名被分成 **14 个桶 (bands)**，每个桶 **8 个哈希 (rows)**
- **匹配规则**：两个文档在至少一个桶中所有 8 个哈希值完全相同，则被判定为近似重复

**不同相似度下的匹配概率**：

| 文档 n-gram 相似度 | 被识别为重复的概率 |
|-------------------|------------------|
| 0.70 | 56% |
| 0.75 | 77% |
| 0.80 | 92% |
| 0.85 | 98.8% |

这意味着系统以高概率捕获相似度 >= 75% 的文档对。

**去重粒度消融实验**：

FineWeb 对比了多种去重策略：

| 策略 | 移除 tokens 比例 | 剩余 tokens | 下游性能 |
|------|-----------------|-------------|---------|
| 单快照 MinHash 去重（最终方案） | 基线 | ~15T | 最优 |
| URL 去重 | 71.5% | 5.6T | 次优 |
| 行级去重（移除所有重复行，仅保留 1 次） | 77.8% | 4.4T | 低于基线 |
| 行级去重 + 10 词阈值 + 短文档过滤 | 85% | 2.9T | 低于基线 |
| 3 行连续重复段落去重 | 80.9% | 3.7T | 低于基线 |

**关键发现**：在单快照 MinHash 去重之上，进一步的跨快照全局去重或行级去重**并没有带来性能提升**，反而会因为过度移除数据而降低性能。这个发现与直觉相反——很多人假设更激进的去重一定更好。FineWeb 的实验表明，**单快照级别的 MinHash 去重已经是足够好的策略**。

#### 6.1.5 启发式过滤规则：C4 规则与自定义规则

FineWeb 系统性地测试了多种启发式过滤规则：

**C4 过滤规则**（来自 Google 的 C4 数据集）：
- 移除没有终止标点符号的行
- 移除包含 "javascript" 的行
- 移除包含 "terms-of-use" / "cookie policy" 的行
- 移除过短的文档
- 移除包含 "lorem ipsum" 的文档
- 移除包含花括号 `{` 的文档

**C4 规则消融结果**：
- 花括号过滤和词长过滤的提升较小，分别只移除 2.8% 和 4.3% 的 tokens
- **终止标点过滤**贡献最大，但移除了约 30% 的 tokens
- 组合使用所有过滤器的性能优于任何单个过滤器

**Gopher 规则**（来自 DeepMind 的 MassiveText）：
- 基于文档内容的启发式规则：词数、停用词出现频率、字符重复比例等
- 使用 Google SafeSearch 移除露骨内容
- 配合 MinHash 去重

**FineWeb 自定义启发式过滤器**（论文的创新部分）：

FineWeb 团队通过消融实验设计了三条最有效的自定义规则：

| 规则 | 阈值 | 移除 tokens 比例 |
|------|------|-----------------|
| 以标点结尾的行的比例过低 | <= 0.12 | 10.14% |
| 重复行中的字符占比过高 | >= 0.1 | 12.47% |
| 短行（< 30 字符）比例过高 | >= 0.67 | 3.73% |

这三条规则合计移除约 25% 的 tokens，但带来了显著的下游性能提升。

#### 6.1.6 教育质量分类器 (Educational Quality Classifier)：FineWeb-Edu 的核心

FineWeb 最重要的创新是 **FineWeb-Edu** 子集——通过一个专门的教育质量分类器过滤出具有高教育价值的内容。

**分类器训练流程**：

```
Step 1: 标注数据生成
  └── 使用 Llama-3-70B-Instruct 对 ~460,000 个 FineWeb 样本打分
  └── 评分范围：0-5（0 = 无教育价值, 5 = 极高教育价值）
  └── 特殊设计：Prompt 侧重于基础教育和中等教育水平的知识
      （避免 LLM 过度偏好高度技术性内容如 arXiv 摘要）

Step 2: 分类器模型训练
  └── 基础模型：Snowflake-arctic-embed（BERT-like 嵌入模型）
  └── 架构：嵌入模型 + 分类头 + 单个回归输出
  └── 训练配置：
      - 在 450,000 条 Llama-3 标注上训练 20 epochs
      - 学习率：3e-4
      - 冻结嵌入层和编码器层（只训练分类头）
  └── 验证集 F1 分数：82%（以分数 3 为二分类阈值）

Step 3: 大规模推理
  └── 对 FineWeb 全部 15T tokens 进行分类打分
  └── 计算成本：~6,000 H100 GPU 小时
  └── 输出：每个文档一个 0-5 的教育质量分数
```

**分数阈值选择**：

| 阈值 | 保留 tokens | 知识/推理基准 | HellaSwag/PIQA |
|------|-----------|-------------|----------------|
| 2 | 更多 | 中等提升 | 无显著下降 |
| **3（最终选择）** | **1.3T (~8%)** | **显著提升** | **轻微下降** |
| 4 | 更少 | 更高提升 | 显著下降 |
| 5 | 极少 | 最高 | 严重下降 |

**最终决策**：阈值 3 提供了最佳的平衡——在知识密集和推理密集基准上获得最大提升，同时对常识推理基准（HellaSwag、PIQA）的影响可控。使用阈值 3 意味着 **移除了 92% 的数据**，只保留 1.3T tokens 的高教育质量内容。

**FineWeb-Edu 的基准提升**：

| 基准测试 | FineWeb (15T) | FineWeb-Edu (1.3T) | 相对提升 |
|---------|-------------|-------------------|---------|
| MMLU | ~33% | ~37% | +12% |
| ARC | ~46% | ~57% | +24% |
| HellaSwag | 基线 | 轻微下降 | -2~3% |

**核心洞察**：这个结果有力地证明了 **"数据质量 > 数据数量"** 的核心假设。经过精心过滤的 1.3T tokens 在知识和推理任务上显著优于未过滤的 15T tokens，尽管数据量缩减了超过 10 倍。

#### 6.1.7 FineWeb 的方法论价值

FineWeb 论文的最大价值不仅在于数据集本身，更在于其 **"数据消融 (data ablation)"** 方法论：

1. **每一步都有消融**：URL 过滤、文本提取、去重策略、启发式规则、质量分类器——每一步都通过训练小模型并在下游任务上评估来验证其效果
2. **组合效应分析**：不仅测试单个组件，还测试组件组合的效果
3. **完全可复现**：所有代码开源在 [datatrove](https://github.com/huggingface/datatrove) 仓库中
4. **反直觉发现**：更激进的去重不一定更好；教育质量过滤比任何启发式规则都更有效

### 6.2 DCLM (DataComp for Language Models)：数据筛选竞赛与基准

**论文**：*DataComp-LM: In search of the next generation of training sets for language models* (NeurIPS 2024 Datasets & Benchmarks Track)
**机构**：多机构联合（包括 Apple、华盛顿大学等）
**链接**：[arXiv:2406.11794](https://arxiv.org/abs/2406.11794)

#### 6.2.1 论文定位与核心贡献

DCLM 的核心创新是提供了一个**标准化的数据筛选基准 (benchmark)**，类似于 ImageNet 对视觉模型的意义。它回答的核心问题是：**在固定的计算预算和模型架构下，什么样的数据筛选策略能训练出最好的语言模型？**

DCLM 提供了三大组件：
1. **标准化语料库**：从 Common Crawl 提取的 **240T tokens** 原始语料池
2. **标准化训练方案**：基于 OpenLM 框架的预训练配方（涵盖 412M 到 7B 参数规模）
3. **标准化评估**：包含 **53 项下游评估任务** 的评测套件

#### 6.2.2 基准设计：两种竞赛赛道

DCLM 设计了两种参与方式：

**Filtering Track（过滤赛道）**：
- 给定固定的 240T token 语料池
- 参与者设计数据筛选策略，从中选择子集
- 使用标准化的训练配方训练模型
- 在 53 项下游任务上评估

**Mixing Track（混合赛道）**：
- 参与者可以引入外部数据源
- 与过滤后的数据混合
- 评估最终混合方案的效果

**多规模评估**：参与者可以在多个模型规模上验证策略：

| 规模 | 参数量 | 训练 tokens |
|------|--------|-----------|
| 400M-1x | 412M | 基准量 |
| 1B-1x | ~1B | 基准量 |
| 3B-1x | ~3B | 基准量 |
| 7B-1x | 7B | 2.6T |

**跨规模迁移性**：论文验证了较小规模的实验结果能可靠地预测大规模的表现：
- 400M → 7B 的 Pearson 相关系数：r = 0.838
- 1B → 7B 的 Pearson 相关系数：r = 0.956
- 3B → 7B 的 Pearson 相关系数：r = 0.982

这意味着研究者可以在 400M 规模上低成本地验证数据策略，然后将最优策略迁移到 7B 规模。

#### 6.2.3 Model-Based Quality Filtering：核心方法

DCLM 最重要的发现是 **model-based filtering（基于模型的质量过滤）** 是所有策略中最有效的。

**fastText 分类器的训练方法**：

```
正样本 (高质量)：
  ├── OpenHermes 2.5 (OH-2.5) 指令数据
  └── Reddit r/ExplainLikeImFive (ELI5) 高质量问答

负样本 (低质量)：
  └── Common Crawl 原始网页文本（随机采样）

分类器：
  └── fastText 二分类模型
  └── 训练目标：区分"高质量参考文档" vs "普通网页文本"
```

**推理与过滤**：
- 对 240T token 池中的每个文档计算质量分数
- 按分数排序，**只保留得分最高的 10%**
- 结果数据集即为 DCLM-Baseline

**为什么选择 fastText**：
- 极快的推理速度：处理 240T tokens 的成本远低于使用 LLM
- 简单但有效：尽管只是一个浅层文本分类器，但效果超越了所有更复杂的方法
- 可复现性强：训练和推理代码完全开源

#### 6.2.4 过滤策略大横评

DCLM 论文的最大价值之一是对各种数据筛选策略进行了 **系统性对比实验**：

| 策略类型 | 具体方法 | Core 指标 | 相对 fastText |
|---------|---------|----------|-------------|
| **Model-based** | **fastText (OH-2.5 + ELI5)** | **最优** | **基线** |
| Model-based | BGE 嵌入分类器 | 次优 | 略低 |
| Perplexity-based | KenLM 困惑度过滤 | 中等 | 低 ~2-3 点 |
| LLM-based | AskLLM（用 LLM 判断质量） | 中等偏上 | 低 ~1-2 点 |
| Graph-based | PageRank 分数过滤 | 较低 | 低 ~3+ 点 |
| Dedup-based | SemDeDup（语义去重） | 较低 | 低 ~3+ 点 |
| Embedding-based | Top-k 平均 logits | 较低 | 低 ~3+ 点 |
| Heuristic | 仅使用启发式规则 | 最低 | 低 ~5+ 点 |

**核心发现**：

1. **fastText 分类器碾压所有方法**：一个简单的 fastText 二分类模型在 Core 指标上比所有替代方案都好。这包括更复杂的深度学习方法（BGE 嵌入）和更昂贵的 LLM-based 方法（AskLLM）
2. **参考数据的选择至关重要**：使用 OpenHermes 2.5 + ELI5 作为正样本比使用 Wikipedia 或 GPT-3 近似数据更有效，在 Core 指标上高出约 3.5 个百分点
3. **过度混合无益**：在激进的 model-based filtering 之后，进一步添加高质量参考数据（Wikipedia、书籍等）的效果是中性甚至负面的

#### 6.2.5 DCLM-Baseline 的最终性能

使用 fastText (OH-2.5 + ELI5) 分类器、保留 top 10% 数据后训练的 7B 模型：

| 指标 | DCLM-Baseline (7B) | MAP-Neo (之前 SOTA) | 提升 |
|------|-------------------|-------------------|------|
| MMLU (5-shot) | **64%** | 57.4% | +6.6pp |
| 训练 tokens | 2.6T | ~4.3T | 少 40% |
| 计算成本 | 基线 | 更高 | 节省 40% |

**这意味着**：仅通过更好的数据筛选，在少用 40% 计算量的情况下，MMLU 提升了 6.6 个百分点。这是 "数据质量 > 数据数量" 最有力的证据之一。

#### 6.2.6 DCLM 的方法论贡献

1. **标准化竞赛框架**：固定了模型架构、训练配方和评估方法，使得不同数据策略可以公平对比
2. **小规模验证大规模**：证明了在 400M-1B 模型上的数据策略效果可以可靠迁移到 7B 规模
3. **简单方法的胜利**：fastText 这种简单的浅层分类器击败了所有复杂方法，为实际工程提供了明确的最佳实践
4. **参考数据选择指南**：揭示了正样本数据的选择对分类器效果的关键影响

### 6.3 Dolma：完全开放的数据处理工具链

**论文**：*Dolma: an Open Corpus of Three Trillion Tokens for Language Model Pretraining Research* (ACL 2024)
**机构**：Allen Institute for AI (AI2)
**链接**：[arXiv:2402.00159](https://arxiv.org/abs/2402.00159)

#### 6.3.1 论文定位与核心贡献

Dolma 的核心差异化在于**完全开放**：不仅开源数据集本身，还开源了完整的数据处理工具链 (Dolma Toolkit)。这使得其他研究者能够：
- 复现整个数据处理流程
- 修改特定步骤并验证效果
- 在自定义数据上应用相同的处理流程

Dolma 是 OLMo 系列模型的训练数据基础。从 Dolma v1 (3T tokens) 到 Dolma 3 (9.3T tokens)，数据规模和质量都在持续迭代。

#### 6.3.2 Dolma 的数据来源与组成

**Dolma v1 (3T tokens) 数据组成**：

| 数据源 | 描述 | 规模 |
|-------|------|------|
| **Common Crawl** | 网页文本（24 个最新快照，约占全部快照的 1/4） | 占主体 |
| **peS2o** | 3800 万篇许可开放的学术论文 | 科学数据 |
| **GitHub 代码** | 多种编程语言的开源代码 | 代码数据 |
| **Project Gutenberg** | 公有领域书籍 | 书籍数据 |
| **Wikipedia** | 百科全书 | 参考数据 |
| **Wikibooks** | 教科书式参考材料 | 参考数据 |
| **Reddit** | 社交媒体对话 | 对话数据 |

原始数据约 200TB 文本，经过处理缩减为约 11TB 的最终数据集。

**Dolma 3 (9.3T tokens) 数据组成**：

Dolma 3 在 v1 基础上大幅扩展，新增了以下关键特性：
- **olmOCR**：使用自研的 OCR 工具处理科学 PDF，提取效果优于传统方法
- **数学问题和解答**：专门收集和生成的数学数据
- **更多代码数据**：扩大了编程语言覆盖范围

Dolma 3 实际上包含三个子数据集：

| 子数据集 | 规模 | 用途 |
|---------|------|------|
| Dolma 3 Mix | 5.9T tokens | 主训练阶段（多样化预训练数据） |
| Dolma 3 Dolmino Mix | 100B tokens | 中训练阶段（数学、代码、QA、指令、推理强化） |
| Dolma 3 Longmino Mix | 50B tokens | 长上下文扩展阶段 |

#### 6.3.3 Dolma Toolkit：四阶段处理架构

Dolma 的工具链采用模块化设计，分为四个核心阶段：

```
阶段 1: Tagging（标注）
  ├── 语言检测 tagger（识别文档语言）
  ├── 毒性检测 tagger（标记有毒内容）
  ├── 困惑度评分 tagger（计算文档困惑度）
  ├── 常见过滤规则 tagger（应用标准启发式过滤）
  └── 自定义 tagger（用户可扩展）
  → 为每个文档或文档片段添加标签

阶段 2: Deduplication（去重）
  ├── 基于 Bloom Filter 的段落级去重
  ├── 使用 Rust 实现的高性能内存 Bloom Filter
  └── 输出属性标记：整篇文档重复 或 特定段落重复
  → 标识重复内容但不直接删除

阶段 3: Mixing（混合）
  ├── 使用 JSONPath/JQ-like 语法进行文档过滤
  ├── 根据标签决定保留或移除
  └── 按指定比例混合来自不同数据流的文档
  → 生成最终的训练数据混合

阶段 4: Tokenization（分词）
  └── 使用指定的分词器将文本转换为 token 序列
  → 输出可直接用于训练的数据
```

**性能分布**：在三个主要阶段中：
- Tagger 阶段占总运行时间的 **80%**
- Deduplication 阶段占 **1.4%**
- Mixer 阶段占 **18.6%**

这说明标注（尤其是需要运行模型的标注）是计算瓶颈。

#### 6.3.4 Dolma 的去重策略

Dolma 的去重采用两级策略：

**段落级 Bloom Filter 去重**：
- 使用 Rust 语言实现的高性能 Bloom Filter
- 在段落粒度进行去重（而非文档级或行级）
- 相比 MinHash，Bloom Filter 更适合精确匹配场景

**Dolma 3 的全局去重创新**：
- 针对万亿 token 规模设计了快速且可扩展的全局去重工具
- 长上下文数据还使用了 **GZIP 压缩率过滤**：移除 GZIP 压缩率在最高 20% 和最低 20% 的文档
  - 最高 20%：高度重复的内容（压缩率极高说明内容冗余）
  - 最低 20%：高度随机的内容（压缩率极低说明内容可能是乱码或非自然语言）

#### 6.3.5 OLMo 3 的三阶段训练策略

Dolma 3 的三个子数据集对应 OLMo 3 的三阶段训练：

**阶段 1：主训练 (Pretraining)**
- 数据：Dolma 3 Mix (5.9T tokens)
- 目标：学习广泛的语言知识和世界知识
- 数据组成：网页、代码、科学论文、书籍、百科等的多样化混合

**阶段 2：中训练 / 退火 (Midtraining / Annealing)**
- 数据：Dolma 3 Dolmino Mix (100B tokens)
- 目标：强化数学、代码、问答、指令遵循和推理能力
- 方法：通过 **微退火实验 (microannealing experiments)** 和集成测试的两部分迭代过程来优化数据配比
- 特点：数据质量极高，针对性地提升关键能力

**阶段 3：长上下文扩展 (Long Context Extension)**
- 数据：Dolma 3 Longmino Mix (50B tokens)
- 目标：扩展模型的上下文窗口
- 过滤：使用 GZIP 压缩率启发式方法确保长文档质量

#### 6.3.6 Dolma 的开放性价值

Dolma 项目的最大贡献在于其对开放性的坚持：

1. **数据完全开源**：原始数据和处理后的数据均可下载
2. **工具链完全开源**：dolma 工具包可直接安装使用 (`pip install dolma`)
3. **处理流程完全文档化**：每一步的决策理由和消融实验结果都有记录
4. **模型完全开源**：OLMo 模型权重、训练代码、评估代码全部公开
5. **数据溯源可追踪**：可以追溯任何一个训练 token 的来源

这种"全链条开放"使得 Dolma/OLMo 成为学术研究的首选基线，因为研究者可以在任何一个环节进行修改和实验。

## 七、合成数据策略深度解析

> 随着高质量自然数据日益稀缺，合成数据成为现代 LLM 预训练和后训练的关键组成部分。本节深入分析各种合成数据策略的技术细节、适用场景和风险。

### 7.1 "教科书即所需" (Textbooks Are All You Need)：Phi 模型的范式开创

**论文**：*Textbooks Are All You Need* (2023)、*Textbooks Are All You Need II: phi-1.5 technical report* (2023)
**机构**：Microsoft Research

#### 7.1.1 核心思想

Phi 系列模型开创了 **"用高质量合成数据取代海量低质量网页数据"** 的范式。其核心假设是：如果训练数据像教科书一样结构清晰、逻辑连贯、循序渐进，那么即使数据量少几个数量级，模型也能学到同样甚至更好的知识。

**Phi-1 (1.3B 参数)**：
- 训练数据：仅 7B tokens
  - 6B tokens：从网页中筛选的"教科书质量"数据
  - 1B tokens：GPT-3.5 生成的合成教科书和练习题
- 效果：在代码生成任务上匹配或超越参数量大 10 倍的模型

**Phi-1.5 (1.3B 参数)**：
- 训练数据：30B tokens
  - 7B tokens：继承自 Phi-1 的数据
  - ~20B tokens：新生成的合成"教科书式"数据，覆盖常识推理和通用知识
- 效果：常识推理基准与参数量大 10 倍、训练数据多 10 倍的模型相当

#### 7.1.2 数据生成策略

Phi 的合成数据生成采用 **主题驱动 (topic-driven)** 方法：
1. 从权威教育来源（如 Stanford 课程大纲、Khan Academy 等）提取主题和知识点
2. 为每个知识点设计 prompt，要求 GPT-3.5/GPT-4 生成教科书式的讲解
3. 同时生成配套的练习题和详细解答
4. 对生成内容进行质量筛选和去重

#### 7.1.3 对行业的影响

Phi 模型的成功直接催生了后续的大规模合成数据工作，包括 Cosmopedia、FineWeb-Edu 的设计理念、以及各家公司对合成数据流水线的投资。

### 7.2 Cosmopedia：开源大规模合成数据集

**项目**：HuggingFace Cosmopedia
**规模**：25B tokens，3000 万个文件

#### 7.2.1 生成方法

Cosmopedia 的目标是复现 Phi 数据集的效果，但完全开源。

**生成模型**：Mixtral-8x7B-Instruct-v0.1
**生成内容类型**：
- 合成教科书
- 博客文章
- 故事
- WikiHow 风格文章

**主题来源**：
- Stanford 课程单元
- Khan Academy 课程
- OpenStax 教科书大纲
- WikiHow 文章主题

**生成流程**：
1. 从上述来源提取主题关键词和课程大纲
2. 构建 prompt：要求模型为特定课程单元生成教科书章节
3. 使用 TGI (Text Generation Inference) 在 H100 集群上大规模推理
4. 总计算时间：超过 **10,000 GPU 小时**

#### 7.2.2 数据质量控制

- **主题多样性**：通过从多个权威教育来源采样主题，确保内容覆盖广泛
- **风格多样性**：生成教科书、博客、故事等多种文体
- **长度控制**：每个生成样本有最大长度限制

### 7.3 WRAP：网页改写增强预训练

**论文**：*Rephrasing the Web: A Recipe for Compute and Data-Efficient Language Modeling* (ACL 2024)
**机构**：CMU + Apple

#### 7.3.1 核心方法

WRAP (Web Rephrase Augmented Pre-training) 的核心思想是：**不是从头生成合成数据，而是改写已有的网页数据**，使其质量提升但保留原始内容的多样性。

**改写模型**：Mistral-7B-Instruct（冻结参数，不做微调）

**四种改写风格**：

| 风格 | 描述 | 目标 |
|------|------|------|
| **Easy** | 简化语言，降低阅读难度 | 提高可读性 |
| **Medium** | 适度改写，改善结构 | 平衡质量和保真度 |
| **Hard** | 深度改写，提升学术性 | 最大化内容质量 |
| **Q/A** | 转换为问答格式 | 增加结构化信息 |

**关键设计决策**：
- 每个样本最多 300 tokens（超过 300 tokens 的改写会导致信息丢失）
- 合成数据与真实数据以 **1:1 比例** 混合
- 保留原始网页数据的多样性，同时通过改写提升质量

#### 7.3.2 性能结果

- **350M 参数模型**，仅使用 C4 语料的 **15%** 的 WRAP 改写数据 → 性能超过在 **全部 C4** 上训练的 1.3B 模型
- 在相同计算预算下，WRAP 使预训练速度提升约 **3 倍**
- 在 Pile 的各子集上，困惑度平均降低超过 **10%**

#### 7.3.3 为什么改写有效

1. **去噪**：网页文本充满语法错误、格式问题和不连贯的段落，改写自然地修复了这些问题
2. **结构化**：将杂乱的网页内容转化为结构清晰的文本，降低了模型的学习难度
3. **多样性保持**：与从头生成不同，改写保留了原始内容的主题分布，避免了合成数据常见的多样性问题

### 7.4 蒸馏 (Distillation)：从大模型到小模型的知识迁移

#### 7.4.1 DeepSeek-R1 蒸馏方案

DeepSeek-R1 展示了当前最成功的推理能力蒸馏方案：

**蒸馏流程**：
1. 使用 DeepSeek-R1（大型 MoE 模型）作为教师模型
2. 精心策划 **800K 训练样本**（覆盖数学推理、代码生成、逻辑推理等）
3. 每个样本包含完整的思维链 (Chain-of-Thought) 过程
4. 使用这些样本对小型密集模型（Qwen2.5 或 Llama-3 backbone）进行微调

**蒸馏结果**：

| 蒸馏模型 | AIME 2024 | MATH-500 | 对比 |
|---------|-----------|----------|------|
| DeepSeek-R1-Distill-Qwen-1.5B | 28.9% | 83.9% | 超越 GPT-4o |
| DeepSeek-R1-Distill-Qwen-7B | 更高 | 更高 | 超越 Claude-3.5-Sonnet |
| DeepSeek-R1-Distill-Llama-70B | 最高 | 最高 | 接近教师模型 |

**关键洞察**：蒸馏最重要的不是数据量（仅 800K 样本），而是 **样本质量和思维链的完整性**。每个样本都包含教师模型的完整推理过程，使得学生模型能学到"如何思考"而不仅是"正确答案是什么"。

#### 7.4.2 Self-Instruct

**Self-Instruct** (Wang et al., 2023) 是最早的合成指令数据生成方法之一：

**流程**：
1. 手动编写少量种子指令（~175 条）
2. 用 LLM（如 GPT-3）基于种子指令生成新的指令
3. 用同一 LLM 为新指令生成回复
4. 过滤低质量的指令-回复对
5. 迭代扩展

**效果**：经过 Self-Instruct 训练的 GPT-3 在用户指令遵循评估上比 vanilla GPT-3 提升了 **33%**，接近 InstructGPT-001 的水平。

#### 7.4.3 Evol-Instruct (WizardLM)

**Evol-Instruct** 在 Self-Instruct 基础上引入了 **指令进化** 机制：

**两种进化方向**：

| 进化类型 | 方法 | 示例 |
|---------|------|------|
| **深度进化 (In-Depth)** | 使指令更复杂 | 添加约束、深化主题、增加推理步骤、具体化、增加推理复杂度 |
| **广度进化 (In-Breadth)** | 生成新的多样化指令 | 覆盖不同主题和任务类型 |

深度进化包含五种具体的 prompt 策略：
1. **添加约束 (Add Constraints)**：在原始指令基础上增加限制条件
2. **深化 (Deepening)**：要求更深层次的理解或分析
3. **增加推理步骤 (Increased Reasoning Steps)**：增加所需的推理链长度
4. **具体化 (Concretizing)**：将抽象指令变为具体场景
5. **增加复杂度 (Complicating Input)**：使输入数据更复杂

### 7.5 数学/代码验证：合成数据的质量保障

#### 7.5.1 数学领域

数学是合成数据最成功的应用领域之一，因为数学答案可以通过形式化方法验证：

**方法**：
1. 使用 LLM 生成数学问题
2. 使用 LLM 生成解题过程（包含思维链）
3. 使用符号计算引擎（如 SymPy）或自动验证器检查答案正确性
4. 只保留答案正确的样本

**优势**：这种"生成-验证"循环可以保证合成数据的准确性，避免了 LLM 幻觉导致的错误训练信号。

#### 7.5.2 代码领域

代码合成数据的质量同样可以通过执行验证：

**方法**：
1. 生成编程问题和测试用例
2. 使用 LLM 生成代码解答
3. 在沙箱环境中执行代码并运行测试用例
4. 只保留通过所有测试的代码

### 7.6 Model Collapse：合成数据的核心风险

#### 7.6.1 什么是 Model Collapse

Model Collapse 是指当模型在自身或其他模型生成的数据上反复训练时，逐步退化的现象。这一现象在 2024 年的 Nature 论文中得到了系统性的实证研究。

**退化过程**：

```
第 1 代模型 → 在真实数据上训练 → 输出质量正常
  ↓ 生成合成数据
第 2 代模型 → 在合成数据上训练 → 输出质量略降
  ↓ 生成合成数据
第 3 代模型 → 在合成数据上训练 → 分布尾部信息丢失
  ↓ ...
第 N 代模型 → 输出高度同质化 → 完全崩溃
```

#### 7.6.2 退化的两个阶段

1. **早期 Model Collapse**：模型首先丢失数据分布尾部（极端值、罕见模式）的信息。输出看起来还正常，但多样性已经开始下降。

2. **晚期 Model Collapse**：数据分布严重收缩，模型输出变得高度同质化，与原始数据分布几乎没有关系。表现为：
   - 生成内容重复度急剧上升
   - 事实准确性显著下降
   - 词汇多样性、句法多样性和语义多样性全面退化
   - 创造性任务表现尤其严重

#### 7.6.3 防范策略

1. **保持真实数据比例底线**：永远不要用合成数据完全替代真实数据。研究表明，真实数据的比例不应低于一定阈值（通常 > 50%）
2. **合成数据 + 真实数据混合**：像 WRAP 那样以 1:1 混合，而不是只用合成数据
3. **多样性保障**：
   - 使用不同的生成 prompt 和风格
   - 从多个模型生成（而非单一模型）
   - 监控生成数据的统计多样性指标
4. **人工审核与过滤**：在关键领域引入人工审核环节
5. **可验证领域优先**：优先在数学、代码等答案可验证的领域使用合成数据
6. **避免迭代训练**：不要在同一模型的输出上反复训练，应该使用不同来源的模型生成数据

### 7.7 合成数据的未来趋势

1. **Scaling Law for Synthetic Data**：如何确定合成数据的最优比例和规模，目前还缺乏系统性的理论框架
2. **领域专用合成**：针对特定领域（医学、法律、金融）的高质量合成数据生成将成为差异化竞争点
3. **多模态合成**：从纯文本扩展到图文、代码-文档、数学-图形等多模态合成数据
4. **合成数据的水印和溯源**：随着合成数据在训练中的占比增加，如何追踪和标记合成数据来源变得越来越重要

---

## 八、数据工程实践建议

基于以上三篇核心论文和合成数据研究，以下是构建预训练数据流水线的实践建议：

### 8.1 流水线构建优先级

```
优先级 1（ROI 最高）：
  ├── 使用专用文本提取器（trafilatura / resiliparse），不要用 CC 默认 WET
  ├── URL 级别去重（成本极低，效果显著）
  └── 单快照 MinHash 去重（5-grams, ~112 哈希函数）

优先级 2（核心差异化）：
  ├── 训练 fastText 质量分类器（参考 DCLM 方法，使用高质量指令数据作为正样本）
  └── 训练教育质量分类器（参考 FineWeb-Edu 方法，使用 LLM 标注 + 小模型推理）

优先级 3（精细调优）：
  ├── 自定义启发式过滤规则（标点比例、重复行比例、短行比例）
  ├── 安全过滤（毒性、PII）
  └── 数据混合比例优化（通过消融实验确定）

优先级 4（高级策略）：
  ├── 合成数据生成与混合
  ├── 多阶段训练的数据策略
  └── 长上下文数据专项处理
```

### 8.2 工具选择

| 任务 | 推荐工具 | 替代方案 |
|------|---------|---------|
| 数据处理流水线 | [datatrove](https://github.com/huggingface/datatrove) (HuggingFace) | [dolma](https://github.com/allenai/dolma) (AI2) |

### 8.3 消融实验方法论

参考 FineWeb 和 DCLM 的方法论，推荐的消融实验框架：

1. **选择小规模代理模型**：400M-1B 参数（DCLM 证明了小规模结果可迁移到大规模）
2. **固定训练配方**：模型架构、超参数、训练步数全部固定
3. **只变一个变量**：每次只改变数据处理的一个环节
4. **使用标准化基准**：至少包含 MMLU、ARC、HellaSwag 三个基准
5. **多次运行取均值**：减少随机性的影响
6. **记录数据统计量**：每步处理后的 token 数、文档数、平均文档长度等
