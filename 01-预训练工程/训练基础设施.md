# 训练基础设施

> 大模型训练是分布式系统工程。本文梳理并行策略、优化器、混合精度和容错机制。

## 一、并行策略全景

```
并行策略
├── DP  (Data Parallelism)       → 数据并行，最基础
├── TP  (Tensor Parallelism)     → 张量并行，单层内切分
├── PP  (Pipeline Parallelism)   → 流水线并行，层间切分
├── CP  (Context Parallelism)    → 上下文并行，序列维度切分
├── EP  (Expert Parallelism)     → 专家并行，MoE 专用
└── ZeRO (Zero Redundancy)       → 优化器状态分片
```

### 数据并行 (DP)
- 每个设备持有完整模型副本
- 不同设备处理不同数据 batch
- 梯度同步：AllReduce
- **ZeRO 优化**：将优化器状态 (Stage 1)、梯度 (Stage 2)、参数 (Stage 3) 分片存储
- **FSDP (Fully Sharded Data Parallelism)**：PyTorch 原生实现的 ZeRO Stage 3

### 张量并行 (TP)
- 单层的权重矩阵切分到多个设备
- 需要 All-Reduce 通信聚合中间结果
- **适合节点内高带宽互连 (NVLink)**
- 典型配置：8-way TP（一个节点内 8 张 GPU）

### 流水线并行 (PP)
- 不同层分配到不同设备
- 微批次 (micro-batch) 形成流水线
- **关键问题**：流水线气泡 (bubble) → 空闲计算时间
- **优化**：1F1B 调度、Interleaved 1F1B

### 上下文并行 (CP)
- 长序列切分到多个设备
- 每个设备处理序列的一部分
- **Ring Attention**：设备间环形传递 KV blocks
- **DeepSpeed-Ulysses**：另一种序列并行方案

### 专家并行 (EP)
- MoE 模型专用
- 不同专家分布在不同设备上
- **核心通信**：All-to-All（token 发送到目标专家设备）
- DeepSeek-V3：8-way EP + 计算-通信重叠

### 典型组合配置

| 模型规模 | DP | TP | PP | CP | EP | 总 GPU |
|---------|----|----|----|----|-----|--------|
| 7B | 8 | 1 | 1 | 1 | - | 8 |
| 70B | 4 | 8 | 2 | 1 | - | 64 |
| 405B (Dense) | 4 | 8 | 16 | 1 | - | 512 |
| 671B (MoE) | 4 | 2 | 4 | 1 | 8 | 256+ |

## 二、优化器

### AdamW (标准)
- 几乎所有大模型的默认优化器
- 带权重衰减的 Adam
- 问题：每个参数需要 2 个额外状态（动量 m 和 v），内存开销大

### Muon / MuonClip (Kimi K2)
- Kimi K2 在万亿参数规模首次使用 Muon 优化器
- **MuonClip**：在 Muon 基础上引入 clip 机制解决不稳定性
- 成果：**15.5T token 训练过程中零 loss spike**
- 这是万亿参数训练稳定性的重大突破

### 8-bit Adam / Adafactor
- 压缩优化器状态以节省内存
- 适用于内存受限场景

## 三、混合精度训练

### 标准方案：BF16 (Brain Float 16)
- 当前主流训练精度
- 相比 FP16，指数位更多 → 数值范围更大 → 不需要 loss scaling

### 前沿方案：FP8 (DeepSeek-V3)
- **DeepSeek-V3** 首次在大规模预训练中采用 FP8
- 技术要点：
  - **细粒度量化 (Fine-Grained Quantization)**：按 block/tile 单独量化
  - **高精度累加**：FP8 乘法结果用 FP32 累加
  - 效果：近乎无损 (near-lossless) 的 BF16 精度
- **需要 NVIDIA Hopper (H100) 或更新架构支持**

### 精度对比
| 精度 | 位数 | 内存占比 | 质量 | 硬件要求 |
|------|------|---------|------|---------|
| FP32 | 32 | 4x | 基线 | 任何 |
| BF16 | 16 | 2x | 几乎无损 | Ampere+ |
| FP8 | 8 | 1x | 近乎无损 | Hopper+ |

## 四、Scaling Laws

### 经典 Scaling Laws

#### Kaplan Scaling Law (OpenAI, 2020)
- Loss ∝ N^(-α) (参数量)
- Loss ∝ D^(-β) (数据量)
- Loss ∝ C^(-γ) (计算量)
- **结论**：应优先扩大模型规模

#### Chinchilla Scaling Law (DeepMind, 2022)
- 修正了 Kaplan：数据量应与参数量同比例增长
- **最优比例**：约 20 tokens/parameter
- Chinchilla 70B (1.4T tokens) > Gopher 280B (300B tokens)
- **这改变了整个行业的训练策略**

### 新方向：推理时 Scaling Law
- OpenAI o1/o3、DeepSeek-R1 证明：推理时的计算量也遵循 Scaling Law
- 更多的 "思考时间" → 更好的推理质量
- 这开辟了**第三个 Scaling 维度**（参数、数据之外）

### 实际观察
- 2024-2025 的模型已经大幅**超出** Chinchilla 最优比例
  - Llama 3 8B 在 15T tokens 上训练（远超 Chinchilla 最优）
  - 原因：推理成本 > 训练成本，更小的过度训练模型推理更便宜

## 五、训练稳定性与容错

### Loss Spike 问题
- 大规模训练中经常出现 loss 突然跳升
- 常见原因：数据质量、学习率、数值精度
- Kimi K2 的 MuonClip 解决方案：零 loss spike

### 检查点 (Checkpointing)
- 定期保存模型状态
- 遇到问题时可以回滚
- OLMo 3 开源了完整的训练检查点序列

### 容错机制
- 大规模集群中 GPU 故障是常态
- 弹性训练 (Elastic Training)
- 自动检测并替换故障节点

## 八、深入解析：Megatron-LM 并行策略

> 参考论文：*Megatron-LM: Training Multi-Billion Parameter Language Models Using Model Parallelism* (Shoeybi et al., 2019) 以及 *Efficient Large-Scale Language Model Training on GPU Clusters Using Megatron-LM* (Narayanan et al., 2021)

### 8.1 张量并行 (Tensor Parallelism) 的数学原理

张量并行的核心思想是**将单个 Transformer 层内的矩阵运算切分到多个 GPU 上并行执行**。Megatron-LM 定义了两种基本的并行线性层：**ColumnParallelLinear（列并行）** 和 **RowParallelLinear（行并行）**，并通过巧妙地组合这两种层来最小化通信开销。

#### 8.1.1 列并行线性层 (ColumnParallelLinear)

列并行将权重矩阵 **A** 沿列维度切分。假设有 t 个 GPU，权重 A ∈ R^{k×n} 被切分为 t 个子矩阵：

```
A = [A₁, A₂, ..., Aₜ]     其中 Aᵢ ∈ R^{k×(n/t)}
```

对于输入 X ∈ R^{b×k}（b 为 batch 维度），每个 GPU i 独立计算：

```
Yᵢ = X · Aᵢ               Yᵢ ∈ R^{b×(n/t)}
```

**关键优势**：输出 Yᵢ 在列维度上自然分片，如果后接的是逐元素操作（如 GeLU 激活函数），可以**直接在分片上独立执行**，无需通信。

#### 8.1.2 行并行线性层 (RowParallelLinear)

行并行将权重矩阵 **B** 沿行维度切分：

```
B = [B₁; B₂; ...; Bₜ]     其中 Bᵢ ∈ R^{(n/t)×m}
```

输入已经按列分片（来自上一层的 ColumnParallelLinear 输出），每个 GPU i 持有 Yᵢ ∈ R^{b×(n/t)}，独立计算：

```
Zᵢ = Yᵢ · Bᵢ              Zᵢ ∈ R^{b×m}
```

最终结果需要跨 GPU 汇总：

```
Z = Z₁ + Z₂ + ... + Zₜ    （通过 AllReduce 实现）
```

#### 8.1.3 f 和 g 共轭算子

Megatron-LM 引入了两个通信原语算子 **f** 和 **g**，它们是一对**共轭算子 (conjugate pair)**：

| 算子 | 前向传播 | 反向传播 |
|------|---------|---------|
| **f** | Identity（直接复制） | AllReduce（梯度聚合） |
| **g** | AllReduce（结果聚合） | Identity（直接复制） |

在前向传播中，输入 X 通过 **f** 复制到所有 GPU（Identity），经过分片计算后通过 **g** 进行 AllReduce 聚合。在反向传播中，梯度从 **g** 直接传回（Identity），经过分片计算后通过 **f** 进行 AllReduce 聚合。这样设计使得**前向和反向各只需要一次 AllReduce**。

#### 8.1.4 MLP 层的张量并行

Transformer 的 MLP 块包含两个线性层和一个 GeLU 激活：

```
MLP(X) = Dropout(GeLU(X · A) · B)
```

Megatron-LM 的切分策略：

```
       GPU 0                        GPU 1
       ┌────────┐                   ┌────────┐
X ──f──│ X·A₁   │                   │ X·A₂   │  ← 列并行（第一个 GEMM）
       │  ↓     │                   │  ↓     │
       │ GeLU   │                   │ GeLU   │  ← 独立执行，无需通信！
       │  ↓     │                   │  ↓     │
       │ Y₁·B₁  │                   │ Y₂·B₂  │  ← 行并行（第二个 GEMM）
       └───┬────┘                   └───┬────┘
           └──────────g (AllReduce)─────┘
                       ↓
                    Z = Z₁+Z₂
                       ↓
                    Dropout
```

核心洞察：**第一个 GEMM 用列并行，第二个 GEMM 用行并行**，GeLU 在分片上独立执行。这样**两个 GEMM 之间不需要通信**，整个 MLP 块前向和反向各只需要一次 AllReduce。

#### 8.1.5 Self-Attention 层的张量并行

多头注意力天然适合张量并行——不同的注意力头可以分配到不同 GPU：

```
MultiHead(Q, K, V) = Concat(head₁, head₂, ..., headₕ) · Wₒ
```

将注意力头均匀分配到 t 个 GPU，每个 GPU 计算 h/t 个头：
- **Q、K、V 的投影矩阵**：按列并行切分（每个 GPU 只负责自己头的投影）
- **输出投影矩阵 Wₒ**：按行并行切分
- 通信模式与 MLP 完全相同：**前向一次 AllReduce，反向一次 AllReduce**

#### 8.1.6 通信量分析

对于 t-way 张量并行，每个 Transformer 层需要：
- 前向传播：**2 次 AllReduce**（MLP 一次 + Attention 一次）
- 反向传播：**2 次 AllReduce**（MLP 一次 + Attention 一次）
- 每次 AllReduce 的通信量：2(t-1)/t × b × s × h bytes（其中 b=batch, s=seq_len, h=hidden_dim）

因此张量并行**强烈依赖高带宽互连**，通常限制在单节点 8 GPU 内（NVLink 带宽 ~900 GB/s），跨节点使用会因带宽不足而成为瓶颈。

### 8.2 流水线并行 (Pipeline Parallelism) 详解

#### 8.2.1 基本概念

流水线并行将模型的 L 层均匀分成 p 个阶段 (stage)，每个阶段放在一个设备上。为了减少流水线气泡，将每个 mini-batch 切分成 m 个 micro-batch。

#### 8.2.2 朴素流水线 (Naive Pipeline)

最简单的方法是按顺序执行所有前向，再按顺序执行所有反向：

```
时间 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→
GPU 0: [F₁][F₂][F₃][F₄][ 等待 ][B₄][B₃][B₂][B₁]
GPU 1:     [F₁][F₂][F₃][F₄][ 等 ][B₄][B₃][B₂][B₁]
GPU 2:         [F₁][F₂][F₃][F₄]  [B₄][B₃][B₂][B₁]
GPU 3:             [F₁][F₂][F₃][F₄][B₄][B₃][B₂][B₁]
```

气泡比例极高：**bubble_ratio = (p-1) / m**。当 p=4, m=4 时，气泡占比高达 75%！

#### 8.2.3 1F1B 调度 (One Forward One Backward)

1F1B 调度大幅减少了气泡：

**三个阶段**：
1. **预热阶段 (Warmup)**：每个 stage 执行若干前向 pass，stage i 比 stage i+1 多执行一个前向
2. **稳态阶段 (Steady State)**：严格交替执行一次前向和一次反向
3. **冷却阶段 (Cooldown)**：执行剩余的反向 pass

```
时间 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→
GPU 0: [F₁][F₂][F₃][F₄][B₁][F₅][B₂][F₆][B₃][F₇][B₄]...[B₇]
GPU 1:     [F₁][F₂][F₃][B₁][F₅][B₂][F₆][B₃]...[B₇]
GPU 2:         [F₁][F₂][B₁][F₅][B₂]...[B₇]
GPU 3:             [F₁][B₁][F₅]...[B₇]
```

气泡数量为 **(p-1)** 个时间单元，气泡比例：

```
bubble_ratio = (p - 1) / m
```

当 m >> p 时，气泡趋近于零。实践中通常设置 m ≥ 4p。

#### 8.2.4 交错式 1F1B (Interleaved 1F1B)

Narayanan et al. (2021) 提出的进一步优化。核心思想：每个 GPU 不再只负责**连续**的一组层，而是负责**多个不连续的层块 (chunk)**。

假设每个 stage 有 v 个虚拟阶段 (virtual stage)：
- 原始 1F1B：GPU 0 负责 Layer 1-8，GPU 1 负责 Layer 9-16...
- 交错 v=2：GPU 0 负责 Layer 1-4 **和** Layer 17-20，GPU 1 负责 Layer 5-8 **和** Layer 21-24...

气泡比例降低为：

```
bubble_ratio = (p - 1) / (m × v)
```

v=2 时气泡减半，v=4 时减为四分之一。但代价是**通信量增加 v 倍**（因为 micro-batch 需要在设备间多次传递）。

#### 8.2.5 Zero Bubble Pipeline Parallelism

最新进展（Qi et al., 2024）将反向传播拆分为 **B（计算输入梯度）** 和 **W（计算权重梯度）**：

```
传统反向: B = dL/dX + dL/dW  （绑定在一起）
Zero Bubble: 拆分为 B (dL/dX) 和 W (dL/dW)
```

- F 和 B 之间有跨 stage 的数据依赖（必须顺序执行）
- 但 **W 可以灵活调度**（只依赖本 stage 的 B 结果）
- 将 W 填入流水线气泡中，理论上可以实现**零气泡**

### 8.3 张量并行 + 流水线并行的组合

在实际训练中，TP 和 PP 组合使用形成 **3D 并行** (搭配数据并行)：

```
                    ┌──────── Node 0 ────────┐  ┌──────── Node 1 ────────┐
Pipeline Stage 0:   │ GPU0  GPU1  GPU2  GPU3  │  │ GPU0  GPU1  GPU2  GPU3  │
                    │ ←── 8-way TP (NVLink) ──│  │ ←── 8-way TP (NVLink) ──│
                    └─────────────────────────┘  └─────────────────────────┘
                              │                            │
                        PP 通信 (跨节点, InfiniBand)
                              │                            │
                    ┌──────── Node 2 ────────┐  ┌──────── Node 3 ────────┐
Pipeline Stage 1:   │ GPU0  GPU1  GPU2  GPU3  │  │ GPU0  GPU1  GPU2  GPU3  │
                    │ ←── 8-way TP (NVLink) ──│  │ ←── 8-way TP (NVLink) ──│
                    └─────────────────────────┘  └─────────────────────────┘

                    Node 0 和 Node 1 是数据并行副本 (同一 pipeline stage)
```

**设计原则**：
- **TP 放在节点内**：通信量大但 NVLink 带宽充足
- **PP 放在节点间**：通信量小（只传激活值），对带宽要求低
- **DP 放在最外层**：最容易扩展，梯度同步用 AllReduce

## 九、深入解析：DeepSeek-V3 的 FP8 训练

> 参考论文：*DeepSeek-V3 Technical Report* (DeepSeek-AI, 2024)

### 9.1 为什么需要 FP8 训练

BF16 混合精度训练是当前主流，但在超大规模模型训练中仍存在瓶颈：
- **内存瓶颈**：671B 参数的 MoE 模型，即使用 BF16 也需要巨大的 GPU 内存
- **计算瓶颈**：矩阵乘法 (GEMM) 是训练中的计算核心，FP8 的吞吐量是 BF16 的 **2 倍**
- **通信瓶颈**：FP8 权重/激活的通信量减半

FP8 训练的挑战在于：FP8 的动态范围极其有限（E4M3 格式只有 4 位指数 + 3 位尾数），直接量化会导致**严重的精度损失**。DeepSeek-V3 的核心贡献就是解决这一问题。

### 9.2 FP8 数据格式

DeepSeek-V3 统一使用 **E4M3** 格式（而非混合使用 E4M3/E5M2）：

```
E4M3 格式 (8 bits):
┌───┬───────────┬─────────┐
│ S │ E₃E₂E₁E₀  │ M₂M₁M₀  │
│1b │  4 bits    │ 3 bits   │
└───┴───────────┴─────────┘

- 符号位: 1 bit
- 指数位: 4 bits → 动态范围: 2^(-6) ~ 2^8 ≈ [0.015625, 448]
- 尾数位: 3 bits → 精度: ~1/8 = 12.5% 相对误差
- 最大可表示值: 448.0
```

对比 BF16：
```
BF16 格式 (16 bits):
┌───┬───────────────┬─────────────┐
│ S │ E₇...E₀       │ M₆...M₀     │
│1b │  8 bits        │ 7 bits       │
└───┴───────────────┴─────────────┘

- 指数位: 8 bits → 动态范围极大
- 尾数位: 7 bits → 精度: ~0.8% 相对误差
```

E4M3 的动态范围比 BF16 小 **2^4 = 16 倍**，精度差 **2^4 = 16 倍**。这就是为什么需要细粒度量化来弥补。

### 9.3 细粒度量化 (Fine-Grained Quantization)

DeepSeek-V3 的核心创新是**放弃全局量化**（对整个张量用一个 scaling factor），改为**分块量化**（对小区域分别计算 scaling factor）：

#### 9.3.1 激活值：Tile-wise 量化 (1×128)

```
激活矩阵 X ∈ R^{m×k}:

┌────────────────────────────────────────────────────┐
│  tile₁  │  tile₂  │  tile₃  │  tile₄  │ ...       │  ← 每行
│ (1×128) │ (1×128) │ (1×128) │ (1×128) │           │
├─────────┼─────────┼─────────┼─────────┤           │
│  tile₅  │  tile₆  │  tile₇  │  tile₈  │ ...       │
├─────────┼─────────┼─────────┼─────────┤           │
│  ...    │  ...    │  ...    │  ...    │           │
└────────────────────────────────────────────────────┘

每个 1×128 tile 独立计算缩放因子:
s = max(|tile|) / 448.0
tile_fp8 = round(tile / s)
```

为什么选择 1×128？
- 128 对齐 GPU warp 大小和内存访问模式
- 每行独立量化，避免行间数值范围差异的干扰
- 计算 max 的开销可以与 GEMM 流水线重叠

#### 9.3.2 权重：Block-wise 量化 (128×128)

```
权重矩阵 W ∈ R^{k×n}:

┌──────────────────────────────────────────────────┐
│    Block₁     │    Block₂     │    Block₃     │  │
│  (128×128)    │  (128×128)    │  (128×128)    │  │
├───────────────┼───────────────┼───────────────┤  │
│    Block₄     │    Block₅     │    Block₆     │  │
│  (128×128)    │  (128×128)    │  (128×128)    │  │
├───────────────┼───────────────┼───────────────┤  │
│    ...        │    ...        │    ...        │  │
└──────────────────────────────────────────────────┘

每个 128×128 block 独立计算缩放因子:
s = max(|block|) / 448.0
block_fp8 = round(block / s)
```

为什么权重用更大的 block？
- 权重在训练过程中变化缓慢，数值分布更均匀
- 更大的 block 减少了 scaling factor 的存储开销
- 128×128 对齐 Tensor Core 的 warp tile 大小

#### 9.3.3 为什么细粒度量化有效

**关键洞察**：分组量化相当于让组内元素**共享指数位**。

假设一个 1×128 tile 内的数值范围是 [a_min, a_max]，全局量化时数值范围可能是 [g_min, g_max]：
- 如果 g_max >> a_max，那么 tile 内的值在全局量化后会被压缩到 FP8 的低精度区间
- 分组量化让每个 tile 独立使用完整的 FP8 精度范围

这等价于**将 E4M3 的 4 位指数扩展为有效的 ~8 位指数**（因为每组有自己的 scaling factor 编码了组级指数信息）。

### 9.4 在线量化 vs 延迟量化

DeepSeek-V3 采用**在线量化 (Online Quantization)**，而非 NVIDIA Transformer Engine 的延迟量化：

```
延迟量化 (Delayed Quantization):
  Step t: 使用 Step t-1 记录的 max(|X|) 作为 scaling factor
  问题: 如果 Step t 的激活分布突变，scaling factor 不匹配 → 精度损失

在线量化 (Online Quantization):
  Step t: 实时计算当前 tile/block 的 max(|X|)，立即量化
  优势: scaling factor 始终精确匹配当前数据
  代价: 需要额外的 reduce 操作计算 max，但可以通过流水线隐藏
```

### 9.5 高精度累加 (High-Precision Accumulation)

FP8 GEMM 的精度瓶颈不仅在于乘法精度，更在于**累加精度**。

#### 9.5.1 问题：Hopper GPU 的累加器精度不足

NVIDIA H100 的 FP8 Tensor Core 使用 **14-bit 累加器**（不是 FP32！）：

```
FP8 GEMM 在 Tensor Core 内部:
  C[i][j] += A[i][k] × B[k][j]    (FP8 × FP8 → 14-bit 累加)

  问题: 当 K 维度很大时 (如 K=4096)，
  大量的小数值累加到 14-bit 寄存器中会产生严重的舍入误差
```

#### 9.5.2 解决方案：每 128 个元素提升到 CUDA Core 做 FP32 累加

```
传统做法 (纯 Tensor Core):
  result = Σ_{k=0}^{K-1} A[i][k] × B[k][j]    // 14-bit 累加，K=4096 次

DeepSeek-V3 做法 (Tensor Core + CUDA Core 交替):
  for chunk = 0, 128, 256, ..., K-128:
      partial_sum = Σ_{k=chunk}^{chunk+127} A[i][k] × B[k][j]  // 14-bit 累加，仅 128 次
      fp32_accumulator += (float)partial_sum                     // 提升到 FP32 累加
  result = fp32_accumulator
```

实现方式：每执行 4 条 WGMMA (Warpgroup Matrix Multiply-Accumulate) 指令后，将中间结果从 Tensor Core 搬到 CUDA Core 进行 FP32 累加。这增加了约 ~5% 的计算开销，但**大幅提升了数值精度**。

### 9.6 近乎无损的训练效果

在 DeepSeek-V3 的 671B MoE 模型上，FP8 训练相比 BF16 基线：

```
精度对比:
┌────────────────────────────┬─────────────┐
│ 指标                       │ 相对误差     │
├────────────────────────────┼─────────────┤
│ 训练 Loss 偏差             │ < 0.25%     │
│ 下游任务评估偏差           │ 可忽略不计   │
│ 训练速度提升               │ ~40%        │
│ 内存节省                   │ ~30%        │
└────────────────────────────┴─────────────┘
```

三大关键技术的协同效果：
1. **细粒度量化** → 解决动态范围不足问题
2. **在线量化** → 解决 scaling factor 滞后问题
3. **高精度累加** → 解决长累加链的精度损失问题

这三者缺一不可——去掉任何一个，FP8 训练都会出现明显的精度退化。

### 9.7 整体训练框架

```
DeepSeek-V3 FP8 混合精度训练框架:

Master Weights: FP32 (存储在优化器中)
     ↓ 量化
Forward/Backward Weights: FP8 (Block-wise 128×128 量化)
     ↓
Activations: FP8 (Tile-wise 1×128 量化，在线计算 scaling factor)
     ↓
GEMM 计算: FP8 × FP8 → FP32 累加 (每 128 元素提升一次)
     ↓
梯度: BF16 (保持梯度精度用于优化器更新)
     ↓
优化器状态: FP32 (Adam 的 m 和 v)
```

## 十、深入解析：MuonClip 优化器 (Kimi K2)

> 参考论文：*Kimi K2: Open Agentic Intelligence* (Moonshot AI, 2025)

### 10.1 从 AdamW 到 Muon：优化器的演进

#### 10.1.1 AdamW 的局限性

AdamW 是当前大模型训练的默认优化器，其更新规则：

```
mₜ = β₁ · mₜ₋₁ + (1 - β₁) · gₜ          (一阶矩/动量)
vₜ = β₂ · vₜ₋₁ + (1 - β₂) · gₜ²         (二阶矩/方差)
θₜ = θₜ₋₁ - η · (m̂ₜ / (√v̂ₜ + ε) + λθₜ₋₁)  (参数更新 + 权重衰减)
```

AdamW 的问题：
- **逐元素缩放**：每个参数独立处理，忽略了参数矩阵的**全局结构信息**
- **内存开销大**：每个参数需要额外存储 m 和 v，总共 12 bytes/param
- **收敛效率**：在大规模训练中，AdamW 的 token 效率不是最优的

#### 10.1.2 Muon 优化器的核心思想

Muon (Matrix Update via Orthogonalization and Nesterov momentum) 由 Keller Jordan 于 2024 年提出，核心思想是**将梯度更新方向正交化**。

**数学直觉**：对于矩阵形式的参数 W ∈ R^{m×n}（如线性层权重），最陡下降方向取决于所使用的范数：
- **Frobenius 范数下的最陡下降** → 普通梯度下降（SGD）
- **谱范数 (Spectral Norm) 下的最陡下降** → Muon

谱范数下的最陡下降方向是梯度矩阵 G 的**正交化版本**：

```
Δ = Orthogonalize(G)

其中 Orthogonalize(G) 返回与 G 最近的正交矩阵
（即满足 Δᵀ·Δ = I 或 Δ·Δᵀ = I 的矩阵中，使 ||Δ - G||_F 最小的那个）
```

这等价于对 G 做**奇异值分解 (SVD)** 后将所有奇异值设为 1：

```
G = U · Σ · Vᵀ    (SVD 分解)
Δ = U · I · Vᵀ    (奇异值全部置为 1)
  = U · Vᵀ        (正交极分解)
```

#### 10.1.3 Newton-Schulz 迭代替代 SVD

直接计算 SVD 的复杂度是 O(mn · min(m,n))，在训练循环中不可接受。Muon 使用 **Newton-Schulz 迭代**来近似计算正交化：

```
初始化: X₀ = G / ||G||_F    (归一化)

迭代 (通常 5-10 步):
  Xₖ₊₁ = a · Xₖ + b · Xₖ · Xₖᵀ · Xₖ + c · Xₖ · Xₖᵀ · Xₖ · Xₖᵀ · Xₖ

其中 a, b, c 是预计算的最优系数 (如 a=3.4445, b=-2.8025, c=0.8025)
```

Newton-Schulz 迭代的优势：
- 只涉及**矩阵乘法**，可以高效利用 Tensor Core
- 不需要精确收敛，5-10 步的近似已经足够
- 可以用 BF16 计算，无需额外精度

#### 10.1.4 完整的 Muon 算法

```
算法: Muon 优化器

输入: 学习率 η, 动量系数 μ, NS 迭代步数 K
初始化: 动量缓冲 buf = 0

每步更新:
  1. 计算梯度 G
  2. Nesterov 动量:
     buf = μ · buf + G
     G_nesterov = G + μ · buf
  3. Newton-Schulz 正交化:
     Δ = NewtonSchulz(G_nesterov, K 步)
  4. 参数更新:
     W = W - η · Δ
```

Muon 的优势：
- **不需要存储 v（二阶矩）**，内存比 AdamW 少约 33%
- **更好的 token 效率**：在相同 token 数下达到更低的 loss
- **更新方向更优**：考虑了参数矩阵的全局结构

### 10.2 Muon 在大规模训练中的不稳定性

Muon 在中小规模模型上表现优异，但在**万亿参数规模**上会出现严重的训练不稳定性：

#### 10.2.1 注意力 Logit 爆炸问题

在 Transformer 的自注意力机制中：

```
Attention(Q, K, V) = softmax(Q · Kᵀ / √d) · V
```

Q·Kᵀ 的数值（即 attention logits）应当保持在合理范围内。但在大规模 Muon 训练中观察到：

```
训练初期: max(|Q·Kᵀ|) ≈ 10-50    (正常)
训练中期: max(|Q·Kᵀ|) → 500+      (开始异常)
训练后期: max(|Q·Kᵀ|) → 1000+     (爆炸！)
```

当 attention logits 过大时：
- softmax 输出趋近于 one-hot → 梯度消失
- 数值溢出 → NaN / Inf → **loss spike** 甚至训练发散

#### 10.2.2 为什么 Muon 更容易出现此问题

Muon 的正交化更新使得**每步更新的谱范数恒定**（所有奇异值为 1）。这意味着 W_q 和 W_k 的权重范数增长不受 AdamW 中自适应学习率的自然抑制，容易导致 Q 和 K 的范数持续增长。

### 10.3 MuonClip：QK-Clip 机制

MuonClip 在标准 Muon 的基础上增加了 **QK-Clip** 机制，直接从源头控制 attention logit 的增长。

#### 10.3.1 QK-Clip 算法

```
算法: QK-Clip (在每个 Muon 更新步之后执行)

输入:
  - W_q, W_k: 当前层的 Query 和 Key 投影权重
  - τ: logit 阈值上界 (如 τ = 100.0)
  - α: Q/K 调整比例系数 (如 α = 0.5)
  - X_sample: 一个小批量样本的隐藏状态

步骤:
  1. 估算最大 attention logit:
     Q = X_sample · W_q
     K = X_sample · W_k
     max_logit = max(|Q · Kᵀ|) / √d

  2. 判断是否需要裁剪:
     if max_logit > τ:
         η = τ / max_logit         (η < 1，缩放因子)
     else:
         η = 1.0                   (不需要裁剪)

  3. 对称缩放 Q 和 K 权重:
     W_q = W_q · η^α              (缩放 Query 权重)
     W_k = W_k · η^(1-α)          (缩放 Key 权重)
```

#### 10.3.2 为什么 QK-Clip 有效

1. **直接作用于根源**：不是在 softmax 之前 clamp logits（会丢失梯度信息），而是**直接缩放权重矩阵**，保持了梯度流的完整性

2. **对称调整**：通过 α 参数平衡 Q 和 K 的缩放，避免单方面缩放导致的表示能力不对称：
   ```
   Q·Kᵀ = (X·W_q)·(X·W_k)ᵀ
   缩放后: (X·W_q·η^α)·(X·W_k·η^(1-α))ᵀ = Q·Kᵀ · η^α · η^(1-α) = Q·Kᵀ · η
   ```
   即 attention logit 整体被缩放了 η 倍，而 Q 和 K 各自承担部分缩放

3. **不影响优化轨迹**：QK-Clip 只在 logit 超过阈值时才触发，正常训练过程中 Muon 的更新完全不受影响

#### 10.3.3 训练过程中的动态行为

Kimi K2 训练中观察到的 attention logit 动态：

```
训练进度:   0%        10%        30%        50%        100%
max logit:  10  ──→  50  ──→  100  ──→  80  ──→   60
                              ↑
                        达到阈值 τ=100
                        QK-Clip 开始生效
                              ↓
                        logit 被约束
                        自然衰减到稳定范围
```

关键观察：max logit 在训练前 ~30% 的步骤中快速上升至阈值 τ（100），被 QK-Clip 约束后，会**自然衰减到稳定范围**（~60-80），不再需要持续裁剪。

### 10.4 Kimi K2 的训练成果

| 指标 | 数值 |
|------|------|
| 模型规模 | 1T 参数 (MoE) |
| 训练数据 | 15.5T tokens |
| Loss Spike 数量 | **0**（零 loss spike） |
| 优化器 | MuonClip (Muon + QK-Clip) |
| 对比 | AdamW 在同规模训练中通常有数十次 loss spike |

这是万亿参数规模预训练中**训练稳定性的重大突破**。MuonClip 证明了 Muon 家族优化器可以安全地扩展到最大规模，同时保持其优越的 token 效率。

### 10.5 MuonClip 的适用范围

需要注意的是，MuonClip 仅用于**2D 矩阵参数**（如线性层权重）。对于以下参数仍使用 AdamW：
- Embedding 层
- LayerNorm / RMSNorm 参数
- 1D bias 项
- 路由器 (Router) 权重

这是因为 Muon 的正交化操作只对矩阵有意义，对向量参数没有定义。

## 十一、深入解析：Chinchilla vs Kaplan Scaling Laws

> 参考论文：
> - *Scaling Laws for Neural Language Models* (Kaplan et al., OpenAI, 2020)
> - *Training Compute-Optimal Large Language Models* (Hoffmann et al., DeepMind, 2022)

### 11.1 Kaplan Scaling Law (2020)

#### 11.1.1 核心发现

Kaplan 等人发现，语言模型的交叉熵损失 L 与三个因素呈幂律关系：

```
L(N) = (N_c / N)^αN        αN ≈ 0.076    (参数量缩放)
L(D) = (D_c / D)^αD        αD ≈ 0.095    (数据量缩放)
L(C) = (C_c / C)^αC        αC ≈ 0.050    (计算量缩放)

其中 N_c, D_c, C_c 是常数前因子
```

#### 11.1.2 联合缩放公式

当参数量 N 和数据量 D 同时变化时：

```
L(N, D) = [(N_c/N)^(αN/αD) + (D_c/D)]^αD

简化形式: L ≈ A/N^α + B/D^β + E

其中 E 是不可约损失 (irreducible loss)，代表自然语言本身的熵下界
```

#### 11.1.3 Kaplan 的最优分配结论

给定固定计算预算 C（其中 C ≈ 6ND，6 FLOPs/token/parameter），Kaplan 推导出：

```
最优参数量: N_opt ∝ C^0.73
最优数据量: D_opt ∝ C^0.27

即: 计算增加 10 倍时
  → 模型应增大 10^0.73 ≈ 5.4 倍
  → 数据应增加 10^0.27 ≈ 1.9 倍
```

**含义**：Kaplan 建议**优先扩大模型**，数据量增长较少。这导致了 GPT-3 等模型使用较大参数量但相对较少的训练数据。

### 11.2 Chinchilla Scaling Law (2022)

#### 11.2.1 Hoffmann 的修正

Hoffmann 等人在更大规模的实验中重新拟合了缩放律，得到了**截然不同**的结论：

```
L(N, D) = A/N^α + B/D^β + E

Chinchilla 拟合参数（第三种估计方法）:
  A ≈ 406.4,  α ≈ 0.34
  B ≈ 410.7,  β ≈ 0.28
  E ≈ 1.69

最优分配:
  N_opt ∝ C^a,  a ≈ 0.50
  D_opt ∝ C^b,  b ≈ 0.50
```

#### 11.2.2 关键差异对比

```
计算预算增加 10 倍:

Kaplan:      模型增大 5.4×,  数据增加 1.9×   → 重模型，轻数据
Chinchilla:  模型增大 3.2×,  数据增加 3.2×   → 均衡增长

Kaplan 最优:   N:D ≈ 1:?? (偏向大模型少数据)
Chinchilla 最优: N:D ≈ 1:20 (每个参数约 20 tokens)
```

#### 11.2.3 为什么两者结论不同

Epoch AI 在 2024 年的复现研究揭示了差异的原因：

1. **参数计数方式不同**：
   - Kaplan 计算的是**非嵌入参数 (non-embedding parameters)**
   - Chinchilla 计算的是**总参数数 (total parameters)**
   - 在小模型中，嵌入层参数占比很大，忽略它们会导致缩放指数偏移

2. **模型规模不同**：
   - Kaplan 使用的模型较小（最大 ~1.5B）
   - Chinchilla 使用了更大的模型（最大 ~70B）
   - 小规模下的偏差在大规模下被放大

3. **学习率调度**：
   - Kaplan 没有为每个模型大小优化学习率
   - Chinchilla 对每个配置都做了超参数搜索

#### 11.2.4 Chinchilla 定律的验证

Chinchilla (70B, 1.4T tokens) vs Gopher (280B, 300B tokens)：

```
模型         参数量    训练数据    计算量(近似)    评估性能
Gopher       280B     300B        ~50×10²¹        基线
Chinchilla    70B     1.4T        ~50×10²¹        优于 Gopher ✓

两者使用**相同的计算量**，但 Chinchilla 将预算均匀分配:
- 模型缩小 4 倍 → 参数计算量减少 4 倍
- 数据增加 4.7 倍 → 训练步数增加 4.7 倍
- 净效果: 计算量基本相同，但性能更好
```

### 11.3 为什么现代模型超越 Chinchilla 最优

2024-2025 年的实践中，几乎所有主流模型都**大幅超出** Chinchilla 最优比例：

| 模型 | 参数量 | 训练 Tokens | Token/Param 比 | Chinchilla 最优比 |
|------|--------|------------|----------------|------------------|
| Chinchilla | 70B | 1.4T | 20:1 | 1.0x (最优) |
| Llama 2 70B | 70B | 2T | 29:1 | 1.4x |
| Llama 3 8B | 8B | 15T | 1875:1 | **94x** |
| Llama 3 70B | 70B | 15T | 214:1 | **10.7x** |
| Mistral 7B | 7B | ~8T | ~1143:1 | ~57x |
| Qwen 2.5 72B | 72B | 18T | 250:1 | **12.5x** |

#### 11.3.1 经济学解释：推理成本 >> 训练成本

Chinchilla 最优化的是**训练计算量**，但实际部署中：

```
总成本 = 训练成本 + 推理成本 × 请求量

假设模型服务 10 亿次请求:
  - Chinchilla 最优 70B: 训练便宜，但每次推理都用 70B 模型，推理成本高
  - 过度训练的 8B:      训练稍贵（多训 10 倍数据），但推理快 8-9 倍

当推理请求量足够大时:
  总成本(8B, 15T tokens) < 总成本(70B, 1.4T tokens)
```

Sardana & Frankle (2024) 的论文 *Beyond Chinchilla-Optimal* 给出了修正公式：

```
考虑推理的最优分配:
N_opt ∝ C_train^a / (R · C_infer)^b

其中 R = 预期推理请求总量
当 R 很大时，N_opt 显著小于 Chinchilla 最优 → 应该训练更小但训练更久的模型
```

#### 11.3.2 性能解释：过度训练仍有收益

即使超出 Chinchilla 最优，继续训练仍能**持续降低 loss**（只是边际收益递减）：

```
Loss = A/N^α + B/D^β + E

D 从 1.4T → 15T (增加 10.7 倍):
  B/D^β 项的减少 = B × (1/(15T)^0.28 - 1/(1.4T)^0.28)

  虽然这种减少小于 Chinchilla 最优分配下的减少，
  但因为推理成本的降低远超训练成本的增加，
  所以从全生命周期来看是值得的。
```

### 11.4 推理时 Scaling Law：第三个维度

OpenAI o1/o3 和 DeepSeek-R1 开辟了新的 scaling 维度：

```
传统 Scaling:
  性能 = f(N, D)                    (参数量, 数据量)

推理时 Scaling:
  性能 = f(N, D, C_inference)       (参数量, 数据量, 推理计算量)

其中 C_inference = 推理时使用的 "思考" tokens 数量
```

这意味着即使预训练 scaling 遇到瓶颈，也可以通过增加推理时的计算量来继续提升性能——本质上是用**计算换智能**的另一个维度。

## 十二、深入解析：ZeRO 与 FSDP

> 参考论文：
> - *ZeRO: Memory Optimizations Toward Training Trillion Parameter Models* (Rajbhandari et al., Microsoft, 2020)
> - *PyTorch FSDP: Experiences on Scaling Fully Sharded Data Parallel* (Zhao et al., Meta, 2023)

### 12.1 混合精度训练的内存开销

在理解 ZeRO 之前，必须先理解标准混合精度训练的内存组成。对于参数量为 Ψ 的模型，使用 Adam 优化器的混合精度训练：

```
内存组成 (每个参数的字节数):

┌────────────────────────────────┬──────┬──────────────────┐
│ 组成部分                       │ 精度  │ 每参数字节数      │
├────────────────────────────────┼──────┼──────────────────┤
│ 模型参数 (前向/反向用)         │ FP16 │ 2Ψ bytes         │
│ 梯度                           │ FP16 │ 2Ψ bytes         │
│ 优化器状态 - FP32 参数副本     │ FP32 │ 4Ψ bytes         │
│ 优化器状态 - 一阶矩 (m)       │ FP32 │ 4Ψ bytes         │
│ 优化器状态 - 二阶矩 (v)       │ FP32 │ 4Ψ bytes         │
├────────────────────────────────┼──────┼──────────────────┤
│ 总计                           │      │ 16Ψ bytes        │
└────────────────────────────────┴──────┴──────────────────┘

对于 7B 模型: 16 × 7×10⁹ = 112 GB（仅模型状态，不含激活值！）
对于 70B 模型: 16 × 70×10⁹ = 1120 GB = 1.1 TB
```

**核心洞察**：模型参数本身 (2Ψ) 只占总内存的 12.5%，优化器状态 (12Ψ) 占了 **75%**。这就是 ZeRO 优化的重点。

### 12.2 数据并行的内存冗余

在标准数据并行 (DDP) 中，每个 GPU 持有**完整**的：
- 模型参数副本 (2Ψ)
- 梯度副本 (2Ψ)
- 优化器状态副本 (12Ψ)

如果有 N_d 个 GPU，总内存使用是 N_d × 16Ψ，但有效信息只有 16Ψ——**冗余度为 N_d 倍**。ZeRO 的目标就是消除这种冗余。

### 12.3 ZeRO Stage 1：优化器状态分片

```
传统 DDP (4 GPUs, 模型参数量 Ψ):
┌──────────┬──────────┬──────────┬──────────┐
│  GPU 0   │  GPU 1   │  GPU 2   │  GPU 3   │
│ Param:2Ψ │ Param:2Ψ │ Param:2Ψ │ Param:2Ψ │
│ Grad: 2Ψ │ Grad: 2Ψ │ Grad: 2Ψ │ Grad: 2Ψ │
│ Optim:12Ψ│ Optim:12Ψ│ Optim:12Ψ│ Optim:12Ψ│
│ 总计:16Ψ │ 总计:16Ψ │ 总计:16Ψ │ 总计:16Ψ │
└──────────┴──────────┴──────────┴──────────┘

ZeRO Stage 1 (4 GPUs):
┌──────────┬──────────┬──────────┬──────────┐
│  GPU 0   │  GPU 1   │  GPU 2   │  GPU 3   │
│ Param:2Ψ │ Param:2Ψ │ Param:2Ψ │ Param:2Ψ │
│ Grad: 2Ψ │ Grad: 2Ψ │ Grad: 2Ψ │ Grad: 2Ψ │
│ Optim:3Ψ │ Optim:3Ψ │ Optim:3Ψ │ Optim:3Ψ │  ← 12Ψ/4 = 3Ψ
│ 总计:7Ψ  │ 总计:7Ψ  │ 总计:7Ψ  │ 总计:7Ψ  │
└──────────┴──────────┴──────────┴──────────┘
```

**工作流程**：
1. 前向传播：正常执行（每个 GPU 有完整参数）
2. 反向传播：正常计算梯度
3. 梯度同步：AllReduce 聚合梯度（与 DDP 相同）
4. **参数更新**：每个 GPU 只更新**自己负责**的 1/N_d 参数
5. 参数广播：AllGather 将更新后的参数分发到所有 GPU

**内存公式**：

```
Stage 1 每 GPU 内存 = 2Ψ + 2Ψ + 12Ψ/N_d = 4Ψ + 12Ψ/N_d

当 N_d = 64: 4Ψ + 0.1875Ψ ≈ 4.2Ψ （vs DDP 的 16Ψ，节省约 4 倍）
```

**通信量**：与 DDP 完全相同（一次 AllReduce），因此**速度无损失**。

### 12.4 ZeRO Stage 2：梯度分片

在 Stage 1 基础上，进一步将梯度也分片：

```
ZeRO Stage 2 (4 GPUs):
┌──────────┬──────────┬──────────┬──────────┐
│  GPU 0   │  GPU 1   │  GPU 2   │  GPU 3   │
│ Param:2Ψ │ Param:2Ψ │ Param:2Ψ │ Param:2Ψ │
│ Grad:0.5Ψ│ Grad:0.5Ψ│ Grad:0.5Ψ│ Grad:0.5Ψ│  ← 2Ψ/4 = 0.5Ψ
│ Optim:3Ψ │ Optim:3Ψ │ Optim:3Ψ │ Optim:3Ψ │
│ 总计:5.5Ψ│ 总计:5.5Ψ│ 总计:5.5Ψ│ 总计:5.5Ψ│
└──────────┴──────────┴──────────┴──────────┘
```

**工作流程**的关键区别：
1. 反向传播中，梯度计算后**立即通过 Reduce-Scatter 进行分片聚合**
2. 每个 GPU 只保留自己负责的那部分聚合梯度
3. 不需要的梯度**立即释放**，节省内存

**内存公式**：

```
Stage 2 每 GPU 内存 = 2Ψ + (2Ψ + 12Ψ)/N_d = 2Ψ + 14Ψ/N_d

当 N_d = 64: 2Ψ + 0.22Ψ ≈ 2.2Ψ
```

**通信量**：用 Reduce-Scatter 替代 AllReduce 的前半部分，总通信量不变，仍与 DDP 持平。

### 12.5 ZeRO Stage 3：参数分片

Stage 3 是最激进的优化——连模型参数本身也分片：

```
ZeRO Stage 3 (4 GPUs):
┌──────────┬──────────┬──────────┬──────────┐
│  GPU 0   │  GPU 1   │  GPU 2   │  GPU 3   │
│Param:0.5Ψ│Param:0.5Ψ│Param:0.5Ψ│Param:0.5Ψ│  ← 2Ψ/4 = 0.5Ψ
│ Grad:0.5Ψ│ Grad:0.5Ψ│ Grad:0.5Ψ│ Grad:0.5Ψ│
│ Optim:3Ψ │ Optim:3Ψ │ Optim:3Ψ │ Optim:3Ψ │
│ 总计:4Ψ  │ 总计:4Ψ  │ 总计:4Ψ  │ 总计:4Ψ  │
└──────────┴──────────┴──────────┴──────────┘
```

**工作流程**：
1. **前向传播前**：需要用到某层参数时，通过 AllGather 从所有 GPU 收集完整参数
2. **前向计算**：用完整参数正常计算
3. **前向传播后**：**立即释放**非本 GPU 负责的参数分片
4. **反向传播前**：再次 AllGather 收集该层的完整参数
5. **反向计算**：用完整参数和激活值计算梯度
6. **反向传播后**：通过 Reduce-Scatter 分片聚合梯度，释放完整参数
7. **参数更新**：每个 GPU 只更新自己的 1/N_d 分片

```
前向传播（逐层执行）:

Layer 1:  AllGather → 计算 → 释放非本地参数
Layer 2:  AllGather → 计算 → 释放非本地参数
...
Layer L:  AllGather → 计算 → 释放非本地参数

反向传播（逆序逐层执行）:

Layer L:  AllGather → 计算梯度 → Reduce-Scatter → 释放
...
Layer 1:  AllGather → 计算梯度 → Reduce-Scatter → 释放
```

**内存公式**：

```
Stage 3 每 GPU 内存 = (2Ψ + 2Ψ + 12Ψ)/N_d = 16Ψ/N_d

当 N_d = 64: 16Ψ/64 = 0.25Ψ （vs DDP 的 16Ψ，节省 64 倍！）

但实际运行时的峰值内存更高，因为每层前向/反向时需要临时持有完整参数:
峰值内存 ≈ 16Ψ/N_d + 2Ψ (临时完整参数) + 激活值内存
```

**通信量分析**：

```
Stage 3 通信量 (每步):
- 前向: L 层 × AllGather = L × 2Ψ × (N_d-1)/N_d
- 反向: L 层 × AllGather + L 层 × Reduce-Scatter
       = L × 2Ψ × (N_d-1)/N_d + L × 2Ψ × (N_d-1)/N_d

总通信量 ≈ 3 × DDP 的通信量
```

**这就是 Stage 3 的代价：通信量增加约 1.5 倍**（DDP 只需要一次 AllReduce，Stage 3 需要前向 AllGather + 反向 AllGather + Reduce-Scatter）。

### 12.6 三个 Stage 的对比总结

| 指标 | Stage 1 | Stage 2 | Stage 3 |
|------|---------|---------|---------|
| 分片内容 | 优化器状态 | 优化器状态 + 梯度 | 优化器状态 + 梯度 + 参数 |
| 每 GPU 内存 (N_d→∞) | 4Ψ | 2Ψ | → 0 |
| 通信量 vs DDP | 1x | 1x | ~1.5x |
| 速度影响 | 几乎无 | 几乎无 | 有一定下降 (10-20%) |
| 最大可训模型 (8 GPU) | ~7B | ~14B | ~30B+ |
| 实现复杂度 | 低 | 中 | 高 |

### 12.7 FSDP：PyTorch 原生的 ZeRO Stage 3

FSDP (Fully Sharded Data Parallel) 是 PyTorch 对 ZeRO Stage 3 的原生实现，由 Meta 开发并集成到 PyTorch 核心库中。

#### 12.7.1 FSDP 的核心机制

```
FSDP 的生命周期 (每个训练步):

1. 前向传播:
   ┌─────────────────────────────────────────────────┐
   │ Layer i:                                        │
   │   pre_forward_hook:                             │
   │     AllGather → 收集完整参数                     │
   │     将 DTensor(sharded) → Tensor(full)           │
   │   forward():                                    │
   │     正常计算 Y = X·W + b                         │
   │   post_forward_hook:                            │
   │     释放非本地参数分片                            │
   │     将 Tensor(full) → DTensor(sharded)           │
   └─────────────────────────────────────────────────┘

2. 反向传播:
   ┌─────────────────────────────────────────────────┐
   │ Layer i (逆序):                                  │
   │   pre_backward_hook:                            │
   │     AllGather → 再次收集完整参数                  │
   │   backward():                                   │
   │     计算梯度 dL/dW, dL/dX                        │
   │   post_backward_hook:                           │
   │     Reduce-Scatter → 分片聚合梯度                │
   │     释放完整参数                                 │
   └─────────────────────────────────────────────────┘
```

#### 12.7.2 FSDP 的通信-计算重叠

FSDP 通过**预取 (Prefetching)** 隐藏通信延迟：

```
时间 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

默认流 (计算):  [Layer i 计算]     [Layer i+1 计算]     [Layer i+2 计算]
通信流:            [Layer i+1 AllGather]  [Layer i+2 AllGather]

↑ Layer i+1 的 AllGather 在 Layer i 计算时就已经开始了
```

在 FSDP2 中，预取策略进一步优化：
- CPU 线程在**独立的 CUDA stream** 上发起 AllGather
- 计算和通信在不同 stream 上并行执行
- 反向传播中也有类似的预取机制（按前向注册顺序的逆序执行）

#### 12.7.3 FSDP1 vs FSDP2

| 特性 | FSDP1 | FSDP2 |
|------|-------|-------|
| 参数表示 | FlatParameter（拉平拼接） | DTensor（分布式张量） |
| 分片方式 | 拉平 → 拼接 → 切分 | 按 dim-0 直接切分每个参数 |
| 状态字典 | 需要特殊处理 | 原生支持分片 state_dict |
| 内存管理 | 依赖 recordStream | 更高效的确定性内存管理 |
| API | 包裹整个模块 | fully_shard() 装饰器 |

#### 12.7.4 FSDP 与 DeepSpeed ZeRO 的关系

```
功能映射:
  DeepSpeed ZeRO Stage 1  ↔  FSDP(sharding_strategy=SHARD_GRAD_OP)  [近似]
  DeepSpeed ZeRO Stage 2  ↔  FSDP(sharding_strategy=SHARD_GRAD_OP)
  DeepSpeed ZeRO Stage 3  ↔  FSDP(sharding_strategy=FULL_SHARD)     [最常用]

额外功能:
  DeepSpeed: CPU/NVMe Offload, 更多内存优化选项
  FSDP: 与 PyTorch 生态深度集成, 更好的 torch.compile 支持
```

### 12.8 实际选择建议

```
模型规模判断树:

模型 < 10B 参数?
  ├─ Yes → 单节点 8 GPU
  │         ├─ DDP 能放下? → 使用 DDP（最简单，最快）
  │         └─ 放不下?    → ZeRO Stage 2 / FSDP(SHARD_GRAD_OP)
  │
  └─ No → 多节点
           ├─ 10B-100B → ZeRO Stage 3 / FSDP(FULL_SHARD)
           │              可选: + CPU Offload
           ├─ 100B-500B → ZeRO Stage 3 + TP + PP (3D 并行)
           └─ 500B+     → 全套: DP + TP + PP + EP + CP
```
