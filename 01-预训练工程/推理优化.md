# 推理优化

> 训练一次，推理无数次。推理优化直接影响模型的实际可用性和成本。

## 一、推理优化全景

```
推理优化
├── 模型压缩       → 量化 (Quantization)
├── KV Cache 优化  → PagedAttention, MLA, SnapKV
├── 解码加速       → 推测解码 (Speculative Decoding)
├── 批处理优化     → 连续批处理 (Continuous Batching)
└── Serving 框架   → vLLM, SGLang, TensorRT-LLM
```

## 二、量化 (Quantization)

### 核心思想
将模型权重从高精度（FP16/BF16）压缩到低精度（INT8/INT4/FP8），减少显存占用和计算量。

### 主流方法

#### GPTQ (2023)
- **论文**：[arXiv:2210.17323](https://arxiv.org/abs/2210.17323)
- **方法**：基于近似二阶信息 (Hessian) 的逐层量化
- **特点**：一次性 (one-shot)，不需要重训练
- **精度**：4-bit 为主，质量损失可接受

#### AWQ (Activation-aware Weight Quantization, 2024)
- **论文**：[arXiv:2306.00978](https://arxiv.org/abs/2306.00978)
- **方法**：根据激活值幅度识别重要通道，保护后再量化
- **特点**：同比特位宽下质量通常优于 GPTQ
- **广泛使用**，是当前实践中最受欢迎的量化方案之一

#### FP8 (8-bit Floating Point)
- NVIDIA Hopper (H100) 原生支持
- DeepSeek-V3 已在训练中使用
- 推理中：几乎无质量损失，2x 吞吐提升

#### 极端压缩 (2-bit)
- **QuIP#**：基于非相干性的量化
- **AQLM**：加性量化
- 质量损失明显，适用于资源极度受限场景

### 实践工具
| 工具 | 支持格式 | 特点 |
|------|---------|------|
| llama.cpp (GGUF) | Q4_K_M, Q5_K_M 等 | CPU + GPU 混合推理 |
| AutoGPTQ | GPTQ | GPU 推理 |
| AutoAWQ | AWQ | GPU 推理 |
| bitsandbytes | NF4, INT8 | QLoRA 训练 |

## 三、KV Cache 优化

### 问题
- Transformer 推理时，每个 token 的 Key/Value 需要缓存
- 长序列 + 大批次 → KV Cache 占用可达数十 GB

### 解决方案

#### PagedAttention (vLLM, 2023)
- **论文**：[arXiv:2309.06180](https://arxiv.org/abs/2309.06180)
- **核心思想**：像操作系统管理虚拟内存一样管理 KV Cache
- 将 KV Cache 分成固定大小的 "页 (page)"
- 消除内存碎片
- 支持跨请求共享 KV Cache（prefix caching）
- **这是 vLLM 的核心创新**

#### 架构级解决方案
- **MLA** (DeepSeek)：14x KV Cache 压缩
- **GQA** (Llama)：4-8x 压缩
- **MQA**：32x+ 压缩

#### KV Cache 驱逐策略
- **H2O (Heavy-Hitter Oracle)**：只保留最重要的 token 的 KV
- **SnapKV**：通过注意力模式识别关键位置
- **Scissorhands**：基于注意力分数的驱逐

## 四、推测解码 (Speculative Decoding)

### 核心思想
```
传统：大模型逐个生成 token → 慢
推测解码：
  1. 小模型（Draft Model）快速生成 N 个候选 token
  2. 大模型（Target Model）一次性验证所有 N 个 token
  3. 接受正确的前缀，拒绝错误的
  → 保证数学上与大模型单独生成完全等价
  → 但速度提升 2-3x
```

### 主要变体

| 方法 | 核心思想 | 优势 |
|------|---------|------|
| 标准推测解码 | 独立小模型做 draft | 通用 |
| Medusa | 给大模型加多个预测头 | 无需额外模型 |
| Eagle/Eagle-2 | 特征级自回归做 draft | 更高接受率 |
| SEQUOIA | 树状推测 + 动态树结构 | 自适应 |
| DeepSeek MTP | 用多 Token 预测头做 draft | 训练时自然获得 |

## 五、Serving 框架

### vLLM
- **最广泛使用**的开源 LLM Serving 框架
- 核心技术：PagedAttention、连续批处理、Prefix Caching
- 支持：TP、推测解码、LoRA
- **适合**：通用场景，社区活跃

### SGLang
- **论文**：[arXiv:2312.07104](https://arxiv.org/abs/2312.07104)
- 核心创新：**RadixAttention** — 自动复用请求间共享前缀的 KV Cache
- 支持结构化生成 (constrained decoding)
- **适合**：结构化输出、共享前缀多的场景
- 在很多基准上比 vLLM 更快

### TensorRT-LLM (NVIDIA)
- NVIDIA 官方优化框架
- 特点：FP8/INT4 量化、in-flight batching
- **通常吞吐量最高**（NVIDIA 硬件上）
- 灵活性不如 vLLM/SGLang

### 前沿趋势：Prefill-Decode 分离

```
传统方式：
  同一组 GPU 同时处理 Prefill (计算密集) 和 Decode (访存密集)

分离方式 (2024-2025)：
  Prefill GPU 集群 → 专用于处理 Prefill（高计算利用率）
  Decode GPU 集群 → 专用于生成（高访存带宽利用率）

代表工作：Mooncake (Moonshot AI)
优势：整体吞吐提升 30-50%
```

## 六、推理成本对比

| 优化手段 | 加速倍数 | 质量影响 | 实现难度 |
|---------|---------|---------|---------|
| FP8 量化 | ~2x | 几乎无 | 低（硬件支持） |
| INT4 量化 (AWQ) | ~3-4x | 轻微 | 低 |
| 推测解码 | ~2-3x | 无 | 中 |
| PagedAttention | ~2-4x (显存) | 无 | 低（用 vLLM） |
| Prefill-Decode 分离 | ~1.3-1.5x | 无 | 高（架构改造） |
| 多优化组合 | ~10x+ | 轻微 | 高 |
