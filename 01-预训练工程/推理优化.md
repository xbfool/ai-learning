# 推理优化

> 训练一次，推理无数次。推理优化直接影响模型的实际可用性和成本。

## 一、推理优化全景

```
推理优化
├── 模型压缩       → 量化 (Quantization)
├── KV Cache 优化  → PagedAttention, MLA, SnapKV
├── 解码加速       → 推测解码 (Speculative Decoding)
├── 批处理优化     → 连续批处理 (Continuous Batching)
└── Serving 框架   → vLLM, SGLang, TensorRT-LLM
```

## 二、量化 (Quantization)

### 核心思想
将模型权重从高精度（FP16/BF16）压缩到低精度（INT8/INT4/FP8），减少显存占用和计算量。

### 主流方法

#### GPTQ (2023)
- **论文**：[arXiv:2210.17323](https://arxiv.org/abs/2210.17323)
- **方法**：基于近似二阶信息 (Hessian) 的逐层量化
- **特点**：一次性 (one-shot)，不需要重训练
- **精度**：4-bit 为主，质量损失可接受

#### AWQ (Activation-aware Weight Quantization, 2024)
- **论文**：[arXiv:2306.00978](https://arxiv.org/abs/2306.00978)
- **方法**：根据激活值幅度识别重要通道，保护后再量化
- **特点**：同比特位宽下质量通常优于 GPTQ
- **广泛使用**，是当前实践中最受欢迎的量化方案之一

#### FP8 (8-bit Floating Point)
- NVIDIA Hopper (H100) 原生支持
- DeepSeek-V3 已在训练中使用
- 推理中：几乎无质量损失，2x 吞吐提升

#### 极端压缩 (2-bit)
- **QuIP#**：基于非相干性的量化
- **AQLM**：加性量化
- 质量损失明显，适用于资源极度受限场景

### 实践工具
| 工具 | 支持格式 | 特点 |
|------|---------|------|
| llama.cpp (GGUF) | Q4_K_M, Q5_K_M 等 | CPU + GPU 混合推理 |
| AutoGPTQ | GPTQ | GPU 推理 |
| AutoAWQ | AWQ | GPU 推理 |
| bitsandbytes | NF4, INT8 | QLoRA 训练 |

## 三、KV Cache 优化

### 问题
- Transformer 推理时，每个 token 的 Key/Value 需要缓存
- 长序列 + 大批次 → KV Cache 占用可达数十 GB

### 解决方案

#### PagedAttention (vLLM, 2023)
- **论文**：[arXiv:2309.06180](https://arxiv.org/abs/2309.06180)
- **核心思想**：像操作系统管理虚拟内存一样管理 KV Cache
- 将 KV Cache 分成固定大小的 "页 (page)"
- 消除内存碎片
- 支持跨请求共享 KV Cache（prefix caching）
- **这是 vLLM 的核心创新**

#### 架构级解决方案
- **MLA** (DeepSeek)：14x KV Cache 压缩
- **GQA** (Llama)：4-8x 压缩
- **MQA**：32x+ 压缩

#### KV Cache 驱逐策略
- **H2O (Heavy-Hitter Oracle)**：只保留最重要的 token 的 KV
- **SnapKV**：通过注意力模式识别关键位置
- **Scissorhands**：基于注意力分数的驱逐

## 四、推测解码 (Speculative Decoding)

### 核心思想
```
传统：大模型逐个生成 token → 慢
推测解码：
  1. 小模型（Draft Model）快速生成 N 个候选 token
  2. 大模型（Target Model）一次性验证所有 N 个 token
  3. 接受正确的前缀，拒绝错误的
  → 保证数学上与大模型单独生成完全等价
  → 但速度提升 2-3x
```

### 主要变体

| 方法 | 核心思想 | 优势 |
|------|---------|------|
| 标准推测解码 | 独立小模型做 draft | 通用 |
| Medusa | 给大模型加多个预测头 | 无需额外模型 |
| Eagle/Eagle-2 | 特征级自回归做 draft | 更高接受率 |
| SEQUOIA | 树状推测 + 动态树结构 | 自适应 |
| DeepSeek MTP | 用多 Token 预测头做 draft | 训练时自然获得 |

## 五、Serving 框架

### vLLM
- **最广泛使用**的开源 LLM Serving 框架
- 核心技术：PagedAttention、连续批处理、Prefix Caching
- 支持：TP、推测解码、LoRA
- **适合**：通用场景，社区活跃

### SGLang
- **论文**：[arXiv:2312.07104](https://arxiv.org/abs/2312.07104)
- 核心创新：**RadixAttention** — 自动复用请求间共享前缀的 KV Cache
- 支持结构化生成 (constrained decoding)
- **适合**：结构化输出、共享前缀多的场景
- 在很多基准上比 vLLM 更快

### TensorRT-LLM (NVIDIA)
- NVIDIA 官方优化框架
- 特点：FP8/INT4 量化、in-flight batching
- **通常吞吐量最高**（NVIDIA 硬件上）
- 灵活性不如 vLLM/SGLang

### 前沿趋势：Prefill-Decode 分离

```
传统方式：
  同一组 GPU 同时处理 Prefill (计算密集) 和 Decode (访存密集)

分离方式 (2024-2025)：
  Prefill GPU 集群 → 专用于处理 Prefill（高计算利用率）
  Decode GPU 集群 → 专用于生成（高访存带宽利用率）

代表工作：Mooncake (Moonshot AI)
优势：整体吞吐提升 30-50%
```

## 七、深度专题：vLLM PagedAttention

> 论文：[Efficient Memory Management for Large Language Model Serving with PagedAttention (arXiv:2309.06180)](https://arxiv.org/abs/2309.06180)，发表于 SOSP 2023。

### 7.1 问题背景：KV Cache 的内存管理困境

在自回归推理中，每生成一个 token 都需要访问之前所有 token 的 Key 和 Value 向量（即 KV Cache）。对于一个 13B 参数的模型，单个序列的 KV Cache 可以达到数百 MB；当需要同时服务数百个请求时，KV Cache 的总占用量可轻松超过数十 GB，成为 GPU 显存的主要瓶颈。

传统 LLM Serving 系统（如 HuggingFace Transformers、FasterTransformer）的做法是为每个请求**预分配一段连续的显存**来存储 KV Cache。这种方式带来三种严重的内存浪费：

1. **预留浪费（Reservation Waste）**：由于输出长度事先未知，系统必须按最大可能长度预留内存，导致大量预留但未使用的空间。
2. **内部碎片（Internal Fragmentation）**：预分配的连续块往往比实际使用量大，块内的剩余空间被浪费。
3. **外部碎片（External Fragmentation）**：不同请求的内存块大小各异，释放后留下的间隙无法被新请求有效利用。

实测表明，在传统系统中，**60%~80% 的 KV Cache 显存被浪费**。

### 7.2 核心思想：操作系统虚拟内存的类比

PagedAttention 的核心灵感来自操作系统中**虚拟内存与分页（Virtual Memory & Paging）**机制。类比如下：

| 操作系统概念 | PagedAttention 对应 |
|------------|-------------------|
| 进程（Process） | 推理请求（Request/Sequence） |
| 虚拟地址空间 | 逻辑 KV Block 序列 |
| 物理页帧（Page Frame） | 物理 KV Block（GPU 显存中固定大小的块） |
| 页表（Page Table） | Block Table（逻辑块→物理块的映射表） |
| 字节（Byte） | Token 的 KV 向量 |
| 按需分页（Demand Paging） | 按需分配物理 KV Block |
| 写时复制（Copy-on-Write） | 共享 Block 被修改时才复制 |

关键洞察：**KV Cache 不需要存储在连续的物理内存中**。只要有一张映射表记录每个逻辑块对应的物理位置，注意力计算就可以正确地从非连续内存中读取 KV 向量。

### 7.3 Block Table 与内存映射机制

#### 数据结构

```
物理 KV Block Pool（GPU 显存）
┌──────┬──────┬──────┬──────┬──────┬──────┐
│ PB-0 │ PB-1 │ PB-2 │ PB-3 │ PB-4 │ PB-5 │ ...
└──────┴──────┴──────┴──────┴──────┴──────┘

请求 A 的 Block Table:
  逻辑块 0 → 物理块 PB-1  (filled: 16/16)
  逻辑块 1 → 物理块 PB-4  (filled: 16/16)
  逻辑块 2 → 物理块 PB-2  (filled: 9/16)   ← 最后一个块部分填充

请求 B 的 Block Table:
  逻辑块 0 → 物理块 PB-3  (filled: 16/16)
  逻辑块 1 → 物理块 PB-5  (filled: 3/16)
```

- 每个物理块（Physical Block）大小固定，通常存储 16 个 token 的 KV 向量。对于 13B 模型，每个块约 12.8KB。
- Block Table 的每个条目记录：(1) 对应的物理块编号；(2) 已填充的 token 数量。
- 物理块按需分配：只有当上一个块填满后，才为下一个逻辑块分配新的物理块。

#### 注意力计算的 CUDA 内核

传统注意力实现假设 KV Cache 存储在连续内存中。PagedAttention 需要**定制 CUDA 内核**，该内核：

1. 读取 Block Table，获取每个逻辑块对应的物理地址。
2. 从非连续的物理块中**逐块读取** KV 向量。
3. 对每个块分别计算注意力分数，最后合并结果。

这种间接访问带来的额外开销约为 **5%~10% 的内核延迟**，但换来的内存效率提升远超这一代价。端到端实现中，通过算子融合（Kernel Fusion），分页的总开销约为 **~20%**。

### 7.4 内存共享机制

PagedAttention 的分页设计天然支持跨序列的内存共享：

#### 并行采样（Parallel Sampling）

当同一个 prompt 需要生成多个不同的输出（如 temperature sampling 生成 N 个候选）时：

```
共享 Prompt "Tell me a joke about..."

序列 1: [共享前缀] → "Why did the chicken..."
序列 2: [共享前缀] → "A programmer walks into..."
序列 3: [共享前缀] → "What do you call..."

内存布局：
Prompt 的 KV Block（PB-0, PB-1, PB-2）← 三个序列共享，只存一份
序列 1 独有的 Block: PB-5, PB-6
序列 2 独有的 Block: PB-7
序列 3 独有的 Block: PB-8, PB-9
```

所有序列的 Block Table 中，Prompt 部分的逻辑块都映射到**同一组物理块**，只有各自生成的部分才分配独立的物理块。

#### 写时复制（Copy-on-Write, CoW）

系统为每个物理块维护一个**引用计数（Reference Count）**。当某个序列需要修改一个被多个序列共享的块时：

1. 检查该物理块的引用计数。
2. 如果引用计数 > 1，分配一个新的物理块，复制原块内容，更新该序列的 Block Table。
3. 将原块的引用计数减 1。

这与操作系统中 `fork()` 后的 Copy-on-Write 机制完全一致。

#### 效果

在并行采样和 Beam Search 场景中，PagedAttention 的内存共享将内存使用量**降低最多 55%**，从而将吞吐量**提升最多 2.2 倍**。

### 7.5 内存效率量化

| 系统 | KV Cache 内存浪费率 |
|------|-------------------|
| 传统预分配系统 | 60%~80% |
| PagedAttention (vLLM) | **< 4%** |

浪费只发生在每个序列的**最后一个物理块**中（部分填充），最多浪费 `block_size - 1` 个 token 的空间。对于 block_size=16，平均浪费仅 8 个 token 的空间——在千 token 级别的序列中几乎可以忽略。

这种极低的浪费率意味着系统可以同时服务更多的请求（更大的 batch size），从而显著提升 GPU 利用率和整体吞吐量。

## 八、深度专题：SGLang RadixAttention

> 论文：[SGLang: Efficiently Programming Large Language Models using SGLang (arXiv:2312.07104)](https://arxiv.org/abs/2312.07104)，发表于 NeurIPS 2024。

### 8.1 问题背景：复杂 LLM 程序中的前缀共享

现代 LLM 应用不是简单的单轮问答，而是复杂的**多轮交互程序**：多轮对话、思维链（Chain-of-Thought）、树状搜索（Tree-of-Thought）、Few-shot 学习、多模态推理等。这些程序的一个共同特征是**大量请求共享相同的前缀**（System Prompt、Few-shot 示例、对话历史等）。

vLLM 的 PagedAttention 虽然解决了单请求内的内存碎片问题，但在**跨请求的前缀共享**方面能力有限——它需要显式的 API 调用来标记共享前缀，无法自动检测和复用。

### 8.2 核心创新：基于 Radix Tree 的 KV Cache 管理

#### 什么是 Radix Tree

Radix Tree（基数树，也叫压缩前缀树）是一种空间高效的树形数据结构。与普通前缀树（Trie）不同，Radix Tree 的边可以标注**多个元素的序列**，而不仅是单个元素。这使得它特别适合存储和查找有大量公共前缀的字符串集合。

#### RadixAttention 的设计

在 SGLang 中，KV Cache 被组织为一棵 Radix Tree：

```
                      [root]
                     /      \
        [System Prompt]     [另一个System Prompt]
           /       \
    [Few-shot例子]  [对话历史A]
       /    \            \
  [查询1]  [查询2]     [回复A + 查询B]
```

- **键（Key）**：Token 序列（prompt 的 token ID 列表）。
- **值（Value）**：对应的 KV Cache 张量（存储在 GPU 显存中）。
- **边（Edge）**：Token 子序列，可以是任意长度。
- **节点（Node）**：表示前缀分叉点或叶子。
- **节点状态**：
  - 绿色 → 新插入的节点
  - 蓝色 → 被命中/复用的缓存节点
  - 红色 → 被驱逐的节点

#### 核心操作

**前缀匹配（Prefix Match）**：
```
新请求: [System Prompt] + [Few-shot例子] + [新查询3]
         ↓
Radix Tree 中搜索 → 匹配到 [System Prompt] + [Few-shot例子]
         ↓
复用已缓存的 KV Cache → 只需为 [新查询3] 计算 Prefill
```

这种自动前缀匹配意味着：如果 System Prompt 有 1000 个 token，且 100 个请求共享同一 System Prompt，系统**只需计算一次** System Prompt 的 KV Cache，节省了 99 次重复计算。

**插入（Insert）**：生成完成后，将新生成的 token 序列及其 KV Cache 插入树中，供后续请求复用。

**驱逐（Eviction）**：当 GPU 显存不足时，采用 **LRU（最近最少使用）策略**从叶子节点开始递归驱逐。优先驱逐最长时间未被访问的缓存。

### 8.3 与 vLLM Prefix Caching 的对比

| 特性 | vLLM (Block-Level Hash) | SGLang (Token-Level Radix Tree) |
|------|------------------------|-------------------------------|
| 粒度 | Block 级（16 tokens 一组） | Token 级（精确到单个 token） |
| 匹配方式 | Hash 匹配 | 树结构前缀匹配 |
| 自动化程度 | 需显式启用 `--enable-prefix-caching` | 默认自动开启 |
| 部分前缀复用 | 以 Block 为最小单位 | 可精确到任意 token 位置 |
| 多轮对话优势 | 中等 | 显著（逐轮增量缓存） |

### 8.4 结构化生成：压缩有限状态机（Compressed FSM）

SGLang 的另一项重要创新是用**压缩有限状态机（Compressed Finite State Machine）**加速结构化输出生成（如 JSON、正则表达式匹配）。

#### 传统约束解码的问题

传统的约束解码（如 Outlines、Guidance）在每一步只推进一个 token，即使约束已经确定了接下来多个 token 的内容（例如 JSON 中的 `"name": "`），仍然逐 token 解码——效率极低。

#### 压缩 FSM 的做法

1. 将约束（JSON Schema、正则表达式等）编译成有限状态机（FSM）。
2. **压缩**：识别 FSM 中的"确定性路径"——当某个状态只有唯一的后继转移时，将连续的单一转移边合并为一条边。
3. **跳跃前进（Jump Forward）**：在解码过程中，遇到确定性路径时直接跳过，一次填充多个 token，无需逐个调用模型。

```
原始 FSM:
  S0 --"{"-- S1 --'"'-- S2 --'n'-- S3 --'a'-- S4 --'m'-- S5 --'e'-- S6 --'"'-- S7

压缩后:
  S0 --'{"name"'-- S7    ← 一步到位
```

效果：结构化解码速度提升最高 **2.5 倍**，在某些场景下**约束解码甚至比无约束解码更快**。

### 8.5 运行时架构

SGLang 的运行时由以下核心组件构成：

```
SGLang 运行时架构
┌─────────────────────────────────────────┐
│            Frontend (Python DSL)         │
│  - gen()   : 生成文本                    │
│  - select(): 从候选中选择               │
│  - fork()  : 分叉执行流                 │
│  - [+]     : 拼接 prompt               │
└───────────┬─────────────────────────────┘
            │ 解释器 (Interpreter)
            ↓
┌─────────────────────────────────────────┐
│         Stream Executor                  │
│  将 DSL 调用翻译为对 Runtime 的请求      │
└───────────┬─────────────────────────────┘
            ↓
┌─────────────────────────────────────────┐
│           Runtime Engine                 │
│  ┌──────────────┐  ┌──────────────────┐ │
│  │ RadixAttention│  │ Compressed FSM   │ │
│  │ (KV Cache管理)│  │ (结构化解码加速)  │ │
│  └──────────────┘  └──────────────────┘ │
│  ┌──────────────┐  ┌──────────────────┐ │
│  │ Continuous    │  │ Tensor Parallel  │ │
│  │ Batching     │  │                  │ │
│  └──────────────┘  └──────────────────┘ │
└─────────────────────────────────────────┘
```

与现有系统（Guidance + llama.cpp、Outlines + vLLM）相比，SGLang 在复杂 LLM 程序上实现了最高 **5 倍**的吞吐量提升。

## 九、深度专题：推测解码的数学原理

> 原始论文：
> - [Fast Inference from Transformers via Speculative Decoding (Leviathan et al., 2023)](https://arxiv.org/abs/2211.17192)，ICML 2023
> - [Accelerating Large Language Model Decoding with Speculative Sampling (Chen et al., 2023)](https://arxiv.org/abs/2302.01318)，DeepMind

### 9.1 核心问题：自回归解码的延迟瓶颈

自回归解码的根本问题在于：**生成 N 个 token 需要 N 次串行的前向传播**。每次前向传播都需要加载完整的模型权重到 GPU 计算单元中，但只产出 1 个 token——计算利用率极低（decode 阶段通常为 memory-bandwidth bound）。

推测解码的核心洞察：**大模型验证多个 token 的代价与生成 1 个 token 几乎相同**（因为 Prefill 可以并行处理多个 token 位置），但传统解码每次只利用了这一次前向传播的 1/N 的能力。

### 9.2 Draft-Verify 循环

推测解码的完整工作流程：

```
循环开始:
  ┌─ Step 1: Draft (草稿生成)
  │   小模型 Mq 自回归地生成 γ 个候选 token:
  │     x₁ ~ q(·|prefix)
  │     x₂ ~ q(·|prefix, x₁)
  │     ...
  │     xᵧ ~ q(·|prefix, x₁, ..., xᵧ₋₁)
  │
  ├─ Step 2: Verify (目标模型验证)
  │   大模型 Mp 对 [prefix, x₁, ..., xᵧ] 执行一次前向传播
  │   → 同时获得 γ+1 个位置的概率分布:
  │     p(·|prefix), p(·|prefix,x₁), ..., p(·|prefix,x₁,...,xᵧ)
  │
  ├─ Step 3: Accept/Reject (接受/拒绝判定)
  │   对每个位置 i = 1, ..., γ，依次判定:
  │     以概率 min(1, p(xᵢ)/q(xᵢ)) 接受 token xᵢ
  │     若接受 → 继续检查下一个
  │     若拒绝 → 从修正分布中重采样，停止本轮
  │
  └─ Step 4: Bonus Token (额外 token)
      如果所有 γ 个 token 都被接受，
      → 从 p(·|prefix, x₁, ..., xᵧ) 中再采样 1 个 token
循环结束 → 回到 Step 1
```

### 9.3 无损性的数学证明

推测解码最关键的性质：**生成的 token 分布与目标模型单独生成的分布完全一致**。这不是近似，而是数学上精确的等价。

#### 接受准则

设 $q(x)$ 为小模型在某位置的输出概率，$p(x)$ 为大模型在同一位置的输出概率。对于小模型采样得到的 token $x$：

```
接受概率 = min(1, p(x) / q(x))
```

- 当 $p(x) \geq q(x)$ 时：无条件接受（大模型比小模型更倾向选这个 token）。
- 当 $p(x) < q(x)$ 时：以概率 $p(x)/q(x)$ 接受（小模型过度自信时需要概率性拒绝）。

#### 拒绝后的修正分布

当 token 被拒绝时，从以下修正分布中重新采样：

```
p'(x) = norm(max(0, p(x) - q(x)))
       = max(0, p(x) - q(x)) / Σ_x max(0, p(x) - q(x))
```

#### 为什么这保证了无损

对于任意 token $x$，最终被输出的概率为：

```
P(output = x) = P(接受 x) + P(拒绝后重采样到 x)
              = q(x) · min(1, p(x)/q(x)) + β · p'(x)
```

其中 $β = Σ_x q(x) · max(0, 1 - p(x)/q(x))$ 是拒绝概率。

**展开验证**：
- 当 $p(x) \geq q(x)$：接受贡献 = $q(x) · 1 = q(x)$，修正分布中 $p'(x) = 0$
- 当 $p(x) < q(x)$：接受贡献 = $q(x) · p(x)/q(x) = p(x)$
- 两种情况下：$P(output = x) = min(p(x), q(x)) + β · p'(x) = p(x)$ ✓

因此，**无论小模型的质量如何，最终输出的分布始终精确等于目标大模型的分布**。小模型的质量只影响**接受率**（即速度），不影响**输出质量**。

### 9.4 期望加速分析

假设每个 token 的接受率为 $α$（独立同分布假设），每轮猜测 $γ$ 个 token，则每轮期望接受的 token 数量为：

```
E[接受 token 数] = (1 - α^(γ+1)) / (1 - α)
```

当 $α → 1$ 时（小模型与大模型高度一致），每轮接近接受 $γ+1$ 个 token。当 $α → 0$ 时（小模型完全不靠谱），每轮只接受 1 个 token（退化为正常解码，但多了小模型的开销）。

实际加速倍数还需考虑小模型与大模型的运行时间比 $c$：

```
加速倍数 ≈ E[接受 token 数] / (c · γ + 1)
```

其中 $c · γ$ 是小模型生成 $γ$ 个 token 的相对时间，$1$ 是大模型验证一次的时间。实践中，当 $α \geq 0.6$、$γ \geq 5$、$c \leq 0.1$ 时，可以实现 **2~3 倍加速**。

### 9.5 树状推测验证（Tree-based Speculative Decoding）

标准推测解码每轮只猜测一条序列。**树状推测**（SpecInfer、SEQUOIA）将猜测扩展为**树状结构**，同时探索多个可能的分支：

```
标准推测（链状）:
  prefix → a → b → c → d

树状推测:
  prefix → a → b → c
              ↘ d → e
          ↘ f → g
              ↘ h
```

#### 验证机制

目标大模型对整棵树执行一次前向传播（通过特殊的 **Tree Attention Mask** 实现，确保每个节点只能看到其祖先节点），同时获得所有节点的验证概率。

#### SEQUOIA 的关键改进

1. **无放回采样（Sampling without Replacement）**：从草稿模型采样时不重复——避免在同一分支上犯两次相同的错误。
2. **动态树结构**：使用动态规划根据当前的 token 接受率 $α$ 和计算预算自适应地选择最优树结构。
3. **数学保证**：即使使用树状结构和无放回采样，仍然严格保持目标模型的输出分布。

效果：在 Llama-2-70B + Llama-2-7B 的配置下，SEQUOIA 相比标准推测解码进一步提升 **30%~50%** 的速度。

## 十、深度专题：AWQ 激活感知权重量化

> 论文：[AWQ: Activation-aware Weight Quantization for LLM Compression and Acceleration (arXiv:2306.00978)](https://arxiv.org/abs/2306.00978)，发表于 MLSys 2024。

### 10.1 核心观察：不是所有权重同等重要

AWQ 论文的出发点是一个关键实验发现：**LLM 中只有约 1% 的权重通道（channels）对模型输出质量至关重要**。跳过这 1% 的权重不量化（保持 FP16），即可将量化后的困惑度（perplexity）大幅降低。

但问题是：**混合精度存储（部分 FP16 + 部分 INT4）对硬件非常不友好**，会严重降低推理速度。AWQ 需要一种方法，在保持统一低精度的同时，给予重要权重更好的保护。

### 10.2 如何识别重要通道：看激活，不看权重

直觉上可能认为"数值大的权重更重要"，但 AWQ 论文证明这是错误的。**重要性取决于权重被乘以的激活值的幅度**。

量化误差分析：

```
输出误差 = |W · X - Q(W) · X|
         = |X| · |W - Q(W)|
         ↑     ↑
    激活幅度  量化误差

→ 激活值越大的通道，其权重的量化误差对输出的影响越大
→ 应该优先保护激活值大的通道对应的权重
```

具体方法：在校准数据集上运行模型，统计每个输入通道的激活值的**平均绝对值**（或高百分位值），以此作为该通道的重要性度量。

### 10.3 等价变换与逐通道缩放

AWQ 的核心技巧是**逐通道缩放（Per-channel Scaling）的等价变换**：

```
原始计算:  Y = X · W

等价变换:  Y = (X · diag(s⁻¹)) · (diag(s) · W)
                    ↑                    ↑
              激活缩小 s 倍         权重放大 s 倍

其中 s 是逐通道的缩放因子向量
```

由于 $diag(s^{-1}) \cdot diag(s) = I$，变换前后的数学结果完全相同。但量化是在变换后的权重 $diag(s) \cdot W$ 上进行的：

```
量化后:  Y ≈ (X · diag(s⁻¹)) · Q(diag(s) · W)
```

对于重要通道（$s_i$ 大），权重被放大后量化，**相对量化误差减小**。

直觉解释：假设某权重原值为 0.001，量化步长为 0.01，则量化误差可达 100%。如果先将其放大 100 倍变为 0.1，量化步长仍为 0.01，量化误差降为 10%。最后乘以 $s^{-1} = 1/100$ 恢复正确量级。

### 10.4 最优缩放因子的搜索

缩放因子 $s$ 的选择需要平衡：
- $s$ 太大：保护了重要通道，但放大了非重要通道的误差。
- $s$ 太小：保护不够。

AWQ 的解决方案：

```
s = activation_magnitude ^ α
```

其中 `activation_magnitude` 是通过校准数据集统计得到的逐通道激活幅度，$α \in [0, 1]$ 是超参数：
- $α = 0$：不缩放（等于普通量化）。
- $α = 1$：缩放因子完全等于激活幅度（最大程度保护重要通道）。
- 实践中最优 $α$ 通常在 **0.5 附近**，通过 Grid Search 在校准集上搜索得到。

搜索过程对每一层独立进行，优化目标是**最小化该层量化前后的输出均方误差**。

### 10.5 AWQ vs GPTQ 详细对比

| 维度 | AWQ | GPTQ |
|------|-----|------|
| **核心方法** | 激活感知的逐通道缩放 + 统一量化 | 基于 Hessian 近似的逐层最优重构 |
| **数学基础** | 等价变换减小重要通道的相对量化误差 | 二阶泰勒展开近似的最优权重舍入 |
| **校准方式** | 收集激活统计量（前向传播） | 计算 Hessian 矩阵（需要梯度信息） |
| **计算成本** | 低（只需前向传播 + Grid Search） | 高（逐列逐层求解最优量化） |
| **泛化性** | 好（不依赖反向传播，不过拟合校准集） | 较差（可能过拟合校准数据的分布） |
| **质量（PPL）** | 同比特位宽下通常略优 | 良好，但在某些模型上不如 AWQ |
| **推理速度** | 1.45x faster than GPTQ | 基准 |
| **vs FP16** | 1.85x faster | ~1.3x faster |
| **适用场景** | 通用，尤其适合跨域部署 | 对单一域微调的模型效果好 |

### 10.6 实践影响

AWQ 已成为当前最广泛使用的 LLM 量化方法之一：
- **TheBloke** 在 HuggingFace 上发布的大量量化模型默认提供 AWQ 版本。
- **vLLM** 和 **SGLang** 原生支持 AWQ 格式的模型加载。
- 4-bit AWQ 在 Llama-2-70B 上的困惑度增加 < 0.5（vs FP16），但显存节省约 75%。

## 十一、深度专题：Prefill-Decode 分离架构（Mooncake）

> 论文：[Mooncake: A KVCache-centric Disaggregated Architecture for LLM Serving (arXiv:2407.00079)](https://arxiv.org/abs/2407.00079)，获得 FAST 2025 最佳论文奖。
> Mooncake 是 Moonshot AI（月之暗面）为 Kimi 大模型服务构建的推理平台。

### 11.1 为什么需要分离 Prefill 和 Decode

Prefill（预填充）和 Decode（逐步生成）是 LLM 推理的两个阶段，它们具有**截然不同的计算特性**：

| 特性 | Prefill 阶段 | Decode 阶段 |
|------|-------------|------------|
| **计算模式** | 计算密集型（Compute-bound） | 访存密集型（Memory-bandwidth-bound） |
| **并行度** | 高（所有 prompt token 同时处理） | 低（每次仅生成 1 个 token） |
| **GPU 利用瓶颈** | 算力（FLOPS） | 显存带宽（GB/s） |
| **关联指标** | TTFT（首 Token 时延） | TBT/ITL（Token 间时延） |
| **延迟特征** | 单次耗时长但可并行 | 每次快但需要串行执行多次 |

在传统的**混合部署**（Prefill 和 Decode 共享同一组 GPU）中，两个阶段互相干扰：

```
问题1: Prefill 占用 GPU 算力 → Decode 请求排队 → TBT 飙升
问题2: Decode 占用大量 KV Cache 显存 → Prefill 无法处理大 batch → TTFT 增加
问题3: 资源配比固定 → 难以独立扩缩容
```

### 11.2 Mooncake 架构总览

```
┌──────────────────────────────────────────────────────┐
│                    Conductor（全局调度器）              │
│  - 请求分发                                           │
│  - KVCache 分布感知                                   │
│  - SLO 约束优化                                       │
└──────────┬────────────────────────┬──────────────────┘
           │                        │
    ┌──────▼──────┐         ┌──────▼──────┐
    │  Prefill     │         │  Decode     │
    │  GPU 集群    │  KV传输  │  GPU 集群   │
    │  (高算力)    │ ◄──────► │ (高带宽)    │
    └──────┬──────┘         └──────┬──────┘
           │                        │
    ┌──────▼────────────────────────▼──────┐
    │         Mooncake Store               │
    │    (分布式 KVCache 存储引擎)          │
    │  ┌────────┐ ┌────────┐ ┌──────────┐ │
    │  │GPU VRAM│ │CPU DRAM│ │ NVMe SSD │ │
    │  └────────┘ └────────┘ └──────────┘ │
    │      Transfer Engine (RDMA)          │
    └──────────────────────────────────────┘
```

### 11.3 核心组件详解

#### Conductor（全局调度器）

Conductor 是 Mooncake 的"大脑"，负责将每个请求分配到一对 Prefill + Decode 实例。其调度算法考虑：

1. **KVCache 亲和性**：优先选择已经缓存了该请求前缀 KV Cache 的 Prefill 实例（减少传输开销）。
2. **负载均衡**：避免热点实例过载。
3. **SLO 约束**：确保 TTFT 和 TBT 满足服务等级目标。

调度算法（Cache-Aware Prefill Scheduling）：
```
对于新请求 R：
  1. 将输入 token 切分为固定大小的 block，计算每个 block 的 hash key
  2. 遍历所有 Prefill 实例，计算每个实例已缓存的前缀匹配长度
  3. 选择匹配最长 + 负载最低 + 满足 SLO 的实例
  4. 将请求发送到选定的 Prefill 实例
```

#### Mooncake Store（分布式 KVCache 存储引擎）

这是 Mooncake 最核心的组件，利用集群中**未被充分利用的异构存储资源**构建分布式 KVCache 池：

- **GPU VRAM**：最快，但容量有限。
- **CPU DRAM**：容量大，延迟适中。
- **NVMe SSD**：容量最大，用于冷数据。

通过分层存储和智能放置，Mooncake Store 可以缓存远超 GPU 显存容量的 KV Cache，实现跨请求的大规模前缀复用。

#### Transfer Engine（传输引擎）

支持多种传输协议：
- **RDMA**（Remote Direct Memory Access）：GPU 间直接传输，延迟极低（< 10μs）。
- **TCP**：适用于跨数据中心场景。
- **CXL/共享内存**：同节点内超低延迟。
- **NVMe over Fabric**：远程 SSD 访问。

Mooncake 的互联网络带宽可达 **8×400 Gbps RDMA**，确保 KV Cache 传输不成为瓶颈。

### 11.4 请求处理流程

```
用户请求到达
    │
    ▼
[Conductor] 选择 Prefill 实例 P 和 Decode 实例 D
    │
    ├─► Step 1: KVCache 预热
    │   从 Mooncake Store 中查找可复用的前缀 KVCache
    │   通过 RDMA 传输到 P 的 GPU 显存
    │
    ├─► Step 2: 分块 Prefill（Chunked Pipeline Parallelism）
    │   将长 prompt 切分为多个 chunk
    │   Chunk 1 Prefill 完成 → 立即将 KV 流式传输到 D
    │   Chunk 2 Prefill 完成 → 继续流式传输
    │   ... （流水线并行，减少 TTFT）
    │
    ├─► Step 3: Decode
    │   D 接收到足够的 KV 后开始 Decode
    │   加入 D 的 Continuous Batching 队列
    │   逐 token 生成直到结束
    │
    └─► Step 4: KVCache 回写
        生成结束后，将完整的 KVCache 存入 Mooncake Store
        供后续相同/相似前缀的请求复用
```

**分块流水线并行（Chunked Pipeline Parallelism, CPP）**是关键创新：不需要等待整个 prompt 的 Prefill 完成才开始 Decode，而是每完成一个 chunk 就立即传输——这对长上下文（如 128K token）请求的 TTFT 至关重要。

### 11.5 性能表现

在使用 Kimi 真实线上流量进行测试：

| 指标 | 提升幅度 |
|------|---------|
| 有效请求容量 | **+59% ~ +498%**（vs 基线方法） |
| SLO 达标率 | 100%（TTFT 和 TBT 均满足约束） |
| GPU 利用率 | Prefill 集群计算利用率显著提升 |
| KVCache 复用率 | 高（得益于 Mooncake Store 的大容量缓存） |

生产规模：Mooncake 已在**数千节点**上运行，每天处理超过 **1000 亿 token**。

### 11.6 与其他分离方案的对比

| 方案 | KVCache 管理 | 传输机制 | 调度策略 |
|------|-------------|---------|---------|
| Mooncake (Moonshot AI) | 分布式存储（GPU+CPU+SSD） | RDMA 多协议 | Cache-Aware + SLO 约束 |
| DistServe (Peking Univ.) | GPU 显存 | GPU 间直传 | 基于负载的分配 |
| Splitwise (Microsoft) | GPU 显存 | NVLink/PCIe | 固定比例分配 |
| TetriInfer | GPU 显存 | 自定义协议 | 动态分区 |

Mooncake 的独特优势在于**将 KVCache 提升为第一等公民**，通过利用闲置的 CPU、DRAM、SSD 资源构建分布式缓存池，在存储换计算（Trading Storage for Computation）的理念下实现了显著的效率提升。
