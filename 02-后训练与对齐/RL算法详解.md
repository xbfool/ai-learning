# RL 算法详解

> 强化学习 (RL) 是当前大模型后训练的核心手段。本文详细对比各算法原理与适用场景。

## 一、算法全景

```
RL for LLM
├── 在线 RL (Online RL) — 训练中生成新数据
│   ├── PPO          → 经典 RLHF，需要 Critic
│   ├── GRPO         → 当前主流，无需 Critic ★★★
│   ├── REINFORCE    → 最简单的策略梯度
│   ├── RLOO         → Leave-One-Out 方差缩减
│   ├── ReMax        → REINFORCE + 基线
│   └── DAPO/DrGRPO  → GRPO 变体
│
├── 离线方法 (Offline) — 使用固定偏好数据
│   ├── DPO          → 直接偏好优化
│   ├── KTO          → 二元反馈优化
│   ├── SimPO        → 简化 DPO
│   └── Online DPO   → DPO 的在线变体
│
└── 自训练 (Self-Training)
    ├── STaR          → 自我教导推理
    └── ReST          → 自我训练 via 过滤
```

## 二、核心算法详解

### GRPO (Group Relative Policy Optimization) ★★★

**来源**：DeepSeek-R1 (2025 年 1 月)
**论文**：[arXiv:2501.12948](https://arxiv.org/abs/2501.12948)

**核心思想**：
```
对于每个 prompt x：
  1. 用当前策略采样 G 个响应：y_1, y_2, ..., y_G
  2. 对每个响应计算奖励：r_1, r_2, ..., r_G
  3. 计算组内相对优势：
     A_i = (r_i - mean(r)) / std(r)
  4. 用 PPO-clip 目标优化策略，但用组内相对优势替代 Value Function
```

**为什么 GRPO 成为主流**：
- **去掉 Critic 模型**：PPO 需要一个与 Actor 一样大的 Critic 模型 → GRPO 直接用组内比较替代
- **GPU 显存节省 ~50%**：不需要 Critic 的参数和优化器状态
- **实现更简单**：Critic 的训练本身就是个难题
- **效果不差**：在数学、代码等可验证领域，GRPO 效果与 PPO 相当甚至更好

**关键超参数**：
- G (组大小)：通常 8-64，更大的 G 提供更稳定的优势估计
- KL 惩罚系数：控制策略偏离参考策略的程度
- Clip 范围：PPO-clip 的裁剪阈值

### PPO (Proximal Policy Optimization)

**经典 RLHF 算法**
```
需要 4 个模型：
  Actor (策略模型) → 被训练
  Critic (价值模型) → 估计 V(s)
  Reference (参考模型) → KL 正则化
  Reward Model → 提供奖励信号

训练循环：
  1. Actor 生成响应
  2. Reward Model 打分
  3. Critic 估计优势 A = R - V(s)
  4. Actor 用 PPO-clip 目标优化
  5. Critic 用 Value Loss 优化
```

**优势**：理论基础扎实，适用范围广
**劣势**：4 个模型内存开销大，Critic 训练不稳定，工程复杂

### DPO (Direct Preference Optimization)

**论文**：[arXiv:2305.18290](https://arxiv.org/abs/2305.18290)

**核心思想**：
```
给定偏好对 (x, y_win, y_lose):
  Loss = -log σ(β · (log π(y_win|x)/π_ref(y_win|x) - log π(y_lose|x)/π_ref(y_lose|x)))

等价于：隐式定义了一个奖励函数 r(x,y) = β · log(π(y|x)/π_ref(y|x))
```

**优势**：
- 不需要单独的奖励模型
- 训练稳定，实现简单
- 只需偏好数据对

**劣势**：
- 离线方法，受限于固定偏好数据
- 缺少在线探索能力
- 效果通常不如在线 RL 方法

### DAPO (Dynamic sampling Policy Optimization)

- 基于 GRPO 的改进变体
- 在 Qwen2.5-32B 上用 veRL 训练
- AIME 2024 达到 50 分，超越 DeepSeek 的 GRPO
- **开源 SOTA RL 算法之一**

## 三、奖励信号设计

### 类型

| 奖励类型 | 描述 | 适用场景 | 代表 |
|---------|------|---------|------|
| 规则/可验证奖励 | 程序化验证答案正确性 | 数学、代码 | DeepSeek-R1 |
| 学习型奖励模型 | 训练一个模型给回答打分 | 通用对话 | InstructGPT |
| AI 反馈 (RLAIF) | 用另一个 LLM 做裁判 | 开放域 | Constitutional AI |
| 过程奖励 (PRM) | 奖励每个推理步骤 | 数学推理 | "Let's Verify Step by Step" |
| 结果奖励 (ORM) | 只奖励最终结果 | 通用 | 大多数 RLHF |

### DeepSeek-R1 的奖励设计
```
数学题：答案正确 → r = 1，错误 → r = 0
代码题：通过所有测试用例 → r = 1，否则 → r = 0
格式奖励：输出格式符合要求 → 额外 r
```
- **简单、可靠、可扩展**
- 不需要训练奖励模型
- 但仅适用于有客观答案的领域

## 四、在线 vs 离线 RL 的权衡

| 维度 | 在线 RL (GRPO/PPO) | 离线 (DPO) |
|------|-------------------|------------|
| 数据 | 训练中在线生成 | 固定偏好数据集 |
| 探索 | 有（生成多样响应） | 无 |
| 效果 | 更强 | 较弱 |
| 计算成本 | 高（需要 rollout） | 低 |
| 工程复杂度 | 高 | 低 |
| 适用场景 | 追求极致效果 | 快速迭代/资源受限 |

**共识 (2025)**：在线 RL 效果明显优于离线方法。对于追求最佳性能的场景，GRPO 是首选。

## 五、RL 训练的涌现现象

### DeepSeek-R1-Zero 的发现
- 纯 RL 训练（无 SFT）也能涌现 Chain-of-Thought 推理
- 涌现行为包括：
  - **自我验证** ("let me check...")
  - **反思** ("wait, that's wrong...")
  - **探索多条路径** ("alternatively...")
  - **"Aha" 时刻**：突然发现新的推理策略

### 启示
- RL 不仅仅是 "对齐"，它能**发现新的能力**
- 推理能力可能是通过 RL 自然涌现的，而非需要显式教导
- 这从根本上改变了对 RL 在 LLM 训练中角色的理解

#### Stage 1：冷启动 SFT（Cold Start）

**目的**：解决 R1-Zero 的可读性问题，为后续 RL 提供良好的起点。

- 收集**数千条**精心设计的长思维链（CoT）数据
- 数据来源包括：
  - 用 few-shot prompting 从 DeepSeek-R1-Zero 生成的高质量推理过程
  - 人工标注和筛选的结构化推理示例
  - 通过 R1-Zero 的输出进行人工后处理，使其更加可读
- 训练目标：让模型学会使用 `<think>...</think>` 格式输出结构化的推理过程
- 这一阶段的关键在于**少量高质量数据**就足以建立格式基础

#### Stage 2：推理导向强化学习

**目的**：通过大规模 RL 训练激发和增强推理能力。

- **算法**：GRPO（Group Relative Policy Optimization）
- **奖励设计**：
  - 数学题：正确答案 → r=1，错误 → r=0（规则验证）
  - 代码题：通过所有测试用例 → r=1，否则 → r=0
  - 格式奖励：输出格式是否符合 `<think>...</think><answer>...</answer>` 要求
  - **语言一致性奖励**：惩罚语言混杂现象（这是 R1-Zero 的主要问题之一）
- **关键超参数**（来自论文）：
  - 学习率：3e-6
  - KL 系数：0.001
  - GRPO clip ratio ε：10（注意这里 ε 远大于标准 PPO 的 0.2）
  - 采样温度：1.0
  - 每个问题采样 G=16 个输出
  - 最大输出长度：32,768 tokens
  - 每步训练 32 个唯一问题 → 批量大小 512
- **硬件**：648 张 H800 GPU，训练约 198 小时

#### Stage 3：拒绝采样 + 监督微调

**目的**：扩展模型的通用能力，弥补纯 RL 训练在非推理任务上的短板。

- **推理数据**（~600K 条）：
  - 从 Stage 2 的 RL 检查点进行**拒绝采样**（Rejection Sampling）
  - 对每个推理 prompt 生成多个候选响应
  - 仅保留正确、连贯、逻辑清晰的输出
  - 涵盖数学、代码、科学推理等任务
- **非推理数据**（~200K 条）：
  - 来自 DeepSeek-V3 的训练管线
  - 涵盖创意写作、事实问答、翻译、角色扮演、自我认知等
  - 这些数据确保模型在通用任务上也有良好表现
- **总计约 800K 条 SFT 数据**

#### Stage 4：综合强化学习

**目的**：最终对齐，同时保持推理能力。

- **奖励信号的组合**：
  - 推理任务：继续使用规则奖励（数学正确性、代码通过率）
  - 通用任务：使用学习型奖励模型（Outcome Reward Model）
  - 安全任务：使用安全奖励模型
- **Prompt 分布**：覆盖所有场景（推理、对话、写作、翻译等）
- **目标**：在增强有用性和安全性的同时，不损害推理能力
- **硬件**：648 张 H800 GPU，训练约 80 小时（约 4 天）

### 6.2 DeepSeek-R1-Zero vs DeepSeek-R1

R1-Zero 是理解整个研究的关键——它证明了**纯 RL 训练**（不经过任何 SFT）就能激发推理能力。

#### R1-Zero 的训练方式

```
DeepSeek-V3-Base → 直接 GRPO 训练（无 SFT）→ R1-Zero
                    │
                    ├── 奖励：仅规则奖励（数学正确性 + 格式合规）
                    ├── 无人工标注的推理链
                    └── 纯粹通过试错学习推理
```

**核心发现**：即使不给模型任何"如何推理"的示例，仅通过奖励信号的引导，模型就能自发学会复杂的推理策略。

#### R1-Zero 的涌现现象

1. **思维链长度自适应增长**
   - 随着训练进行，模型在困难问题上自动生成更长的思考过程
   - 简单问题保持简短推理
   - 这表明模型学会了"按需分配计算资源"

2. **自我验证（Self-verification）**
   - 模型自发出现 "let me check this answer..." 这样的验证行为
   - 会重新计算中间步骤以确认结果

3. **反思与回溯（Reflection）**
   - "Wait, that doesn't seem right..." 式的自我纠错
   - 发现前一步推理有误后，会尝试新的推理路径

4. **多路径探索**
   - "Alternatively, we can approach this problem by..."
   - 在一条路径受阻时，主动尝试不同的解题思路

5. **"Aha" 时刻**
   - 训练过程中，模型突然"发现"新的推理策略
   - 表现为性能指标的跳跃式提升
   - 论文中展示的案例：模型在某个训练步骤突然学会了一种更高效的验证方法

#### R1-Zero 的问题

| 问题 | 描述 | R1 如何解决 |
|------|------|------------|
| 语言混杂 | 中英文、代码混合输出，可读性极差 | Stage 2 加入语言一致性奖励 |
| 格式混乱 | 推理过程缺乏结构化格式 | Stage 1 冷启动 SFT 建立格式基础 |
| 重复循环 | 推理过程中出现无意义的重复 | 冷启动数据 + 格式奖励约束 |
| 仅限推理 | 在通用任务（写作、对话）上表现差 | Stage 3 混合 SFT 数据 |

  DeepSeek-R1-Zero (纯 RL):   71.0%   ← 仅通过 RL！
  DeepSeek-R1 (完整流水线):    79.8%

**关键洞察**：R1-Zero 仅通过纯 RL 就已经非常接近 OpenAI o1 的水平。R1 的四阶段流水线主要解决的是可读性、通用性和安全性问题，而非推理能力本身。

### 6.3 GRPO 数学推导

#### 从 REINFORCE 到 PPO 再到 GRPO

**起点：策略梯度定理（Policy Gradient Theorem）**

强化学习的基本目标是最大化期望回报：

```
J(θ) = E_{τ~π_θ}[R(τ)]
```

策略梯度的基本形式（REINFORCE）：

```
∇_θ J(θ) = E_{τ~π_θ}[∇_θ log π_θ(a|s) · R(τ)]
```

问题：方差极大。解决方案：引入基线（baseline）。

```
∇_θ J(θ) = E_{τ~π_θ}[∇_θ log π_θ(a|s) · (R(τ) - b)]
```

当 b = V(s)（状态价值函数）时，R(τ) - V(s) = A(s,a)（优势函数），方差最小。

**PPO 的做法**：训练一个 Critic 网络来近似 V(s)，然后用 GAE（Generalized Advantage Estimation）计算优势：

```
A^GAE_t = Σ_{l=0}^{∞} (γλ)^l · δ_{t+l}
其中 δ_t = r_t + γV(s_{t+1}) - V(s_t)
```

**GRPO 的关键创新**：用**组内相对比较**替代 Critic 网络来估计优势。

#### GRPO 优势估计

对于每个 prompt q，采样 G 个完整响应 {o_1, o_2, ..., o_G}，每个响应获得奖励 r_i。优势计算为：

```
Â_i = (r_i - mean({r_1, ..., r_G})) / std({r_1, ..., r_G})
```

**直觉理解**：
- 不需要学习"什么是好的"（V(s)），只需要知道"在这一组里，哪些是相对更好的"
- 这是一种**组内相对排名**的思想
- 当 G 足够大时，组内均值是对 V(s) 的蒙特卡洛近似

**数学论证**：
```
E[mean({r_1, ..., r_G})] = E_{o~π_θ}[r(o|q)] ≈ V^π(q)

因此：Â_i ≈ (r_i - V^π(q)) / σ_r

标准化（除以 std）进一步降低了不同 prompt 之间奖励尺度不同带来的影响。
```

#### GRPO 完整目标函数

```
J_GRPO(θ) = E_q [ 1/G · Σ_{i=1}^{G} · 1/|o_i| · Σ_{t=1}^{|o_i|} {
    min(
        ρ_{i,t} · Â_i,
        clip(ρ_{i,t}, 1-ε, 1+ε) · Â_i
    )
} ] - β · D_KL(π_θ || π_ref)
```

其中概率比（importance ratio）为：
```
ρ_{i,t} = π_θ(o_{i,t} | q, o_{i,<t}) / π_{θ_old}(o_{i,t} | q, o_{i,<t})
```

**各组件解释**：

| 组件 | 作用 | 详细说明 |
|------|------|---------|
| `ρ_{i,t}` | 概率比 | 新旧策略在 token 级别的概率比值 |
| `clip(ρ, 1-ε, 1+ε)` | 裁剪 | 限制策略更新幅度，防止灾难性更新 |
| `min(·, ·)` | 悲观更新 | 取未裁剪和裁剪目标的较小值 |
| `1/\|o_i\|` | 长度归一化 | 防止长响应主导梯度 |
| `β · D_KL` | KL 惩罚 | 防止策略偏离参考模型太远 |

#### GRPO 的 KL 散度估计器

GRPO 采用了一个特殊的 KL 散度无偏估计器（来自 DeepSeekMath 论文）：

```
D_KL(π_θ || π_ref) ≈ E_{x~π_θ}[ π_ref(x)/π_θ(x) - log(π_ref(x)/π_θ(x)) - 1 ]
```

用 token 级别的 log 概率表示：
```
per_token_kl = exp(log_π_ref - log_π_θ) - (log_π_ref - log_π_θ) - 1
```

**为什么选择这个估计器而非标准 KL**：
- 标准 KL 估计器 `D_KL = E[log(π_θ/π_ref)]` 可能为负值（有偏估计）
- 上述估计器始终 ≥ 0（因为 e^x - x - 1 ≥ 0 对所有 x 成立）
- 方差更低，训练更稳定
- 当 π_θ = π_ref 时，估计值恰好为 0

**注意**：PPO 中 KL 惩罚通常被加到 token 级别的奖励中（即修改 reward signal），而 GRPO 中 KL 惩罚是直接从目标函数中**减去**的。这是一个重要的实现区别。

### 6.4 蒸馏结果

DeepSeek 团队将 R1 的推理能力蒸馏到了一系列小模型中，取得了令人瞩目的结果。

#### 蒸馏方法

```
蒸馏流程：
  1. 使用 DeepSeek-R1 生成大量高质量推理数据（包含完整思维链）
  2. 用这些数据对小模型进行监督微调（SFT）
  3. 不需要额外的 RL 训练——纯 SFT 蒸馏
```

**关键洞察**：蒸馏的效果显著优于在小模型上直接做 RL 训练。论文实验表明，对小模型直接应用 GRPO 训练，效果远不如用大模型生成的推理数据进行蒸馏。这表明推理能力的"发现"需要足够大的模型容量，但推理能力的"模仿"可以在小模型上完成。

  - 蒸馏的本质是"知识压缩"，而非"能力发现"

## 七、DPO 论文深度解析

> **论文**：*Direct Preference Optimization: Your Language Model is Secretly a Reward Model* ([arXiv:2305.18290](https://arxiv.org/abs/2305.18290))
> **作者**：Rafael Rafailov, Archit Sharma, Eric Mitchell 等 (Stanford)
> **发表**：NeurIPS 2023
> **核心贡献**：证明 RLHF 的目标可以等价转化为一个简单的分类损失，无需显式训练奖励模型。

### 7.1 从 RLHF 出发

#### 标准 RLHF 目标

传统 RLHF 的优化目标是：

```
max_π E_{x~D, y~π(·|x)} [r(x, y)] - β · D_KL(π || π_ref)
```

用文字表述：最大化奖励模型给出的期望奖励，同时通过 KL 散度惩罚来约束策略不要偏离参考策略太远。

完整写出：

```
max_π E_{x~D} [ E_{y~π(·|x)} [r(x,y)] - β · D_KL(π(·|x) || π_ref(·|x)) ]
```

展开 KL 散度项：

```
= max_π E_{x~D} [ E_{y~π(·|x)} [ r(x,y) - β · log(π(y|x) / π_ref(y|x)) ] ]
```

#### 求解最优策略（闭式解）

DPO 论文的第一个关键洞察是：上述 KL 约束的 RL 目标存在**解析最优解**。

对于固定的 x，优化问题变为：

```
max_π E_{y~π} [ r(x,y) - β · log(π(y|x) / π_ref(y|x)) ]
```

这等价于最小化：

```
min_π E_{y~π} [ β · log(π(y|x) / π_ref(y|x)) - r(x,y) ]
     = min_π E_{y~π} [ β · log(π(y|x) / (π_ref(y|x) · exp(r(x,y)/β))) ]
     = min_π β · D_KL(π(·|x) || π_ref(·|x) · exp(r(·,x)/β) / Z(x))
```

其中 Z(x) = Σ_y π_ref(y|x) · exp(r(x,y)/β) 是配分函数（归一化常数）。

KL 散度在两个分布相等时取最小值 0，因此最优策略为：

```
π*(y|x) = π_ref(y|x) · exp(r(x,y)/β) / Z(x)
```

**这就是 RLHF 的最优策略的闭式解。** 它告诉我们：最优策略就是参考策略按照奖励的指数进行"重加权"。

### 7.2 奖励的重参数化

从最优策略表达式出发，我们可以反解出奖励函数：

```
π*(y|x) = π_ref(y|x) · exp(r(x,y)/β) / Z(x)

两边取 log：
log π*(y|x) = log π_ref(y|x) + r(x,y)/β - log Z(x)

解出 r(x,y)：
r(x,y) = β · log(π*(y|x) / π_ref(y|x)) + β · log Z(x)
```

**关键公式**：

```
r(x,y) = β · log(π*(y|x) / π_ref(y|x)) + β · log Z(x)
```

这意味着：**奖励函数可以用最优策略和参考策略的对数概率比来表示**。β·log Z(x) 是一个仅依赖于 x（prompt）而不依赖于 y（response）的常数项。

### 7.3 Bradley-Terry 偏好模型

#### BT 模型的定义

在偏好学习中，我们有人类对两个响应的偏好判断。Bradley-Terry 模型将偏好概率建模为：

```
P(y_w ≻ y_l | x) = σ(r(x, y_w) - r(x, y_l))
```

其中 σ 是 sigmoid 函数：σ(z) = 1/(1+exp(-z))。

直觉：奖励差越大，被偏好的响应"获胜"的概率越高。

#### 标准 RLHF 中的奖励模型训练

传统 RLHF 的第一步是训练一个参数化的奖励模型 r_φ(x,y)：

```
L_RM(φ) = -E_{(x, y_w, y_l) ~ D} [ log σ(r_φ(x, y_w) - r_φ(x, y_l)) ]
```

然后用这个学到的奖励模型通过 PPO 来训练策略。

### 7.4 DPO 的推导：将奖励替换为策略

**DPO 的核心技巧**：将重参数化后的奖励代入 Bradley-Terry 模型。

将 `r(x,y) = β · log(π*(y|x) / π_ref(y|x)) + β · log Z(x)` 代入偏好模型：

```
P(y_w ≻ y_l | x) = σ(r(x, y_w) - r(x, y_l))
    = σ(β · log(π*(y_w|x)/π_ref(y_w|x)) + β·log Z(x)
      - β · log(π*(y_l|x)/π_ref(y_l|x)) - β·log Z(x))
```

**关键观察**：β·log Z(x) 项在做差时完美抵消！

```
P(y_w ≻ y_l | x) = σ(β · [log(π*(y_w|x)/π_ref(y_w|x)) - log(π*(y_l|x)/π_ref(y_l|x))])
```

这意味着偏好概率**仅**取决于策略和参考策略的对数比值差，不依赖于任何显式的奖励模型或难以计算的配分函数。

#### DPO 损失函数

用参数化策略 π_θ 替代最优策略 π*，DPO 的训练目标为：

```
L_DPO(θ) = -E_{(x, y_w, y_l) ~ D} [ log σ(β · (log π_θ(y_w|x)/π_ref(y_w|x)
                                              - log π_θ(y_l|x)/π_ref(y_l|x))) ]
```

更紧凑的形式：

```
L_DPO(θ) = -E [ log σ(β · (h_θ(x, y_w) - h_θ(x, y_l))) ]

其中 h_θ(x, y) = log(π_θ(y|x) / π_ref(y|x))  ← 隐式奖励
```

### 7.5 DPO 为什么有效？

#### 隐式奖励解读

DPO 隐式地定义了一个奖励函数：

```
r_implicit(x, y) = β · log(π_θ(y|x) / π_ref(y|x))
```

**直觉**：如果策略比参考模型更倾向于生成某个响应 y，那么 y 的隐式奖励就更高。这实际上是将"策略的偏好"等同于"奖励的评价"。

#### 梯度分析

DPO 损失的梯度可以写为：

```
∇_θ L_DPO = -β · E [ σ(r̂_l - r̂_w) · (∇_θ log π_θ(y_w|x) - ∇_θ log π_θ(y_l|x)) ]

其中 r̂ = β · log(π_θ/π_ref) 是隐式奖励
```

**梯度解读**：
- `σ(r̂_l - r̂_w)` 是加权系数：当模型**错误地**给 y_l 更高的隐式奖励时（即排序错误），这个权重更大
- 梯度的方向：增加 y_w 的对数概率，减少 y_l 的对数概率
- **自适应权重**：已经正确排序的样本对，梯度权重自动减小（隐式的难例挖掘）

#### DPO 与 RLHF 的等价性

**定理**（DPO 论文 Theorem 1）：在 Bradley-Terry 偏好模型下，如果 RLHF 找到了最优奖励函数，那么 DPO 和 RLHF 的全局最优解是相同的。

```
条件：
  1. 偏好数据由 Bradley-Terry 模型生成
  2. RLHF 的奖励模型完美拟合偏好数据
  3. PPO 找到全局最优

则：DPO 的全局最优策略 = RLHF 的全局最优策略
```

**实际差异**：
- 理论上等价，但实际训练中存在优化路径差异
- DPO 是**离线**方法，使用固定的偏好数据集
- RLHF 是**在线**方法，训练过程中不断生成新的响应
- 实践中，在线方法通常表现更优，因为数据分布随策略更新

### 7.6 DPO 的局限性

| 局限 | 详细说明 |
|------|---------|
| **分布偏移** | 策略 π_θ 在训练中不断变化，但偏好数据是固定的。当 π_θ 偏离生成偏好数据的策略时，训练效果退化 |
| **无在线探索** | 无法生成新的响应来发现更好的策略，受限于初始偏好数据的覆盖范围 |
| **偏好数据质量** | 效果高度依赖于偏好数据的质量和多样性 |
| **隐式奖励退化** | β·log(π_θ/π_ref) 作为奖励可能不够灵活，无法表达复杂的奖励结构 |
| **长度偏见** | 倾向于生成更长的响应（因为更长的响应有更多 token 被优化） |

**后续改进方向**：
- **Online DPO / Iterative DPO**：在训练中在线采样新数据，缓解分布偏移
- **SimPO**：移除对参考模型的依赖，简化训练
- **KTO**：不需要成对偏好数据，仅需要二元反馈（好/坏）

### 8.2 核心算法差异

#### 优势函数估计（最根本的区别）

**PPO：使用 Critic 网络 + GAE**

```
PPO 优势估计：
  1. Critic 网络 V_φ(s) 近似状态价值函数
  2. TD 误差：δ_t = r_t + γ·V_φ(s_{t+1}) - V_φ(s_t)
  3. GAE：Â_t = Σ_{l=0}^{T-t} (γλ)^l · δ_{t+l}

  优点：token 级别的细粒度优势估计
  缺点：
    - Critic 需要与 Actor 一样大（参数量翻倍）
    - Critic 训练本身就是难题（bootstrapping 不稳定）
    - Critic 的误差会传播到 Actor 的梯度中
```

**GRPO：使用组内相对比较**

```
GRPO 优势估计：
  1. 对每个 prompt 采样 G 个完整响应
  2. 对每个响应计算奖励 r_i
  3. 优势 Â_i = (r_i - mean(r)) / std(r)
  4. 同一响应的所有 token 共享相同的优势值

  优点：
    - 完全不需要 Critic 网络
    - 实现简单，无 bootstrapping
    - 优势估计来自真实奖励，无近似误差
  缺点：
    - 同一响应内所有 token 共享优势（无 token 级别的信用分配）
    - 需要大的组大小 G 以获得稳定估计
    - 采样 G 个响应增加了推理开销
```

#### 信用分配（Credit Assignment）

```
PPO（token 级别信用分配）：
  "The answer is 42" → 每个 token 有不同的优势值
    "The"    → Â₁ = 0.1   （低贡献）
    "answer" → Â₂ = 0.3   （中等贡献）
    "is"     → Â₃ = 0.1   （低贡献）
    "42"     → Â₄ = 0.9   （高贡献 — 关键 token）

GRPO（序列级别信用分配）：
  "The answer is 42" → 所有 token 共享同一优势
    "The"    → Â = 0.94
    "answer" → Â = 0.94
    "is"     → Â = 0.94
    "42"     → Â = 0.94
```

这种差异在理论上应该让 PPO 更精细，但实际上：
- Critic 网络的近似误差可能导致 token 级别优势不准确
- 对于**结果可验证**的任务（数学、代码），序列级别的信用分配已经足够
- GRPO 的简单性带来的训练稳定性优势通常大于 PPO 精细信用分配的理论优势

#### KL 散度的处理方式

```
PPO for LLM：
  KL 惩罚加入 token 级别奖励
  r̃_t = r_t - β · log(π_θ(a_t|s_t) / π_ref(a_t|s_t))
  → KL 被嵌入到奖励信号中，通过 Critic/GAE 传播

GRPO：
  KL 惩罚直接从目标函数中减去
  J = L_clip - β · D_KL(π_θ || π_ref)
  → KL 作为独立的正则项，使用无偏估计器计算

  KL 估计器：D_KL ≈ exp(log π_ref - log π_θ) - (log π_ref - log π_θ) - 1
```

#### Clip 机制的对比

```
两种算法都使用 PPO-clip 目标：
  L_clip = min(ρ · Â, clip(ρ, 1-ε, 1+ε) · Â)

但 ε 的典型值不同：
  PPO:  ε = 0.2（标准值）
  GRPO: ε 可以更大（DeepSeek-R1 使用 ε=10，这实际上几乎禁用了裁剪！）

DeepSeek-R1 使用如此大的 ε 的原因：
  - KL 散度惩罚已经起到约束策略更新的作用
  - 大 ε 允许更激进的策略更新，加速探索
  - 这是一个"KL 惩罚代替裁剪"的设计选择
```

### 8.3 计算效率对比

| 维度 | PPO | GRPO | GRPO 优势 |
|------|-----|------|----------|
| 模型数量 | 4 个（Actor, Critic, Ref, RM） | 2-3 个（Actor, Ref, [RM]） | 少 1-2 个模型 |
| GPU 显存 | ~4x 模型参数 | ~2x 模型参数 | 节省约 50% |
| Critic 训练 | 需要额外的反向传播 | 无 | 完全省去 |
| 推理开销 | 每个 prompt 1 个响应 | 每个 prompt G 个响应 | PPO 更低 |

| 训练稳定性 | 较低（Critic 引入噪声） | 较高（真实奖励比较） | 更稳定 |

### 8.4 适用场景

```
GRPO 更适合：
  ✓ 可验证奖励的任务（数学、代码、逻辑推理）

  ✓ 追求实现简洁性
  ✓ RLVR（Reinforcement Learning with Verifiable Rewards）

PPO 更适合：
  ✓ 需要学习型奖励模型的通用 RLHF
  ✓ 需要 token 级别细粒度信用分配的场景
  ✓ 奖励信号稀疏且需要精细传播的任务
  ✓ 有充足 GPU 资源的大规模训练

2025 年的趋势：
  → 推理模型（reasoning）：GRPO 是事实标准
  → 通用对齐（general alignment）：PPO 仍有一席之地
  → 混合方案越来越常见：先用 GRPO 做推理 RL，再用 PPO/DPO 做通用对齐
```

## 九、奖励设计策略详解

奖励设计是 RL 训练成败的关键。不同的奖励策略适用于不同场景，各有优劣。

### 9.1 可验证奖励（Verifiable Rewards）

**核心思想**：通过程序化方式客观判断响应的正确性。

#### 数学领域

```python
# 示例：DeepSeek-R1 的数学奖励函数
def math_reward(response: str, ground_truth: str) -> float:
    """
    从模型输出中提取最终答案，与标准答案比较。
    """
    # 提取 \boxed{} 中的答案
    predicted = extract_boxed_answer(response)

    # 数学等价性判断（不仅仅是字符串匹配）
    # 例如：1/2 == 0.5 == 2/4 都应该判为正确
    if is_mathematically_equivalent(predicted, ground_truth):
        return 1.0
    else:
        return 0.0
```

**实际考虑**：
- 答案提取需要鲁棒的正则匹配（处理不同格式）
- 数学等价性判断需要符号计算工具（如 SymPy）
- 某些问题的答案有多种等价形式（分数、小数、化简等）

#### 代码领域

```python
# 示例：代码题的可验证奖励
def code_reward(generated_code: str, test_cases: list) -> float:
    """
    在沙盒中执行代码并验证测试用例。
    """
    passed = 0
    total = len(test_cases)

    for test_input, expected_output in test_cases:
        try:
            # 在安全沙盒中执行（有超时限制）
            actual_output = sandbox_execute(generated_code, test_input, timeout=10)
            if actual_output == expected_output:
                passed += 1
        except (TimeoutError, RuntimeError):
            pass  # 超时或运行错误视为失败

    # 可以选择二元奖励或按通过率给分
    # 二元：全部通过才给分
    return 1.0 if passed == total else 0.0
    # 按比例：return passed / total
```

#### 格式奖励

```python
# 示例：格式合规性奖励
def format_reward(response: str) -> float:
    """
    检查输出是否符合要求的格式。
    """
    reward = 0.0

    # 检查是否使用了 <think> 标签
    if "<think>" in response and "</think>" in response:
        reward += 0.5

    # 检查是否有最终答案标记
    if "<answer>" in response and "</answer>" in response:
        reward += 0.5

    return reward
```

**可验证奖励的优势与局限**：

| 优势 | 局限 |
|------|------|
| 完全客观，无主观偏差 | 仅适用于有客观答案的领域 |
| 不需要训练奖励模型 | 无法评估回答质量（如表达流畅性） |
| 无奖励黑客攻击风险 | 无法处理开放性问题 |
| 计算成本极低 | 格式奖励可能过于简单 |
| 可以大规模并行计算 | 部分正确的答案得不到部分奖励 |

### 9.2 结果奖励模型（Outcome Reward Model, ORM）

**核心思想**：训练一个神经网络模型，对完整的响应进行质量评分。

#### 训练流程

```
ORM 训练流程：
  1. 收集人类偏好数据 D = {(x, y_w, y_l)}
     其中 y_w 是被偏好的响应，y_l 是被拒绝的响应

  2. 训练奖励模型（通常基于 LLM + 线性头）：
     r_φ: (x, y) → R  （输出标量奖励值）

  3. 损失函数（Bradley-Terry）：
     L(φ) = -E [ log σ(r_φ(x, y_w) - r_φ(x, y_l)) ]

  4. 推理时：给定 (x, y)，输出 r_φ(x, y) 作为奖励信号
```

#### 架构设计

```
典型 ORM 架构：

输入: [prompt] [response] [EOS]
                              │
                              ▼
        ┌────────────────────────────────────┐
        │        预训练 LLM (Backbone)         │
        │    例如 Llama-3-8B、Qwen2.5-7B      │
        └──────────────┬─────────────────────┘
                       │ 最后一个 token 的隐状态
                       ▼
        ┌──────────────────────────┐
        │    线性投影层 (1 维输出)    │
        │    hidden_dim → 1         │
        └──────────────┬───────────┘
                       │
                       ▼
                   奖励分数 r ∈ R
```

#### ORM 的问题

1. **奖励黑客攻击（Reward Hacking）**
   ```
   问题：策略可能找到"欺骗"奖励模型的方式来获取高分，而非真正提升质量。

   典型案例：
   - 模型学会输出特定的"讨好"词汇（"Certainly!", "Great question!"）
   - 生成更长的响应（奖励模型可能偏好长文本）
   - 使用模板化的结构来获取高分

   缓解手段：
   - KL 惩罚约束策略不要偏离太远
   - 定期更新奖励模型
   - 使用多个奖励模型的集成
   ```

2. **分布外泛化（OOD Generalization）**
   ```
   奖励模型在训练数据分布外的预测不可靠。
   当策略生成奖励模型未见过的响应类型时，
   奖励分数可能完全不准确。
   ```

### 9.3 过程奖励模型（Process Reward Model, PRM）

> **代表论文**：*Let's Verify Step by Step* (OpenAI, 2023)

**核心思想**：不仅评价最终结果，还对**每个推理步骤**进行正确性评估。

#### PRM vs ORM 对比

```
数学推理示例：

问题：求 2^10 的值

模型推理过程：
  Step 1: 2^10 = 2^5 × 2^5           ← 正确
  Step 2: 2^5 = 32                    ← 正确
  Step 3: 32 × 32 = 1024              ← 正确
  答案: 1024                          ← 正确

ORM 评价：仅看最终答案 1024 → 正确 → r = 1.0
PRM 评价：逐步评价
  Step 1: ✓ (正确分解)        → r₁ = 1.0
  Step 2: ✓ (正确计算)        → r₂ = 1.0
  Step 3: ✓ (正确乘法)        → r₃ = 1.0
  最终:   ✓                   → r₄ = 1.0

反例（含错误步骤但碰巧答案正确）：

  Step 1: 2^10 = 2^8 × 2^3           ← 错误！(8+3≠10)
  Step 2: 2^8 = 256                   ← 正确（但基于错误前提）
  Step 3: 256 × 8 = 2048              ← 正确计算
  Step 4: 实际上让我重算...2^10=1024   ← 纠正
  答案: 1024                          ← 正确

ORM 评价：答案正确 → r = 1.0 （无法发现中间错误）
PRM 评价：
  Step 1: ✗ (分解错误)        → r₁ = 0.0  ← 发现错误位置！
  Step 2: ✓ (计算本身正确)     → r₂ = 1.0
  Step 3: ✓ (计算本身正确)     → r₃ = 1.0
  Step 4: ✓ (自我纠正)        → r₄ = 1.0
```

#### PRM 的训练

```
PRM 训练数据：需要步骤级别的标注
  - 人工标注（昂贵但准确）
  - 蒙特卡洛估计（自动但有噪声）：
    对每个步骤，多次从该步骤继续生成并验证最终答案
    该步骤的"正确率"作为步骤奖励的估计

蒙特卡洛估计过程：
  对于推理链的第 k 步：
    1. 从第 k 步开始，多次独立续写至最终答案
    2. 统计最终答案的正确率 p_k
    3. r_k = p_k 作为该步骤的过程奖励

  直觉：如果一个步骤是正确的，从它出发续写更容易得到正确答案；
        如果一个步骤是错误的，从它出发续写大概率得到错误答案。
```

#### PRM 的性能

来自 OpenAI 的 "Let's Verify Step by Step" 实验结果：

```
MATH 测试集（使用 Best-of-N 选择策略）：

  多数投票 (Majority Voting):    ~70%
  ORM (Best-of-N):              ~72%
  PRM (Best-of-N):              ~78.2%   ← 显著优于 ORM

关键结论：
  - PRM 在 Best-of-N 选择中显著优于 ORM
  - PRM 的优势随着 N 增大而增大
  - 过程监督训练出的奖励模型更可靠
```

#### PRM 的局限

| 局限 | 详细说明 |
|------|---------|
| 标注成本高 | 人工标注每个步骤的正确性非常昂贵 |
| 步骤边界定义 | 如何界定"一步"没有明确标准 |
| 适用范围窄 | 主要适用于多步推理任务 |
| 蒙特卡洛估计的噪声 | 自动标注方法精度有限 |
| 规模化困难 | 步骤级别奖励信号的密度增加训练复杂度 |

### 9.4 AI 反馈奖励（RLAIF / Constitutional AI）

**核心思想**：用另一个 LLM（通常是更强的模型）来充当"裁判"，替代人类标注。

```
RLAIF 流程：
  1. 生成响应对：对于 prompt x，让策略生成 y_1, y_2
  2. LLM 裁判评估：
     prompt = "以下是对用户问题的两个回答，请判断哪个更好：
               用户问题：{x}
               回答A：{y_1}
               回答B：{y_2}
               更好的回答是："
  3. 收集 LLM 的偏好判断作为训练信号
  4. 用这些偏好数据训练奖励模型或直接用于 DPO

优势：
  - 标注成本极低（无需人工）
  - 可以大规模生成偏好数据
  - 一致性高于人类标注者（人类标注者之间经常不一致）

局限：
  - LLM 裁判可能有自己的偏见（如偏好更长的回答）
  - 存在"自我循环"风险：用 AI 评价 AI 可能放大错误
  - 在专业领域（医学、法律），LLM 判断可能不准确
```

### 9.5 DeepSeek-R1 的奖励设计策略（综合案例）

DeepSeek-R1 的奖励设计堪称教科书级别的简洁高效：

```
Stage 2（推理 RL）的奖励设计：

  r_total = r_accuracy + r_format + r_language

  其中：
    r_accuracy = {
        数学题: 答案正确 → 1.0, 错误 → 0.0
        代码题: 全部测试通过 → 1.0, 否则 → 0.0
    }

    r_format = {
        正确使用 <think>...</think> 格式 → 正向奖励
        缺少格式标记 → 负向奖励
    }

    r_language = {
        输出语言一致 → 正向奖励
        语言混杂 → 负向奖励
    }

Stage 4（综合 RL）的奖励设计：

  推理任务: 继续使用规则奖励
  通用任务: 使用学习型奖励模型
  安全任务: 使用安全奖励模型
```

**设计哲学**：
1. **能用规则就不用模型**：规则奖励简单、准确、不会被黑客攻击
2. **只在必要时引入学习型奖励**：通用对话等无客观标准的任务才使用 RM
3. **分阶段、分场景使用不同奖励**：而非试图用单一奖励模型覆盖所有场景
4. **奖励信号的粒度要匹配任务**：数学用结果奖励足矣，不必强行使用 PRM

### 9.6 奖励设计的最佳实践

```
实用决策树：

你的任务有客观答案吗？
├── 是 → 使用可验证奖励（规则奖励）
│       ├── 数学 → 答案正确性 + 格式合规
│       ├── 代码 → 测试用例通过率
│       └── 逻辑推理 → 规则验证
│
└── 否 → 需要学习型奖励
        │
        ├── 有高质量偏好数据？
        │   ├── 是 → ORM（训练奖励模型）或 DPO（跳过奖励模型）
        │   └── 否 → RLAIF（用 LLM 生成偏好数据）
        │
        └── 任务是多步推理？
            ├── 是 + 有标注预算 → PRM
            └── 是 + 无标注预算 → ORM + Best-of-N 采样

混合奖励策略（推荐）：
  r_total = w₁·r_verifiable + w₂·r_ORM + w₃·r_format + w₄·r_safety

  根据任务类型动态调整权重：
    推理任务：w₁=0.8, w₂=0.0, w₃=0.1, w₄=0.1
    通用对话：w₁=0.0, w₂=0.6, w₃=0.1, w₄=0.3
    创意写作：w₁=0.0, w₂=0.7, w₃=0.2, w₄=0.1
```
