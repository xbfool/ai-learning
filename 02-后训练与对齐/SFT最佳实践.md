# SFT 最佳实践

> 监督微调 (Supervised Fine-Tuning) 是后训练的第一步，教模型理解指令格式和对话模式。

## 一、核心原则

### "少即是多" (LIMA 原则)
- **论文**：[LIMA: Less Is More for Alignment](https://arxiv.org/abs/2305.18290) (2023)
- 仅 1,000 条高质量指令数据就能显著对齐模型
- **数据质量 >> 数据数量**
- 启示：花 10x 精力提高数据质量，比收集 10x 更多数据更有价值

### DeepSeek-R1 的冷启动 SFT
- 在纯 RL 训练前，用少量高质量 CoT（链式推理）数据做 SFT
- 目的：给模型一个 "格式化思考" 的良好起点
- R1-Zero（纯 RL 无 SFT）也能涌现推理能力，但 R1（SFT + RL）效果更好

## 二、SFT 数据构建

### 数据来源

| 来源 | 优势 | 劣势 |
|------|------|------|
| 人工标注 | 质量最高 | 成本高、难以规模化 |
| 大模型蒸馏 | 可大规模生成 | 受限于教师模型能力 |
| Self-Instruct | 自动化 | 多样性有限 |
| Evol-Instruct | 自动提高复杂度 | 可能偏离自然分布 |
| 拒绝采样 | 结合 RL 训练 | 需要奖励信号 |

### 关键维度
1. **多样性**：覆盖尽可能多的任务类型和风格
2. **复杂度**：包含简单到复杂的梯度
3. **格式规范**：统一的指令-响应格式
4. **安全性**：拒绝有害请求的示例

## 三、多轮 SFT

### Llama 3 的方案
- 多轮对话 SFT：模型学习在多轮交互中保持连贯
- 拒绝采样：从多个候选中选择最优响应
- 多模型混合：从不同模型采样最佳响应

### 数据混合
- 按能力维度（通用对话、代码、数学、创意、安全）混合 SFT 数据
- 各能力维度的比例需要通过消融实验确定
- 代码/数学占比通常高于自然比例

## 四、SFT vs RL 的分工

```
SFT 的作用：
- 教会模型 "格式"（如何结构化回答）
- 教会模型 "基本能力"（理解指令、生成代码等）
- 提供 RL 的良好初始化点

RL 的作用：
- 提升 "质量"（更好的推理、更准确的回答）
- 学习 "探索"（SFT 只学模仿，RL 学探索）
- 超越训练数据的能力边界
```

## 五、实践建议

1. **优先投资数据质量**，而非数量
2. **使用多阶段 SFT**：通用 → 领域 → 格式
3. **SFT 后接 RL**：不要止步于 SFT
4. **监控过拟合**：SFT 很容易过拟合到训练数据的特定模式
5. **保留评估集**：确保 SFT 数据与评估数据无交叉

## 六、核心论文深度解读

### 6.1 LIMA: Less Is More for Alignment (2023)

**论文**：[arXiv:2305.18290](https://arxiv.org/abs/2305.18290)
**作者**：Meta AI (Chunting Zhou, Pengfei Liu, et al.)

#### 核心假说：表面对齐假说 (Superficial Alignment Hypothesis)

LIMA 提出了一个对整个对齐领域产生深远影响的假说：

> 模型的知识和能力几乎完全在预训练阶段学习。对齐（SFT/RLHF）只需要教会模型用正确的**格式和风格**与用户交互，即"表面对齐"。

这意味着：
- 预训练决定了模型**能做什么** → 能力上限
- SFT/RLHF 只决定模型**如何呈现**已有的能力 → 表达方式
- 因此，少量高质量的格式示例就足够了

#### 实验设计

```
基座模型: LLaMA-65B
SFT 数据: 仅 1,000 条精心策划的指令-响应对
  ├── 750 条来自社区论坛 (Stack Exchange, wikiHow, Pushshift Reddit)
  ├── 250 条人工编写
  └── 每条都经过人工审查和修改

对比:
  - GPT-4 (几万/几十万条 RLHF 数据)
  - Alpaca (52K 条 Self-Instruct 数据)
  - DaVinci003 (RLHF)
```

#### 关键发现

1. **1000 条 > 52000 条**：LIMA 在人类评估中显著优于使用 52K 数据的 Alpaca
2. **质量碾压数量**：精心策划的 1000 条 ≈ 数万条普通质量数据
3. **无需 RLHF**：在很多场景下，纯 SFT 已经足够好
4. **但有上限**：在安全性和某些需要深度推理的任务上，仅 SFT 仍有不足

#### 数据策略细节

LIMA 的数据策略对后续工作影响巨大：

```
数据选择标准：
  1. 多样性 > 数量：覆盖尽可能多的主题和任务类型
  2. 高质量响应：每个响应都应该是"理想回答"
  3. 风格统一：保持一致的回答风格和详细程度
  4. 自然分布：来自真实用户需求，不是人工编造
```

#### 对行业的影响

LIMA 之后，业界的 SFT 策略发生了根本转变：
- **之前**：尽可能多地收集 SFT 数据（10万+条）
- **之后**：聚焦于少量高质量数据（1K-10K 条），投入更多精力做数据质量控制
- Llama 3、DeepSeek-R1 等后续模型都采用了"少量精选 SFT + RL"的策略

### 6.2 Self-Instruct: 自动化指令数据生成 (2023)

**论文**：[arXiv:2212.10560](https://arxiv.org/abs/2212.10560)
**作者**：Yizhong Wang et al.

#### 核心思想

用 LLM 自身来生成训练自己的指令数据：

```
种子指令集 (175 条人工编写)
  ↓
LLM 生成新指令
  ↓
LLM 为新指令生成输入-输出对
  ↓
过滤低质量数据 (ROUGE 去重 + 规则过滤)
  ↓
新的指令数据集 (52K+ 条)
  ↓
用于 SFT 训练
```

#### 生成流程详解

1. **指令生成**：从种子集中随机采样 6 条作为 few-shot 示例，让模型生成新指令
2. **分类判断**：判断新指令是"分类任务"还是"生成任务"
3. **输入生成**：如果需要输入，让模型生成合适的输入实例
4. **输出生成**：给定指令和输入，让模型生成输出
5. **质量过滤**：
   - ROUGE-L > 0.7 的视为与已有指令重复，剔除
   - 过滤长度异常、格式不符的数据
   - 过滤模型拒绝回答的情况

#### 局限性

- 受限于种子模型的能力（早期用 GPT-3）
- 生成的数据多样性仍有天花板
- 可能引入模型偏见的自我强化

### 6.3 Evol-Instruct: 进化式指令复杂化 (WizardLM, 2023)

**论文**：[arXiv:2304.12244](https://arxiv.org/abs/2304.12244)

#### 核心思想

不是随机生成新指令，而是**系统性地让已有指令变得更复杂**：

```
简单指令: "写一个排序函数"
  ↓ 深度进化
复杂指令: "实现一个支持多种排序策略(快排/归并/堆排)的泛型排序框架，
          支持自定义比较器，处理空值和重复值，时间复杂度分析"

进化维度：
  ├── 增加约束 (Add Constraints)
  ├── 深化 (Deepening)
  ├── 具体化 (Concretizing)
  ├── 增加推理步骤 (Increase Reasoning Steps)
  └── 复杂化输入 (Complicate Input)
```

#### 进化策略

| 策略 | 描述 | 示例 |
|------|------|------|
| 增加约束 | 添加限制条件 | "用 O(n) 时间复杂度" |
| 深化 | 要求更深层理解 | "解释为什么这样做" |
| 具体化 | 用具体场景替代抽象 | "为电商订单系统..." |
| 增加步骤 | 需要更多推理步骤 | "先分析再设计再实现再测试" |
| 宽度扩展 | 增加相关但不同的子任务 | "同时处理并发和持久化" |

#### 效果
- WizardLM-13B 使用 Evol-Instruct 数据训练后，在复杂指令遵循上显著超越 Alpaca
- 证明指令的**复杂度分布**对模型能力有重要影响

### 6.4 Llama 3 的完整后训练流水线

**论文**：[arXiv:2407.21783](https://arxiv.org/abs/2407.21783) (Section 4: Post-Training)

Llama 3 的后训练是目前公开的最完整的工业级 SFT+RL 流水线描述。

#### 整体流程

```
Llama 3 Base
  │
  ├── Round 1: SFT
  │     ├── 人工标注数据
  │     ├── 合成数据（用 Llama 3 自身生成）
  │     └── 多来源数据混合
  │
  ├── Round 1: Reward Model 训练
  │     └── 人类偏好排序数据
  │
  ├── Round 1: DPO
  │     └── 用 Reward Model 采样偏好对
  │
  ├── Round 2-6: 迭代 (每轮重复 SFT → RM → DPO)
  │     ├── 每轮用改进的模型生成更好的合成数据
  │     ├── 每轮用更新的 Reward Model 做偏好判断
  │     └── 迭代提升
  │
  └── Llama 3 Instruct
```

#### 数据构建策略

```
能力维度及其数据来源：
┌────────────┬──────────────────────────────────────────┐
│ 能力维度    │ 数据来源                                  │
├────────────┼──────────────────────────────────────────┤
│ 通用对话    │ 人工标注 + 合成                           │
│ 代码       │ 代码生成+执行验证 → 拒绝采样选最佳         │
│ 数学       │ 问题+解题步骤+答案验证 → 拒绝采样           │
│ 推理       │ CoT 标注 + 合成                           │
│ 长文本     │ 长文档摘要+问答                            │
│ 工具调用   │ Function Calling 标注数据                  │
│ 多语言     │ 翻译+多语言标注                            │
│ 安全       │ 红队测试 + 安全拒绝标注                    │
└────────────┴──────────────────────────────────────────┘
```

#### 拒绝采样 (Rejection Sampling)

Llama 3 大量使用拒绝采样来提升数据质量：

```
过程：
  1. 给定一个 prompt
  2. 用多个模型（包括不同版本的 Llama 3）各生成 K 个响应
  3. 用 Reward Model 对所有响应打分
  4. 选择得分最高的响应作为 SFT 训练数据
  5. 在每一轮迭代中，模型质量提升 → 采样质量提升 → 循环正反馈

关键：这不是 RL，但使用了 Reward Model
  → 是 SFT 和 RL 之间的 "中间地带"
  → 简单、稳定、效果好
```

#### 数据混合调优

Llama 3 团队花了大量精力在数据混合比例上：
- 不同能力维度的数据比例需要仔细调节
- 代码和数学数据的比例通常需要**高于自然分布**
- 安全数据过多会导致 "过度拒绝"（拒绝合理请求）
- 通过消融实验确定最优比例

### 6.5 DeepSeek-R1 的冷启动 SFT 策略

**论文**：[arXiv:2501.12948](https://arxiv.org/abs/2501.12948)

DeepSeek-R1 的冷启动 SFT 是一个精巧的设计，值得单独分析。

#### 为什么需要冷启动 SFT？

```
R1-Zero 实验发现：
  纯 RL (GRPO) 训练可以涌现推理能力
  但存在问题：
  1. 输出格式不稳定（混合不同语言、格式混乱）
  2. 推理过程冗长且不必要
  3. 可读性差

冷启动 SFT 的作用：
  给模型一个 "格式化思考" 的起点
  → RL 在此基础上提升推理质量
  → 不会陷入格式混乱的局部最优
```

#### 冷启动数据构建

```
数据量：数千条（不是数万条）
数据内容：
  ├── 高质量的链式推理 (CoT) 示例
  │   ├── 包含 <think>...</think> 标签的思考过程
  │   └── 包含最终答案
  │
  ├── 来源：
  │   ├── 人工编写的推理示例
  │   ├── 从 DeepSeek-R1-Zero 采样的优质推理轨迹（经人工筛选）
  │   └── 少量 few-shot 提示 DeepSeek-V3 生成的 CoT
  │
  └── 质量控制：
      ├── 答案必须正确
      ├── 推理过程必须清晰、有逻辑
      └── 格式必须规范（<think> 标签）
```

#### 完整 R1 训练流水线

```
Stage 1: 冷启动 SFT
  DeepSeek-V3 Base → 数千条 CoT 数据 SFT → DeepSeek-R1-SFT

Stage 2: 推理 RL
  DeepSeek-R1-SFT → GRPO (数学/代码用可验证奖励) → DeepSeek-R1-RL

Stage 3: 拒绝采样 + 非推理数据整合
  DeepSeek-R1-RL 生成大量推理轨迹 → 过滤正确答案
  + 收集通用对话、创意写作等非推理 SFT 数据

Stage 4: 最终 SFT
  DeepSeek-V3 Base → 在 Stage 3 数据上 SFT → DeepSeek-R1 (Final)
  注意：最终 SFT 是从 Base 模型重新开始的！不是在 RL 模型上继续！
```

**为什么最终 SFT 要从 Base 模型重新开始？**
- RL 训练可能引入一些不希望的行为模式
- 从 Base 重新 SFT 可以 "蒸馏" RL 模型的推理能力到一个干净的模型
- 同时整合推理和非推理能力

## 七、SFT 的常见陷阱

### 陷阱 1：过拟合到格式
```
症状：模型过度遵循 SFT 数据的特定格式
  - 总是列出要点，即使不需要
  - 总是以特定方式开头（"Certainly!", "Great question!"）
  - 回答长度固定

解决：
  - 增加格式多样性
  - 使用不同长度的回答
  - 包含简短回答的示例
```

### 陷阱 2：能力遗忘
```
症状：SFT 后某些预训练能力下降
  - 常见于小模型
  - 特别是数学/代码能力在通用 SFT 后下降

解决：
  - 在 SFT 数据中混入目标领域的数据
  - 减少 SFT epoch 数
  - 使用更低的学习率
```

### 陷阱 3：安全-有用性权衡
```
症状：安全训练过度导致模型拒绝合理请求
  - "我不能帮你做这个..."（但请求完全合理）
  - 过度谨慎

解决：
  - 仔细平衡安全数据比例
  - 包含 "边界案例" 的正面示例（应该回答但看似敏感的问题）
  - Llama 3 特别提到了这个问题
```

### 陷阱 4：评估数据污染
```
症状：SFT 数据与评估基准有交叉
  - 导致基准分数虚高
  - 不反映真实能力

解决：
  - 严格的数据去污染 (decontamination)
  - 使用 n-gram 匹配检测交叉
  - 保留独立的评估集
```
