# 评估基准与方法论

> AI 模型的评估基准 (Benchmark) 是衡量模型能力进步的标尺。2024-2026 年，随着模型能力快速提升，基准测试经历了"饱和 → 升级 → 再饱和"的快速迭代周期。本文系统梳理主流基准、评估框架、以及基准污染问题。

---

## 一、标准 LLM 基准

### 1.1 MMLU / MMLU-Pro

**MMLU (Massive Multitask Language Understanding)**
- **组织**：UC Berkeley (Dan Hendrycks 等)
- **论文**：[Measuring Massive Multitask Language Understanding](https://arxiv.org/abs/2009.03300) (ICLR 2021)
- **数据集**：15,908 道四选一选择题，覆盖 57 个学科（从小学数学到专业法律、医学）
- **学科范围**：STEM、人文、社会科学、专业领域（如临床知识、法学、计算机科学等）
- **评估方式**：多选正确率 (Accuracy)
- **现状 (2025)**：前沿模型已达 90%+ 准确率，基准趋于饱和

**MMLU-Pro**
- **组织**：多方合作
- **论文**：[MMLU-Pro: A More Robust and Challenging Multi-Task Language Understanding Benchmark](https://arxiv.org/abs/2406.01574)
- **改进点**：
  - 选项从 4 个扩展到 **10 个**，显著降低猜对概率
  - 删除了简单/模糊题目，聚焦推理密集型任务
  - 对提示变化的敏感度从 4-5% 降低到 2%
- **当前得分 (2025 末)**：
  - Gemini 3 Pro Preview: ~89.8%
  - Claude Opus 4.5 (Reasoning): ~89.5%
  - 中等水平模型: ~80-85%
- **扩展**：MMLU-ProX 将 11,829 道题翻译为 29 种语言，支持跨语言对比

```
MMLU 系列演进：
  MMLU (2020)      → 57 学科, 4 选项, 15K 题 → 已饱和
  MMLU-Pro (2024)  → 10 选项, 推理聚焦 → 当前主流
  MMLU-ProX (2025) → 29 种语言, 跨语言评估 → 多语言扩展
  MMLU-CF (2025)   → Microsoft 的去污染版本 → 解决数据泄露
```

### 1.2 HumanEval / MBPP (代码生成)

**HumanEval**
- **组织**：OpenAI
- **论文**：[Evaluating Large Language Models Trained on Code](https://arxiv.org/abs/2107.03374) (2021)
- **数据集**：164 道手写 Python 编程题，给定函数签名和 docstring，模型需补全函数体
- **评估指标**：pass@k（k 次采样中至少一次通过所有测试用例的概率）
- **GitHub**：[openai/human-eval](https://github.com/openai/human-eval)
- **原始结果**：Codex 首发 28.8% pass@1
- **当前得分 (2025)**：前沿模型 pass@1 已超 96%（如 o1-mini 96.2%），基准趋于饱和

**MBPP (Mostly Basic Programming Problems)**
- **组织**：Google Research
- **论文**：[Program Synthesis with Large Language Models](https://arxiv.org/abs/2108.07732) (2021)
- **数据集**：974 道入门级 Python 编程题，每题包含自然语言描述 + 3 个 assert 测试
- **评估指标**：pass@1
- **特点**：题目难度较 HumanEval 低，聚焦数值、列表、字符串操作

**进阶版本**：
- **HumanEval Pro / MBPP Pro** (2024)：评估自调用代码生成能力
  - HumanEval Pro 的 pass@1 显著低于 HumanEval（o1-mini: HumanEval 96.2% vs HumanEval Pro 76.2%）
- **EvalPlus**：扩展测试用例，更严格的正确性验证
  - 排行榜：[evalplus.github.io/leaderboard](https://evalplus.github.io/leaderboard.html)
- **BigCodeBench**：HuggingFace 推出的下一代代码基准
  - 更复杂的编程任务，包含库调用和多步逻辑

### 1.3 MATH / GSM8K (数学推理)

**MATH**
- **组织**：UC Berkeley (Dan Hendrycks 等)
- **论文**：[Measuring Mathematical Problem Solving With the MATH Dataset](https://arxiv.org/abs/2103.03874) (NeurIPS 2021)
- **数据集**：12,500 道竞赛数学题（来自 AMC 10/12、AIME 等），5 个难度级别，7 个学科
- **特点**：每题有完整的 LaTeX 步骤解答，最终答案用 `\boxed{}` 标记
- **原始结果**：GPT-3 仅约 5%
- **当前得分 (2025)**：前沿推理模型（o1、DeepSeek-R1）在 MATH-500 上已达 97%+，接近饱和

**GSM8K (Grade School Math 8K)**
- **组织**：OpenAI
- **论文**：[Training Verifiers to Solve Math Word Problems](https://arxiv.org/abs/2110.14168) (2021)
- **作者**：Karl Cobbe 等
- **数据集**：8,500 道小学水平数学应用题（7.5K 训练 + 1K 测试）
- **特点**：每题需 2-8 步基础算术运算，设计为中学生能解决
- **现状 (2025)**：前沿模型达 ~95% 准确率，已基本饱和
- **改进版本**：
  - **GSM8K-Platinum**：修正了原始数据中的错误，揭示了原本被隐藏的模型差异
  - **GSM8K-V**：纯视觉版本，最佳模型 (Gemini-2.5-Pro) 仅 46.93%，远低于文本版 95%

### 1.4 GPQA (研究生级科学问答)

- **组织**：NYU 等 (David Rein 等)
- **论文**：[GPQA: A Graduate-Level Google-Proof Q&A Benchmark](https://arxiv.org/abs/2311.12022) (2023)
- **数据集**：
  - Extended: 546 题
  - Main: 448 题
  - **Diamond**: 198 题（最核心、最高质量的子集）
- **学科**：生物学、物理学、化学
- **设计原则**："Google-Proof" --- 即使有无限制的互联网搜索，非领域专家也难以答对
- **人类基线**：领域专家 ~69.7%，非领域专家 ~33%
- **当前得分 (2025 末)**：
  - 顶级模型已达 91-93%（如 Gemini 3 Pro ~91.9%）
  - **AI 已大幅超越人类领域专家**
  - 基准接近饱和
- **GitHub**：[idavidrein/gpqa](https://github.com/idavidrein/gpqa)

### 1.5 ARC-AGI / ARC-AGI-2 (抽象推理)

- **组织**：ARC Prize Foundation (Francois Chollet)
- **网站**：[arcprize.org](https://arcprize.org/)
- **核心理念**：测试"流体智力" --- 在未见过的抽象任务上进行推理和泛化的能力
- **任务形式**：输入-输出的网格变换模式，给定几个示例，推断变换规则并应用到测试输入

```
ARC-AGI 系列演进：
  ARC-AGI (2019)   → 原始版本，400 道公开 + 400 道私密测试题
  ARC-AGI-2 (2025) → 更难版本，2025 年竞赛使用
  ARC-AGI-3 (2026) → 首次重大格式变化，要求交互式推理
```

**ARC-AGI-2 (2025 竞赛)结果**：
- **纯 LLM 在 ARC-AGI-2 上得分为 0%** --- 表明当前 LLM 缺乏真正的抽象推理能力
- Kaggle 顶级方案：24%（$0.20/任务成本约束下）
- Poetiq（学习型测试时推理）：54%（$30/任务，使用 Gemini 3 Pro）
- 人类基线：~85%+

**ARC-AGI-3 (计划 2026 初发布)**：
- 首次引入交互式推理
- 需要探索、规划、记忆、目标获取和对齐等能力

**关键洞察**：ARC-AGI 是当前最能揭示 LLM 泛化缺陷的基准。纯语言模型在此基准上的 0% 得分表明，大规模语言建模并不自动赋予真正的抽象推理能力。

### 1.6 HLE (Humanity's Last Exam)

- **组织**：Center for AI Safety (CAIS) + Scale AI
- **发起人**：Dan Hendrycks（CAIS 主任）
- **论文**：[Humanity's Last Exam](https://arxiv.org/abs/2501.14249) (2025)
- **网站**：[agi.safe.ai](https://agi.safe.ai/)
- **排行榜**：[scale.com/leaderboard/humanitys_last_exam](https://scale.com/leaderboard/humanitys_last_exam)
- **数据集**：2,500 道极高难度题目，覆盖 100+ 学科
- **众包设计**：来自 50 个国家、500 个机构的 1,000+ 学科专家贡献
- **人类专家基线**：~90%（在各自专业领域）

**当前得分 (2026 年 2 月)**：
| 模型 | 得分 |
|------|------|
| Gemini 3 Deep Think | ~48.4% |
| Gemini 3 Pro Preview | ~37.5% |
| GPT-5 Pro | ~31.6% |

- 约一年内从个位数提升到接近 50%，但仍远低于人类专家水平
- 模型存在严重的**校准误差**（34%-89%），系统性地过度自信
- 创建动机：Dan Hendrycks 认为现有基准（如 MMLU）对前沿模型已过于简单

### 1.7 LiveBench

- **组织**：Abacus.AI 等
- **论文**：[LiveBench: A Challenging, Contamination-Free LLM Benchmark](https://arxiv.org/abs/2406.19314) (ICLR 2025 Spotlight)
- **网站**：[livebench.ai](https://livebench.ai/)
- **核心设计**：
  1. **每月更新**：基于最新新闻、论文、数据集生成题目，天然抗污染
  2. **客观评分**：所有题目有可验证的标准答案，不依赖 LLM 裁判
  3. **覆盖面广**：数学、编程、推理、语言、指令遵循、数据分析
- **当前得分**：顶级模型 < 70%，保持较好的区分度
- **GitHub**：[LiveBench/LiveBench](https://github.com/LiveBench/LiveBench)

### 1.8 Arena-Hard

- **组织**：LMSYS / LMArena
- **论文**：[From Crowdsourced Data to High-Quality Benchmarks: Arena-Hard and BenchBuilder Pipeline](https://arxiv.org/abs/2406.11939) (ICML 2025)
- **方法论**：
  - 从 Chatbot Arena 真实用户对话中自动挖掘高质量难题
  - 使用 BenchBuilder 管线：收集 → 去重 → 聚类 → 筛选
  - LLM-as-Judge 框架进行两局对比评估，消除位置偏差
  - Bradley-Terry 模型计算最终得分
- **规模**：500 道高难度开放式提示
- **性能**：
  - 模型区分度是 MT-Bench 的 **3 倍**
  - 与 Chatbot Arena 人类偏好的一致性达 **98.6%**
  - 评估成本仅约 $20
- **与 AlpacaEval 2.0 对比**：AlpacaEval 2.0 (Length-Controlled) 与 Chatbot Arena 的 Spearman 相关性达 0.98，成本 < $10

## 二、Agent 基准

### 2.1 SWE-bench / SWE-bench Verified

- **组织**：Princeton NLP (Carlos E. Jimenez, John Yang, Ofir Press, Karthik Narasimhan 等)
- **论文**：[SWE-bench: Can Language Models Resolve Real-world Github Issues?](https://arxiv.org/abs/2310.06770) (ICLR 2024 Oral)
- **网站**：[swebench.com](https://www.swebench.com/)

**方法论**：
```
SWE-bench 任务流程：
  输入：一个真实 GitHub Issue 的文本描述
  环境：对应仓库在该 PR 之前的 commit 状态 (Docker 镜像)
  要求：修改代码以解决 Issue
  评估：运行 "Fail-to-Pass" 测试 --- 修复前失败、修复后通过的测试

数据来源：12 个流行 Python 仓库的 Issue-PR 对
  原始版本：2,294 个任务实例
  SWE-bench Verified：500 个经人工验证的高质量实例
  SWE-bench Lite：300 个自筛选子集
```

**当前排行榜 (2026 年 2 月, SWE-bench Verified)**：

| 模型/系统 | 得分 | 备注 |
|-----------|------|------|
| Claude Opus 4.5 | 最高 | SWE-bench Verified 榜首 |
| Gemini 3 Flash | 第二 | |
| MiniMax M2.5 | 第三 | |
| Claude 4.5 Sonnet | ~43.6% | SWE-bench Pro |
| Gemini 3 Pro Preview | ~43.3% | SWE-bench Pro |

**SWE-bench Pro** (Scale AI)：更难的变体，顶级系统得分约 45%

**得分演进**：
```
SWE-bench Verified 历史得分：
  2024.03  Devin 首发          ~14%
  2024.06  快速提升            ~30%
  2024.12  突破 50%           ~50%
  2025.06  Claude 3.5          ~65%
  2025 下半年                  ~72%
  2026.02  v2.0 更新后重新评估  进一步提升
```

### 2.2 GAIA

- **组织**：Meta AI (前 Facebook AI Research)
- **论文**：[GAIA: a benchmark for General AI Assistants](https://arxiv.org/abs/2311.12983)
- **网站**：[huggingface.co/spaces/gaia-benchmark/leaderboard](https://huggingface.co/spaces/gaia-benchmark/leaderboard)
- **数据集**：466 道真实世界问题，3 个难度级别
- **评估维度**：推理、多模态处理、网页浏览、工具使用
- **人类基线**：92%
- **当前得分 (2025)**：h2oGPTe Agent 首次达到 75%（"C 级"成绩），超越 OpenAI Deep Research
- **进化版本**：
  - **Gaia2** (2025)：评估异步环境中的 Agent 行为，环境独立于 Agent 动作进化
  - Gaia2 上 GPT-5 (high) 仅 42% pass@1

### 2.3 WebArena

- **组织**：Carnegie Mellon University (CMU)
- **论文**：[WebArena: A Realistic Web Environment for Building Autonomous Agents](https://arxiv.org/abs/2307.13854)
- **网站**：[webarena.dev](https://webarena.dev/)
- **数据集**：812 个长序列 Web 操作任务，基于 241 个模板
- **环境**：自托管的 4 类真实网站（电商、论坛、CMS、代码管理）的完整模拟
- **评估方式**：端到端任务完成率
- **人类基线**：78.24%

**得分演进**：
```
WebArena 历史得分：
  2023  GPT-4 Agent 首发    ~14.4%
  2024  快速提升            ~30-40%
  2025  最佳单 Agent 系统   ~61.7%（两年内从 14% → 62%）
```

**扩展版本**：
- **VisualWebArena** (CMU)：视觉多模态 Web Agent 评估
- **WebArena Verified** (NeurIPS 2025)：修复评估漏洞，增强可重现性
- **WebChoreArena**：532 个高挑战任务，强调海量记忆和跨页面推理

### 2.4 BrowseComp

- **组织**：OpenAI
- **论文**：[BrowseComp: A Simple Yet Challenging Benchmark for Browsing Agents](https://arxiv.org/abs/2504.12516) (2025)
- **数据集**：1,266 道高难度信息检索题
- **设计原则**：
  - 要求 Agent 在数十甚至上百个网页间持续导航
  - 答案简短且可精确验证
  - 区别于 SimpleQA 等已被"简单搜索"饱和的基准
- **得分**：
  - Deep Research（OpenAI）：51.5% 单次尝试，~78% Best-of-N 聚合
  - 其他模型显著低于 Deep Research
- **扩展**：MM-BrowseComp --- 多模态版本

### 2.5 OSWorld

- **组织**：XLang Lab
- **论文**：[OSWorld: Benchmarking Multimodal Agents for Open-Ended Tasks in Real Computer Environments](https://arxiv.org/abs/2404.07972) (NeurIPS 2024)
- **网站**：[os-world.github.io](https://os-world.github.io/)
- **数据集**：369 个真实桌面操作任务（Ubuntu、Windows、macOS）
- **评估方式**：在真实操作系统环境中执行任务，验证最终状态
- **人类基线**：~72%
- **当前最佳 (2025 末)**：CoACT-1 达 60.76%（人类能力的 84.4%）
- **OSWorld-Verified (2025)**：改进版，迁移到 AWS、支持 50x 并行化、任务质量增强

**得分分布**：
```
OSWorld 模型表现分层：
  Agentic 框架:       45-61%  (CoACT-1 等)
  高级基础模型:        35-44%  (GPT-5, Gemini 系列)
  专化模型:           25-40%
  人类:               ~72%
```

### 2.6 tau-bench

- **组织**：Sierra (AI 客服公司)
- **论文**：[tau-bench: A Benchmark for Tool-Agent-User Interaction in Real-World Domains](https://arxiv.org/abs/2406.12045) (2024)
- **网站**：[taubench.com](https://taubench.com/)
- **GitHub**：[sierra-research/tau-bench](https://github.com/sierra-research/tau-bench)

**核心设计**：
```
tau-bench 模拟真实客服场景：
  用户 (由 LLM 模拟) ← 多轮对话 → Agent (被测模型)
                                        │
                                   工具调用 (API)
                                        │
                                   数据库读写

评估不仅检查 Agent 回复，还验证数据库状态变更
例：取消订单任务 → 验证数据库中订单状态确实变为 "cancelled"
```

**领域覆盖**：
- 原版：零售 (Retail) + 航空 (Airline)
- tau^2-bench (2025)：新增电信 (Telecom) 领域

**独特指标**：pass^k --- 同一任务重复 k 次的成功率
- 顶级模型（GPT-4）首次成功率 < 50%，重复 8 次仅 ~25%
- Anthropic 在 Claude 模型卡中采纳了此指标来衡量一致性

## 三、评估框架

### 3.1 lm-evaluation-harness (EleutherAI)

- **组织**：EleutherAI
- **GitHub**：[EleutherAI/lm-evaluation-harness](https://github.com/EleutherAI/lm-evaluation-harness)
- **定位**：LLM 评估的"事实标准"工具

**核心特点**：
```
lm-evaluation-harness
├── 60+ 标准学术基准，数百个子任务/变体
├── 统一接口：任何因果语言模型都能在完全相同的输入和代码基上评测
├── 可扩展：自定义任务只需编写 YAML 配置
├── 多后端支持：HuggingFace, GPTQ, vLLM, OpenAI API 等
└── 广泛采用：NVIDIA, Cohere, BigScience, Mosaic ML, Hugging Face Open LLM Leaderboard
```

**使用者**：
- Hugging Face **Open LLM Leaderboard** 的后端引擎
- 被数百篇研究论文引用
- Mozilla Foundation 认证的 LLM 审计工具

### 3.2 HELM (Stanford)

- **组织**：Stanford CRFM (Center for Research on Foundation Models)
- **网站**：[crfm.stanford.edu/helm](https://crfm.stanford.edu/helm/latest/)
- **论文**：[Holistic Evaluation of Language Models](https://arxiv.org/abs/2211.09110) (2022)

**核心特点**：
```
HELM 设计理念：
  H - Holistic (全面)：多维度评估，不仅看准确率
  E - Evaluation
  L - Language
  M - Models

评估维度：
├── 准确性 (Accuracy)
├── 校准度 (Calibration) — 模型对自己答案的置信度是否准确
├── 鲁棒性 (Robustness) — 对输入变化的稳定性
├── 公平性 (Fairness) — 不同人群间的表现差异
├── 偏见与刻板印象 (Bias & Stereotypes)
├── 毒性 (Toxicity)
└── 效率 (Efficiency)
```

- 当前评估 30+ 主流 LLM，涵盖闭源和开源
- 强调**透明性**和**多维度**评估，而非单一分数排名

### 3.3 OpenCompass (上海 AI 实验室)

- **组织**：上海人工智能实验室 (Shanghai AI Lab)
- **网站**：[opencompass.org.cn](https://opencompass.org.cn/)
- **GitHub**：[open-compass/opencompass](https://github.com/open-compass/opencompass)

**核心特点**：
```
OpenCompass 评估体系：
├── 六大通用能力维度
│   ├── 考试 (Examination)
│   ├── 知识 (Knowledge)
│   ├── 语言 (Language)
│   ├── 理解 (Understanding)
│   ├── 推理 (Reasoning)
│   └── 安全 (Safety)
│
├── 评估方式
│   ├── 客观评估：选择题、填空题、封闭问答
│   └── 主观评估：LLM 辅助 + 人类反馈
│
├── OpenCompass 2.0 三大组件
│   ├── CompassKit — 评估工具包
│   ├── CompassHub — 数据集中心
│   └── CompassRank — 排行榜（含开源 + 专有基准）
│
└── 支持 100+ 数据集，广泛模型兼容
    Llama, Mistral, InternLM, GPT-4, Qwen, GLM, Claude...
```

**特色**：
- 中国最权威的 LLM 评估平台，国产模型覆盖最全
- 配套 **VLMEvalKit** 评估多模态模型（220+ LMMs, 80+ 基准）
- 定期发布评估报告和排行榜更新

### 3.4 Chatbot Arena (LMArena)

- **组织**：LMSYS Org → Arena Intelligence Inc. (2025 年独立公司化)
- **网站**：[lmarena.ai](https://lmarena.ai/) (原 lmsys.org)
- **论文**：[Chatbot Arena: An Open Platform for Evaluating LLMs by Human Preference](https://arxiv.org/abs/2403.04132)

**方法论**：
```
Chatbot Arena 评估流程：
  1. 用户输入问题
  2. 系统随机分配两个匿名模型 (Model A, Model B)
  3. 两个模型同时生成回答
  4. 用户投票：A 更好 / B 更好 / 平手 / 都差
  5. 投票后揭示模型身份
  6. 使用 Bradley-Terry 模型拟合所有投票数据
  7. 计算 Elo-like 评分 + 置信区间
```

**数据规模 (截至 2025 末)**：
| Arena | 总投票数 | 模型数 |
|-------|---------|--------|
| 文本 Arena | 4,921,958 | 100+ |
| 视觉 Arena | 585,217 | 90 |
| 搜索 Arena | 122,219 | 15 |
| 文生图 Arena | 4,073,799 | 38 |

- **月活跃用户**：500 万+，覆盖 150 个国家
- **月对话量**：6,000 万+

**评分系统演进**：
```
早期：Online Elo 系统（来自国际象棋）
  ↓
中期：Bradley-Terry 模型（更稳定的评分、更精确的置信区间）
  ↓
当前：分类别排行榜（编程、数学、创意写作、指令遵循等）
```

**优势与局限**：
| 优势 | 局限 |
|------|------|
| 最接近真实用户偏好 | 受众偏差（技术用户偏多） |
| 匿名对比消除品牌效应 | 缺少客观任务评估 |
| 数据量大，统计可靠 | 反博弈控制透明度不足 |
| 覆盖模型广，更新快 | 短对话偏向（缺少长任务评估） |

## 四、基准污染问题 (Benchmark Contamination)

> 基准污染是当前 LLM 评估面临的最严重挑战之一：当训练数据包含评估题目时，高分反映的是"记忆"而非"能力"。

### 4.1 问题定义

```
基准污染 (Data Contamination)：
  模型的训练数据中包含（部分或全部）评估基准的测试题目和答案。
  结果：模型通过记忆而非推理获得高分，评估结果失去意义。
```

### 4.2 已证实的污染案例

**GSM8K 污染**：
- 移除被污染样本后，部分模型准确率**下降高达 13%**
- 这 13% 是下界而非上界
- Phi-3 在去污染后 GSM8K 下降 5.3%、MMLU 下降 6.7%
- Mistral 受影响较小：GSM8K 下降 0.5%、MMLU 下降 3.6%
- 31 个 LLM 的系统分析表明：数学推理领域存在**广泛的数据污染**

**MMLU 污染**：
- ChatGPT 和 GPT-4 在猜测缺失选项测试中，精确匹配率分别达 52% 和 57%
- 表明模型可能已经"见过"部分 MMLU 测试题
- Qwen2.5-3B 被检测出在 MMLU 上存在污染

**代码基准污染**：
- HumanEval/MBPP 的题目在 GitHub 公开代码中广泛存在
- 代码重构（变量重命名、结构变换）被证明可以有效降低污染效应

### 4.3 污染规模研究 (ICML 2025)

- 实验量化了三个维度上的污染影响：
  - 模型参数量（最多 1.6B）
  - 样本被见过的次数（最多 144 次）
  - 训练 token 总量（最多 40B）
- **关键发现**：即使被污染 144 次，如果训练数据规模超过 Chinchilla 最优量的 5 倍，污染效应可以被"遗忘"
- 这意味着：大规模训练的现代模型可能自然稀释了污染效应，但并非完全消除

### 4.4 应对策略

| 策略 | 方法 | 代表 | 局限 |
|------|------|------|------|
| **动态基准** | 定期更新题目，基于最新数据 | LiveBench | 需要持续维护 |
| **去污染版本** | 构建已知无污染的替代数据集 | MMLU-CF, GSM8K-Platinum | 可能改变难度分布 |
| **推理时去污染** | 检测并修正受污染的模型输出 | ITD (Inference-Time Decontamination) | 计算开销大 |
| **代码重构** | 对代码题进行语义等价变换 | CodeCleaner | 仅适用于代码 |
| **污染检测** | 主动检测训练数据中的基准泄露 | AntiLeakBench | 检测不完全 |
| **私密测试集** | 永不公开测试数据 | ARC-AGI 私密集 | 限制可重现性 |

### 4.5 关键论文

- [A Survey on Data Contamination for Large Language Models](https://arxiv.org/abs/2502.14425) --- 全面综述
- [When Benchmarks Leak: Inference-Time Decontamination for LLMs](https://arxiv.org/abs/2601.19334) --- 推理时去污染
- [MMLU-CF: A Contamination-free Multi-task Language Understanding Benchmark](https://github.com/microsoft/MMLU-CF) --- Microsoft 的去污染 MMLU (ACL 2025)
- [How Much Can We Forget about Data Contamination?](https://openreview.net/forum?id=Pf0PaYS9KG) --- ICML 2025

## 六、趋势与洞察

### 6.1 基准饱和加速

```
基准饱和周期（从发布到前沿模型达 90%+）：
  MMLU (2020):      ~4 年
  GSM8K (2021):     ~3 年
  MATH (2021):      ~3.5 年
  HumanEval (2021): ~3 年
  GPQA (2023):      ~2 年
  MMLU-Pro (2024):  ~1.5 年（预计）

趋势：基准饱和周期从 4 年缩短到 1-2 年
```

### 6.2 评估范式转移

```
评估方法论演进：
  第一代：静态选择题基准 (MMLU, GSM8K, MATH)
    → 简单，但易饱和、易污染

  第二代：动态/在线基准 (LiveBench, Chatbot Arena)
    → 抗污染，但维护成本高

  第三代：Agent 环境基准 (SWE-bench, WebArena, OSWorld)
    → 接近真实任务，但评估成本极高

  第四代（新兴）：交互式/对抗基准 (ARC-AGI-3, tau-bench)
    → 测试真实泛化能力，最接近 AGI 评估
```

### 6.3 关键洞察

1. **没有单一基准能全面衡量模型能力**：需要多维度基准组合
2. **Chatbot Arena 仍是最受信赖的"综合排名"**：因为基于真实用户偏好
3. **ARC-AGI 是当前最能揭示 LLM 局限性的基准**：纯 LLM 得分为 0%
4. **HLE 代表了"上限测试"的方向**：前沿模型仍不到 50%
5. **Agent 基准正在成为最重要的评估维度**：因为 Agent 是最重要的应用方向
6. **基准污染是真实且严重的问题**：部分高分可能有 5-13% 的"虚假成分"
