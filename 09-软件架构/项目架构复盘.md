# 项目架构复盘

> 用自己的真实项目练习架构分析。每个项目从三个角度审视：
> 1. **架构风格**: 属于什么模式？为什么选这个？
> 2. **权衡决策**: 做了哪些取舍？回头看是否合理？
> 3. **演进方向**: 如果继续发展，架构应该怎么变？

---

## Gateway + Skill Pack 体系

### 现状描述
- optima-gateway 作为 Agent 运行时
- Skill Pack 机制：browser、cron、file-manager 三个 pack，通过 mergePacks() 组合
- Session 生命周期绑定（destroy() 自动清理 cron 任务等）

### 架构分析

**架构风格**:
- 微内核 (Microkernel) — Gateway 是核心，Skill Pack 是插件
- 还是微服务？还是 modular monolith？
- → 读完 Fundamentals Ch11-15 后回来分析

**关键决策**:
- mergePacks() 合并 vs 独立注册 — 选了合并，为什么？
- Session 绑定 vs 全局资源池 — 选了绑定，代价是什么？
- onDestroy 清理 vs GC 式回收 — 选了显式清理

**待思考**:
- 如果 Skill Pack 增长到 20 个，mergePacks() 还能用吗？
- 跨 Session 共享浏览器实例是否有价值？
- Pack 之间的依赖关系如何管理？

## analyze → edit → render Worker 链路

### 现状描述
- 三个独立 worker，通过消息/API 串联
- 每个 worker 有自己的运行环境（analyze 需要 GPU + CUDA）

### 架构分析

**架构风格**:
- 管道模式 (Pipeline) — 数据流经多个处理阶段
- → 读完 DDIA Ch10-11 后对照分析

**关键决策**:
- 同步调用 vs 异步消息队列
- 每阶段独立容器 vs 单一大容器
- 错误处理：重试 vs 补偿 vs 人工介入

**待思考**:
- 中间结果的持久化策略
- 某个 worker 失败后的回滚机制
- 如何做到部分重跑（只重跑 render 不重跑 analyze）
