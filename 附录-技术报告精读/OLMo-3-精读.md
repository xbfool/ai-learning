## 一、为什么读这篇报告

大模型时代的"开源"是一个被严重滥用的词。绝大多数所谓的"开源模型"只公开了推理权重，训练数据、训练代码、中间检查点全部封闭。这意味着：你可以用它，但你无法理解它、复现它、审计它。

OLMo 3 是目前**唯一真正完全开源**的前沿级语言模型项目。它开源了：

1. **预训练数据 Dolma 3**（9.3T tokens 语料池）
2. **数据处理代码** Dolma Toolkit（从原始 Common Crawl 到最终训练集的完整管线）
3. **训练代码**（完整的分布式训练框架）
4. **全过程训练检查点**（不是只有最终权重，而是训练过程中的所有中间状态）
5. **评估代码与框架** OLMES
6. **后训练数据与代码** Tulu 3 / Dolci

对于想理解"一个大模型到底是怎么从零训出来的"的学习者来说，OLMo 3 是唯一能看到完整流程的项目。

### 核心要点

- OLMo 3 用 ~6T tokens 的高质量数据训练 32B 模型，性能匹配使用 15-18T tokens 的 Llama 3.1 和 Qwen 2.5 同尺寸模型
- 揭示了一个核心洞察：**数据质量和后训练技巧比模型规模更重要**
- 训练成本约 $2.25M（32B 模型），相对于同类模型非常经济
- Apache 2.0 协议，商业使用完全自由

## 二、"完全开源"的真正含义 ★★★

这是 OLMo 3 报告中最具意义的部分。在 AI 安全和可信 AI 的讨论中，"可审计性"和"可复现性"是核心诉求，而 OLMo 3 是目前唯一满足这些要求的前沿模型。

### 开源程度对比表

| 开源组件 | OLMo 3 | Llama 3.1 | Qwen 2.5 | Gemma 2 |
|----------|--------|-----------|----------|---------|
| 模型权重 | 完全公开 | 公开 | 公开 | 公开 |
| 推理代码 | 公开 | 公开 | 公开 | 公开 |
| 训练代码 | **完全公开** | 不公开 | 不公开 | 不公开 |
| 预训练数据 | **Dolma 3 完全公开** | 不公开 | 不公开 | 不公开 |
| 训练检查点（全过程） | **所有中间检查点** | 仅最终权重 | 仅最终权重 | 仅最终权重 |
| 数据处理代码 | **Dolma Toolkit 公开** | 不公开 | 不公开 | 不公开 |
| 评估框架 | **OLMES 公开** | 部分 | 部分 | 部分 |
| 后训练数据 | **Dolci / Tulu 3 公开** | 不公开 | 不公开 | 不公开 |
| 协议 | Apache 2.0 | Llama License | Qianwen License | Gemma License |
| 商业使用 | 完全自由 | 7 亿月活以上需授权 | 有限制条款 | 有限制条款 |

### "玻璃盒" vs "黑盒"

传统的"开源权重"模式本质上是一个**黑盒**：你得到的是训练完成后的最终产物，无法知道模型学了什么数据、训练过程中发生了什么、哪些设计决策影响了最终效果。

OLMo 3 的"完全开源"构成了一个**玻璃盒**：

```
玻璃盒（OLMo 3）                    黑盒（Llama / Qwen）
────────────────────                ────────────────────
可审计：训练数据完全透明              无法审计：不知道训练数据包含什么
可复现：任何人都能从零重跑            无法复现：缺少数据和代码
可 Fork：可以修改任何环节重新训练     只能微调：无法改变预训练决策
可研究：训练动态可通过检查点分析      无法研究：只有最终快照
```

### 这对谁有价值？

| 角色 | 价值 |
|------|------|
| **研究者** | 可以研究训练动态、数据影响、能力涌现等基础问题 |
| **企业** | 合规审计（如欧盟 AI Act），训练数据透明 |
| **开发者** | 可以基于完整管线做领域定制，不只是微调 |
| **监管方** | 唯一可审计的前沿模型，可验证数据合规性 |

### 核心要点

- "开源权重"和"完全开源"有本质区别，OLMo 3 是后者的标杆
- 完全开源使得可复现性、可审计性、可 Fork 性成为可能
- Apache 2.0 是当前最宽松的开源协议，商业使用无任何限制
- 这为"真正的开源 AI"设立了行业标准

## 三、Dolma 3 数据集 ★★★

Dolma 3 是 OLMo 3 的预训练数据集，也是目前最大且最透明的开源预训练语料之一。数据工程是 OLMo 3 能以更少数据量达到竞品性能的关键。

### 数据规模

- **总语料池**: 9.3T tokens（约 11,600 GB 原始文本）
- **最终生产集 (Dolma 3 Mix)**: 5.9T tokens
- 数据处理工具：开源的 Dolma Toolkit

### 数据源分布

| 数据源 | 原始大小 (GB) | 占比 (%) | 说明 |
|--------|--------------|----------|------|
| Web (Common Crawl) | ~9,812 | ~84.5% | 多个 CC 快照经过多层过滤后的网页文本 |
| Code (GitHub) | ~1,043 | ~9.0% | 高质量开源代码，多语言覆盖 |
| Social (Reddit) | ~339 | ~2.9% | 社区讨论，经过质量和安全过滤 |
| Academic (Semantic Scholar) | ~268 | ~2.3% | 学术论文，通过 OlmoOCR 处理 |
| Books | ~100 | ~0.9% | 书籍文本 |
| Wiki | ~50 | ~0.4% | 维基百科等百科内容 |

> 注：百分比为近似值，基于报告中公开的数据。

### 数据处理流水线

Dolma 3 的数据处理分为多个阶段，每个阶段都有明确的技术方案：

```
原始数据（Common Crawl 等）
    │
    ▼
[阶段 1] URL 级过滤与去重
    ├── URL 黑名单排除已知低质量站点
    ├── 遵循 robots.txt，排除禁止爬取的内容
    ├── 排除付费墙内容和受版权保护的来源
    └── 每域名同一 URL 仅保留一页
    │
    ▼
[阶段 2] 文档级过滤
    ├── 语言检测（过滤非目标语言）
    ├── 质量评分（fastText 分类器）
    ├── 安全过滤（有害内容分类器）
    └── 长度过滤（排除过短/过长异常文档）
    │
    ▼
[阶段 3] 段落级去重
    ├── 使用 Bloom Filter（Rust 实现，线性时间复杂度）
    ├── 跨所有 CC 快照全局去重
    └── 移除重复段落、广告内容等
    │
    ▼
[阶段 4] PII 检测与掩码
    ├── 电子邮件地址检测与替换
    ├── 电话号码检测与替换
    └── IP 地址检测与替换
    │
    ▼
[阶段 5] 有害内容过滤
    ├── fastText 分类器过滤色情/暴力/仇恨内容
    └── 多维度安全标注
    │
    ▼
[阶段 6] 基准去污染
    └── 确保训练数据不包含评估基准的内容
    │
    ▼
Dolma 3 生产集（5.9T tokens）
```

### OlmoOCR：学术 PDF 处理

OlmoOCR 是 AI2 自研的 PDF-to-Text 工具，是 Dolma 3 的重要创新之一：

- 处理了来自 Semantic Scholar 的 **1.01 亿篇**学术论文
- 生成约 **972B tokens** 的结构化文本
- 相比传统 OCR 工具，能更好地保留数学公式、表格、图注等结构化信息
- OlmoOCR 工具本身也是开源的（[github.com/allenai/olmocr](https://github.com/allenai/olmocr)）

### Swarm 优化：找到最优数据混合比例 ★★

数据混合比例（即各数据源各占多少）对最终模型性能有巨大影响。OLMo 3 使用了一种名为 **Swarm** 的优化方法来系统性地搜索最优混合：

**方法流程**：

1. **训练代理模型**：训练 30 个 30M 参数的小模型（训练成本极低）
2. **探索混合空间**：每个代理模型使用不同的数据混合比例
3. **评估与迭代**：根据评估结果调整混合比例，多轮迭代
4. **迁移到大模型**：将小模型上找到的最优混合比例应用到 7B/32B 模型

**质量上采样策略**：

Swarm 优化发现，对高质量数据进行适度的上采样（即让模型多看几遍）是有益的：

| 策略 | 细节 |
|------|------|
| 上采样对象 | 质量评分 Top 1% 的数据 |
| 最大重复次数 | 7 次 |
| 控制方法 | 截断幂指数曲线（truncated power-exponential curve） |
| 平均改进 | +0.056 BPB（bits per byte） |
| 最大改进 | +0.209 BPB |

> BPB（Bits Per Byte）是衡量语言模型困惑度的指标，越低越好。0.056 的平均改进在大规模训练中是显著的。

**为什么 Swarm 重要**：

传统做法是凭经验或简单消融实验来确定数据混合比例，而 Swarm 将其转化为一个系统性的优化问题。用 30M 参数的代理模型搜索成本极低（可能只需几百美元），但找到的最优混合在 7B 和 32B 规模上同样有效。这是 OLMo 3 能用更少数据达到竞品性能的关键技术之一。

### 核心要点

- Dolma 3 从 9.3T 总池中精选 5.9T tokens，质量优先于数量
- 数据处理管线涵盖 URL 级、文档级、段落级三层过滤和去重
- OlmoOCR 处理了 1.01 亿篇学术论文，是独特的数据来源
- Swarm 优化用 30 个 30M 代理模型搜索最优混合比例，成本极低但效果显著
- PII 掩码和基准去污染确保了数据的合规性

## 四、三阶段预训练 ★★

OLMo 3 的预训练不是一次性完成的，而是分为三个阶段，每个阶段使用不同的数据集、不同的训练策略、面向不同的能力目标。

### 阶段概览

| 阶段 | 名称 | 数据集 | Tokens (32B) | 计算占比 | 核心目标 |
|------|------|--------|-------------|----------|----------|
| Stage 1 | 广度训练 | Dolma 3 Mix | 5.9T | ~90% | 构建广泛的语言理解和知识基础 |
| Stage 2 | 中期训练 | Dolmino | 100B | ~5-10% | 数学/推理/代码能力定向增强 |
| Stage 3 | 长上下文扩展 | Longmino | 100B | ~5% | 8K → 65K 上下文窗口 |

### Stage 1：广度训练

这是计算量最大的阶段，目标是让模型建立广泛的语言理解和世界知识。

**训练配置**：

| 参数 | 7B 模型 | 32B 模型 |
|------|---------|----------|
| 训练数据 | Dolma 3 Mix | Dolma 3 Mix |
| 训练 Tokens | ~4T | ~5.9T |
| 上下文长度 | 4,096 | 4,096 |
| 优化器 | AdamW | AdamW |
| 学习率调度 | Warmup 5000 步 → 线性衰减到峰值 1/10 | Warmup 5000 步 → 线性衰减到峰值 1/10 |
| 硬件 | 256 H100 GPU | 1024 H100 GPU |
| 吞吐量 | ~7.7K tokens/device/s | ~7.7K tokens/device/s |

**文档打包与掩码**：

训练时使用文档打包（document packing）将多个文档拼接到一个序列中以提高 GPU 利用率，同时使用文档内掩码（intra-document masking）确保注意力不会跨文档计算，避免引入虚假的跨文档关联。

### Stage 2：中期训练 (Midtraining) ★

中期训练是 OLMo 3 最具特色的创新之一。它是在广度训练和后训练之间插入的一个**能力定向增强**阶段。

**Dolmino 数据混合比例**：

| 数据类型 | 占比 | 说明 |
|---------|------|------|
| Web（高质量筛选） | 28% | 经过质量评分二次筛选的优质网页 |
| Code | 20% | 高质量编程数据 |
| Math | 19% | 数学问题、证明、解答 |
| QA（问答） | 14% | 阅读理解和问答数据 |
| Thinking（推理链） | 8% | 包含推理链的数据 |
| Instruction（指令） | 6% | 指令遵循数据 |
| PDF | 5% | OlmoOCR 处理的学术文档 |

**训练策略**：

- 数据池总量 ~2.2T tokens，从中采样 100B tokens
- 7B 模型和 32B 模型各使用 100B tokens
- 使用 512 个 GPU 运行两个并行训练任务，每个覆盖 100B tokens
- 总训练时间约 **1.5 天**
- 训练完成后进行模型合并（model merging）并评估，选择最佳检查点
- 学习率在中期训练开始时重新预热，然后按新调度衰减

**关键发现**：

中期训练的 100B tokens 仅占广度训练 5.9T tokens 的 ~1.7%，但对最终性能的提升极为显著。这说明在正确的时机注入正确类型的数据，效率远高于简单增加预训练数据量。增加数学和代码数据比例可以显著提升推理能力，但会以其他领域性能下降为代价，因此团队精心调整了比例以取得平衡。

### Stage 3：长上下文扩展

**Longmino 数据配置**：

| 参数 | 7B 模型 | 32B 模型 |
|------|---------|----------|
| 训练 Tokens | 50B | 100B |
| 目标上下文 | 8K → 65K | 8K → 65K |
| 长短数据比 | 1:2（34% 长 + 66% 短） | 1:2（34% 长 + 66% 短） |
| 位置编码扩展 | YaRN | YaRN |

**技术细节**：

- 长上下文数据来源：合成长文档 + OlmoOCR 处理的学术 PDF（天然长文档）
- 短上下文数据：复用中期训练阶段的 Dolmino 数据
- 位置编码扩展使用 YaRN（Yet another RoPE extensioN）方法
- 团队测试了多种 RoPE 扩展方案，发现**只在 Full Attention 层（而非 Sliding Window 层）应用 YaRN 效果最佳**
- 混合长短数据（1:2 比例）是为了防止纯长文档训练导致短文档任务性能退化

### 训练成本

| 模型 | H100 GPU 小时 | 估算美元成本 |
|------|-------------|-------------|
| OLMo 3 7B | ~220K | ~$500K |
| OLMo 3 32B | ~1M | ~$2.25M |

作为对比，Llama 3.1 的训练使用了 16K H100 GPU 训练数月，估算成本在 $10M 以上。OLMo 3 32B 用约 1/5 的成本，达到了接近的性能水平，**训练效率约为 Llama 3.1 的 2.5 倍**（更少 tokens，更好的数据质量）。

### 核心要点

- 三阶段策略将"构建知识"、"增强推理"、"扩展上下文"解耦为独立阶段
- 中期训练用仅 1.7% 的额外数据，显著提升数学/代码/推理能力
- 长上下文扩展使用 1:2 的长短数据混合比，平衡长短任务性能
- 32B 模型训练成本约 $2.25M，效率远超 Llama 3.1

## 五、架构设计

OLMo 3 使用的是 Dense Transformer 架构（非 MoE），在 Llama 风格的基础上做了多项针对性优化。

### 架构参数

| 参数 | 7B | 32B |
|------|-----|------|
| 层数 | 32 | 64 |
| 隐藏维度 | 4,096 | 5,120 |
| FFN 维度 | 11,008 | 13,824 |
| 注意力头数 (Q) | 32 | 40 |
| KV 头数 | 8 | 8 |
| GQA 比率 | 4:1 | 5:1 |
| 上下文长度 | 4,096 → 65K | 4,096 → 65K |
| 词表大小 | 100,278 | 100,278 |
| 参数总量 | ~7B | ~32B |

### 关键架构创新

#### 1. Sliding Window Attention（滑动窗口注意力）

这是 OLMo 3 最重要的架构选择之一。传统的 Full Attention 的计算复杂度为 O(n^2)，对长序列非常昂贵。OLMo 3 采用了混合注意力策略：

```
层 1:  Sliding Window (4K 窗口)
层 2:  Sliding Window (4K 窗口)
层 3:  Sliding Window (4K 窗口)
层 4:  Full Attention              ← 每 4 层一个全注意力层
层 5:  Sliding Window (4K 窗口)
层 6:  Sliding Window (4K 窗口)
层 7:  Sliding Window (4K 窗口)
层 8:  Full Attention              ← 保持全局信息流
...
最后一层: Full Attention           ← 始终使用全注意力
```

**设计逻辑**：
- 3/4 的层使用 Sliding Window Attention，窗口大小 4,096 tokens
- 每 4 层插入一个 Full Attention 层，保持全局信息流
- 最后一层始终使用 Full Attention，确保输出可以关注到全部上下文
- 这种 3:1 的比例在效率和性能之间取得了良好折中

**效率收益**：
- Sliding Window 层的注意力计算量从 O(n^2) 降为 O(n * w)，其中 w = 4096
- 对于 65K 上下文，理论加速约 4x（相比全部使用 Full Attention）
- KV Cache 在 Sliding Window 层只需保存最近 4K tokens 的 KV 状态

#### 2. Grouped Query Attention (GQA)

GQA 通过让多个 Query 头共享一组 Key-Value 头来减少 KV Cache 的内存占用：

| 模型 | Q 头数 | KV 头数 | 压缩比 |
|------|--------|---------|--------|
| OLMo 3 7B | 32 | 8 | 4:1 |
| OLMo 3 32B | 40 | 8 | 5:1 |

32B 模型的 KV Cache 相比标准 Multi-Head Attention 减少了 5 倍。

#### 3. QK-Norm

在注意力计算中，Query 和 Key 的点积可能产生非常大的数值，导致 softmax 后的注意力权重趋近于 one-hot（即几乎只关注一个 token），引发训练不稳定。

OLMo 3 在注意力计算前对 Q 和 K 向量进行 L2 归一化（QK-Norm），有效防止了注意力数值爆炸问题。

#### 4. Post-Normalization with RMSNorm

- 使用 RMSNorm（Root Mean Square Normalization）替代 LayerNorm
- 采用 Post-normalization 方案（在注意力/FFN 之后归一化）
- RMSNorm 比 LayerNorm 更高效（省去了均值计算和偏置项）

#### 5. RoPE + YaRN 位置编码

- 基础位置编码使用 RoPE（Rotary Position Embedding）
- 长上下文扩展使用 YaRN 缩放方法
- YaRN 只应用于 Full Attention 层（不应用于 Sliding Window 层）
- 从 8K 扩展到 65K 上下文

### 与其他架构的对比

| 特性 | OLMo 3 | Llama 3.1 | Qwen 2.5 | DeepSeek-V3 |
|------|--------|-----------|----------|-------------|
| 架构类型 | Dense | Dense | Dense | MoE |
| 注意力 | SWA + Full | Full | Full | MLA |
| KV 压缩 | GQA | GQA | GQA | MLA |
| 归一化 | RMSNorm (Post) | RMSNorm (Pre) | RMSNorm (Pre) | RMSNorm (Pre) |
| 位置编码 | RoPE + YaRN | RoPE | RoPE | RoPE |
| 上下文 | 65K | 128K | 128K | 128K |

### 核心要点

- Sliding Window Attention 3:1 比例是效率与性能的最佳折中
- GQA 5:1（32B 模型）大幅减少 KV Cache 内存占用
- QK-Norm 解决注意力数值爆炸的训练稳定性问题
- YaRN 只在 Full Attention 层应用是一个重要的实践经验

## 六、Tulu 3 后训练 ★★★

OLMo 3 的后训练是整个项目中技术密度最高的部分。后训练将一个只会补全文本的基座模型转变为能够遵循指令、进行推理、生成安全回答的实用模型。OLMo 3 的后训练框架名为 Tulu 3，包含 5 个阶段。

### 后训练全流程

```
OLMo 3 Base（预训练完成）
    │
    ▼
[阶段 1] Prompt 策划
    ├── 57% 来自公开数据集
    └── 43% 为合成生成
    │
    ▼
[阶段 2] SFT（有监督微调）
    ├── 数据：Dolci-Think-SFT ~2.3M 样本
    └── 目标：教会模型遵循指令、外化推理
    │
    ▼
[阶段 3] DPO（直接偏好优化）
    ├── 数据：Dolci-Think-DPO 偏好对
    └── 目标：从好坏回答差异中学习
    │
    ▼
[阶段 4] RLVR（可验证奖励 RL）★★★
    ├── 框架：OlmoRL（基于 GRPO）
    ├── 奖励：数学求解器 + 代码单元测试
    └── 目标：在可验证任务上持续提升
    │
    ▼
[阶段 5] 标准化评估
    └── OLMES 框架全面评测
    │
    ▼
OLMo 3 Instruct / Think（最终模型）
```

### 阶段 1：Prompt 策划

Prompt 策划是后训练的基础环节，决定了模型将学会处理哪些类型的任务。

| 来源 | 占比 | 说明 |
|------|------|------|
| 公开数据集 | 57% | 已有的高质量指令数据集（如 FLAN、Open Assistant 等） |
| 合成生成 | 43% | 使用强模型（如 GPT-4、Claude）生成高质量 prompt |

合成 prompt 的优势在于可以针对模型弱项定向补充。例如，如果评估发现模型在多步数学推理上较弱，可以专门合成这类 prompt。

### 阶段 2：SFT（Supervised Fine-Tuning）

SFT 阶段使用 **Dolci-Think-SFT** 数据集，包含约 **2.3M 个训练样本**。

**数据覆盖范围**：

| 任务类别 | 说明 | 示例 |
|---------|------|------|
| 数学推理 | 多步数学问题及详细解答过程 | 竞赛数学、应用题 |
| 科学推理 | 物理、化学、生物等科学问题 | 实验设计、因果分析 |
| 编程 | 代码生成、调试、解释 | LeetCode 风格 |
| 对话 | 多轮对话、角色扮演 | 开放式聊天 |
| 指令遵循 | 格式控制、约束满足 | "用三段话回答" |
| 安全 | 拒绝有害请求、安全引导 | 对抗性 prompt 处理 |

**Think 模式的特殊设计**：

对于 Think 模型变体，SFT 数据中包含显式的推理链（thinking tokens）。训练格式为：

```
<think>
[详细的推理过程，可能包含多步推导、假设验证、自我纠正]
</think>

[最终答案]
```

这种格式教会模型将内部推理过程外化为显式文本。

### 阶段 3：DPO（Direct Preference Optimization）

DPO 使用 **Dolci-Think-DPO** 偏好数据集，训练模型区分高质量和低质量的回答。

**数据构建方法**：

- **On-policy 数据**：用当前模型生成同一 prompt 的多个回答，选择最好和最差的配对
- **Off-policy 数据**：用不同规模的模型（如 Qwen 不同尺寸）生成回答，大模型的回答为"chosen"，小模型的回答为"rejected"

**一个关键发现**：

DPO 对推理模型出人意料地有效。在传统认知中，DPO 主要用于提升回答的风格和安全性，对推理能力的帮助有限。但 OLMo 3 团队发现，当 DPO 的偏好对来自不同推理水平的模型时，DPO 能有效提升推理链的质量和最终答案的准确率。

### 阶段 4：RLVR（Reinforcement Learning with Verifiable Rewards）★★★

RLVR 是 OLMo 3 后训练中最具技术深度的部分，也是 Think 模型能力跃升的关键。

#### 核心思想

传统 RLHF（Reinforcement Learning from Human Feedback）依赖人类标注的偏好数据和一个学习得到的奖励模型（Reward Model）。这种方法存在两个问题：
1. 奖励模型可能学到错误的偏好（reward hacking）
2. 人类标注昂贵且有主观性

RLVR 的解决方案是：**用可验证的客观信号替代主观的人类偏好**。

| 任务类型 | 验证方式 | 奖励信号 |
|---------|---------|---------|
| 数学题 | 符号化数学求解器（symbolic solver） | 答案正确 → 奖励 1；错误 → 奖励 0 |
| 编程题 | 代码单元测试（unit tests） | 全部通过 → 奖励 1；失败 → 奖励 0 |
| 对话/开放任务 | LLM Judge（用强模型评判） | 评分归一化为 [0, 1] |

#### OlmoRL 框架

OlmoRL 是 AI2 基于 GRPO（Group Relative Policy Optimization）开发的 RL 训练框架，包含 **6 大技术改进**：

| 改进 | 技术名称 | 解决的问题 | 效果 |
|------|---------|-----------|------|
| 1 | 截断重要性采样 (TIS) | 防止旧样本的梯度信号主导训练 | 训练更稳定 |
| 2 | 零梯度信号过滤 | 过滤全对或全错的 prompt（无学习信号） | 训练效率提升 |
| 3 | 主动采样 | 优先采样模型当前最需要提升的题目 | 学习速度加快 |
| 4 | Token 级损失 | 在 token 级而非序列级计算损失 | 更精细的梯度信号 |
| 5 | Clip-higher 修改 | 修改 PPO clip 范围，允许更大的正向更新 | 正确行为获得更多强化 |
| 6 | Group-Advantaged 过滤 | 过滤组内优势值方差过小的样本 | 减少噪声梯度 |

#### RLVR 训练细节

- 32B Think 模型在 RLVR 阶段使用 **224 个 H100 GPU**
- 初始训练后又进行了**额外 21 天**的持续 RL 训练
- 在 Dolci-Think-RL 数据集上多轮迭代
- 这种长时间的 RL 训练显著提升了困难数学任务（如 AIME）上的表现

#### RLVR 的效果

以 GSM8K（小学数学推理基准）为例：

| 阶段 | GSM8K 分数 | 提升 |
|------|-----------|------|
| SFT 后 | 基线 | - |
| DPO 后 | +1.2 | 主要改善回答格式 |
| RLVR 后 | +3.3 | 推理准确率显著提升 |

在更难的数学基准上（如 MATH、AIME），RLVR 的提升更加明显。

### 阶段 5：标准化评估

使用 OLMES 评估框架对最终模型进行全面评测，确保各维度能力达标。评估覆盖通用知识（MMLU）、指令遵循（IFEval）、推理（GPQA）、数学（MATH/GSM8K）、编程（HumanEval/MBPP）等多个维度。

### 核心要点

- 后训练分 5 个阶段，从 prompt 策划到 RLVR，逐步提升能力
- SFT 使用 2.3M 样本，覆盖数学/科学/编程/对话/安全六大类
- DPO 对推理模型的效果超出预期，关键是偏好对要来自不同推理水平
- RLVR 用可验证奖励替代主观偏好，在数学上带来 +3.3 分提升
- OlmoRL 的 6 项技术改进使 RL 训练更稳定、更高效

## 八、OLMES 评估框架

OLMES（Open Language Model Evaluation System）是 AI2 为 OLMo 系列开发的标准化评估框架，旨在解决当前 LLM 评估中的可复现性和可比较性问题。

### 为什么需要 OLMES？

当前 LLM 评估存在严重的可比较性问题：

| 问题 | 说明 |
|------|------|
| Prompt 模板不同 | 不同模型使用不同的 prompt 格式，结果不可比 |
| Few-shot 设置不一 | 有的用 0-shot，有的用 5-shot |
| 后处理差异 | 答案提取方式不同导致分数差异 |
| 数据泄露 | 训练数据可能包含评估基准内容 |
| 选择性报告 | 只报告最好的基准结果 |

### OLMES 的设计

**OlmoBaseEval 基准集**：

OLMES 设计了一套标准化的评估任务集：

| 类别 | 包含基准 | 评估维度 |
|------|---------|---------|
| 通用知识 | MMLU, ARC, HellaSwag | 世界知识、常识推理 |
| 推理 | GPQA, BBH | 科学推理、逻辑推理 |
| 数学 | GSM8K, MATH | 数学计算、数学推理 |
| 编程 | HumanEval, MBPP | 代码生成、代码理解 |
| 指令遵循 | IFEval | 格式控制、约束满足 |
| 安全 | TruthfulQA, ToxiGen | 真实性、无害性 |

**开发任务集 vs 持留任务集**：

| 类型 | 用途 | 可见性 |
|------|------|--------|
| 开发任务集 (Dev Set) | 用于训练过程中的调参和消融 | 训练团队可见 |
| 持留任务集 (Held-Out Set) | 仅用于最终评测，不参与任何决策 | 训练期间不使用 |

这种分离设计防止了对评估基准的过拟合。

### 分层聚类降噪

单个基准的分数波动可能很大（尤其是样本量小的基准），OLMES 使用分层聚类（hierarchical clustering）将相关基准分组，计算组合指标来降低噪声。

### 去污染方法论

为确保评估结果的公正性，OLMES 实施了严格的去污染流程：

1. **n-gram 匹配**：检查训练数据中是否包含评估基准的 n-gram
2. **语义相似度**：使用 embedding 检测语义上接近但文字不完全相同的重叠
3. **公开报告**：去污染结果完全公开，任何人都可以审计

### 核心要点

- OLMES 解决了 LLM 评估的可复现性和可比较性问题
- 开发/持留任务集分离防止对基准过拟合
- 分层聚类降噪提高了评估结果的稳定性
- 完全开放的去污染流程确保结果公正

## 九、核心洞察 ★★

阅读 OLMo 3 报告后，可以提炼出以下对理解大模型训练具有普遍意义的洞察。

### 洞察 1：数据质量 > 数据数量

这是 OLMo 3 最重要的结论之一。

| 模型 | 训练 Tokens | 基座性能（平均） |
|------|------------|----------------|
| OLMo 3 32B | ~6T | 与 Llama 3.1 接近 |
| Llama 3.1 70B | ~15T | 当然更强（参数量也更大） |
| Qwen 2.5 32B | ~18T | 略强 |

OLMo 3 用 6T 高质量数据达到了其他模型用 15-18T 普通数据的接近水平。这意味着：

- 精心筛选的 6T tokens 约等于随机采集的 15T tokens
- 数据处理管线的工程投入可以用 2-3 倍的效率回报
- Swarm 优化找到的最优混合比例进一步放大了数据质量的优势

### 洞察 2：DPO 对推理模型出人意料地有效

传统认知中，DPO 主要用于改善回答风格和安全性，对推理能力帮助有限。但 OLMo 3 发现：

- 当偏好对来自不同推理水平的模型时，DPO 能有效提升推理链质量
- DPO + RLVR 的组合比单独使用 RLVR 效果更好
- DPO 似乎为 RLVR 提供了更好的初始化状态

### 洞察 3：单独 SFT 或 RL 在中等规模不够，需要结合

| 训练方式 | 效果 |
|---------|------|
| 仅 SFT | 学会格式和基本遵循，但推理能力有限 |
| 仅 RL（从 Base 出发） | 如 R1-Zero 实验所示，格式混乱但推理有涌现 |
| SFT + DPO + RLVR | 格式整洁 + 推理能力强 + 安全对齐 |

在 7B-32B 的中等规模上，单独任何一种方法都不足以产出高质量的推理模型。OLMo 3 的五阶段后训练管线展示了各方法的互补性。

### 洞察 4：Sliding Window Attention 是效率/性能的好折中

OLMo 3 的 3:1 SWA 策略（3 层 Sliding Window + 1 层 Full Attention）在以下方面取得了良好平衡：

| 维度 | 全 Full Attention | 3:1 SWA | 全 Sliding Window |
|------|------------------|---------|-------------------|
| 计算效率 | 最低 | 较高 | 最高 |
| 长距离信息流 | 最好 | 良好 | 受限 |
| KV Cache | 最大 | 较小 | 最小 |
| 实际性能 | 基准 | 接近基准 | 明显下降 |

这种设计也被 Gemma 2、Step 3.5 Flash 等后续模型采用（比例略有不同），验证了其有效性。

### 洞察 5：中期训练是能力涌现的关键阶段

中期训练（Midtraining）的思路可以概括为：

> 在模型已经建立了广泛的语言基础之后、进入后训练之前，通过短暂但高强度的定向数据训练，大幅提升特定能力。

这比在预训练阶段简单增加某类数据的比例更有效，因为：
- 模型已经有了足够的语言理解基础，可以更高效地学习新知识
- 不会干扰广度训练阶段的知识积累
- 成本极低（100B tokens vs 5.9T tokens）

### 洞察 6：中间检查点对研究训练动态极有价值

OLMo 3 公开的全过程检查点使得以下研究成为可能：

- **能力涌现时间点**：特定能力（如多步推理）是在训练的什么阶段涌现的？
- **遗忘与干扰**：中期训练中增加数学数据是否导致了其他知识的遗忘？
- **Scaling 行为**：损失下降曲线在不同阶段呈现什么模式？
- **数据影响归因**：某类数据对最终性能的贡献可以通过对比有无该数据的检查点来量化

这类研究在只有最终权重的情况下**完全无法进行**。

### 核心要点

- 数据质量的 2-3 倍效率优势是可以量化的
- DPO + RLVR 的组合对推理模型至关重要，不应跳过 DPO
- 中期训练以极低成本（1.7% 额外数据）带来显著能力提升
- SWA 3:1 是一个被多个项目验证的有效架构选择
- 全过程检查点对研究社区的价值无法被仅公开权重替代

## 十、核心贡献总结

### 贡献 1：完全开源标杆

OLMo 3 重新定义了 "开源 AI" 的含义。在 OLMo 3 之前，"开源"在 LLM 领域几乎等同于"公开权重"。OLMo 3 证明了可以在数据、代码、检查点、评估全部公开的情况下，训练出具有竞争力的模型。这为未来的 AI 开源设立了新的标准。

### 贡献 2：Dolma 3 数据工程

Dolma 3 不仅是一个数据集，更是一套完整的数据工程方法论：

| 组件 | 贡献 |
|------|------|
| Dolma Toolkit | 开源的万亿级数据处理框架 |
| OlmoOCR | 学术 PDF 到结构化文本的高质量转换 |
| Duplodocus | 万亿规模的 Bloom Filter 去重框架 |
| 质量感知上采样 | 系统化的数据质量利用方法 |

### 贡献 3：Swarm 优化

用 30 个 30M 参数代理模型搜索最优数据混合比例的方法具有广泛适用性。任何需要决定数据混合比例的预训练项目都可以借鉴这一方法，以极低成本获得数据混合的最优解。

### 贡献 4：RLVR 后训练

OlmoRL 框架的 6 项技术改进（TIS、零梯度过滤、主动采样、Token 级损失、Clip-higher、Group-Advantaged 过滤）为基于 GRPO 的 RL 训练提供了工程实践经验。特别是 RLVR 用可验证奖励替代奖励模型的思路，显著降低了推理模型训练的门槛。

### 贡献 5：成本效率

| 维度 | OLMo 3 32B | 对比 |
|------|------------|------|
| 训练成本 | ~$2.25M | Llama 3.1 估算 >$10M |
| 训练数据 | ~6T tokens | Llama 3.1 ~15T tokens |
| 训练效率 | ~2.5x | 相对 Llama 3.1 |
| 性能 | 接近同尺寸竞品 | 在 MATH/Code 上领先 |

OLMo 3 证明了在预算有限的条件下，通过数据质量、训练策略、后训练技巧的组合，可以达到远超预算级别的性能。

### 一句话总结

> OLMo 3 用完全透明的流程、更少的数据、更低的成本，训练出了具有前沿竞争力的模型，为 AI 开源社区提供了从数据到模型的完整蓝图。
