## 一、为什么读这篇论文

### 1.1 震撼行业的性价比

DeepSeek-V3 用 **$5.57M**（约 4000 万人民币）训练了一个 671B 参数的 MoE 模型，在几乎所有主流基准上达到或超越 GPT-4o 水平。作为对比，Llama 3.1 405B 消耗了约 30.8M H100 GPU 小时（估算成本超过 $60M），GPT-4 的训练成本传闻超过 $100M。DeepSeek-V3 以约 1/10 的成本达到了同等性能，这在 2024 年底的 AI 行业引发了巨大震动。

### 1.2 三大架构创新定义 2025 技术范式

DeepSeek-V3 不是简单的"把模型做大"，而是在三个关键维度上实现了工程创新：

| 创新 | 核心贡献 | 影响 |
|------|---------|------|
| **无辅助损失 MoE** | 解决了 MoE 负载均衡与模型质量的冲突 | 后续几乎所有 MoE 模型采用类似方案 |
| **MLA (Multi-head Latent Attention)** | KV Cache 压缩 93%+，推理内存大幅降低 | 成为高效推理的标准方案 |
| **FP8 混合精度训练** | 首次在 671B 模型上验证 FP8 训练 | 证明低精度训练的可行性和经济性 |

### 1.3 广泛的后续影响

DeepSeek-V3 的技术被后续几乎所有重要开源模型借鉴或采用：

- **DeepSeek-R1**：直接使用 V3-Base 作为基座，叠加 RL 训练推理能力
- **Kimi K2.5 / K2**：采用 MoE + MLA 架构范式
- **GLM-5**：借鉴 DeepSeek 稀疏注意力机制
- **Qwen3**：采用类似的 MoE 路由策略
- **Step 3.5 Flash**：参考 FP8 训练和 MoE 设计

可以说，理解 DeepSeek-V3 就是理解 2025 年大模型训练的技术基础。

## 二、模型规模总览

### 2.1 完整参数配置

| 参数 | 数值 | 说明 |
|------|------|------|
| **总参数量** | 671B (6710 亿) | 包括所有专家的参数 |
| **每 Token 激活参数** | 37B (370 亿) | 仅约 5.5% 参数被激活 |
| **Transformer 层数** | 61 | 前 3 层使用 Dense MLP，后 58 层使用 MoE |
| **隐藏维度** | 7168 | 模型内部表示的宽度 |
| **注意力头数 (n_h)** | 128 | Multi-head Attention 的头数 |
| **每头维度 (d_h)** | 128 | 每个注意力头的维度 |
| **KV 压缩维度 (d_c)** | 512 | MLA 的压缩潜在空间维度 |
| **解耦 Query/Key 维度 (d_R)** | 64 | RoPE 单独处理的维度 |
| **路由专家数 (N_r)** | 256 | 可路由的稀疏专家 |
| **共享专家数 (N_s)** | 1 | 所有 token 都经过的专家 |
| **每 Token 激活专家数 (K_r)** | 8 | Top-8 路由选择 |
| **专家中间维度** | 2048 | 每个路由专家的 FFN 中间层维度 |
| **共享专家中间维度** | 2048 | 共享专家的 FFN 中间层维度 |
| **词表大小** | 128K | Byte-level BPE tokenizer |
| **最大上下文长度** | 128K tokens | 通过 YaRN 两阶段扩展 |

### 2.2 参数分布

```
DeepSeek-V3 参数分布 (671B 总参数)

┌──────────────────────────────────────────────────┐
│  Embedding + Output Head     ≈ 1.8B  (0.3%)     │
│  Attention (MLA) × 61 层    ≈ 26.8B  (4.0%)     │
│  共享专家 × 58 层           ≈ 1.7B   (0.3%)     │
│  路由专家 256×58 层         ≈ 640B   (95.4%)  ◄── 绝大部分参数在 MoE 专家中
│  其他 (Norm, Gate, etc.)    ≈ 0.7B   (0.1%)     │
└──────────────────────────────────────────────────┘
Total = 671B

每 Token 实际激活：
  Attention (26.8B) + 共享专家 (1.7B) + 8 个路由专家 (8 × ~1.1B ≈ 8.8B)
  ≈ 37B 参数
```

### 2.3 关键数字

- **激活比例**：37B / 671B = 5.5%，即每个 token 只"使用"不到 6% 的模型参数
- **专家激活比例**：每 token 激活 8/256 = 3.1% 的路由专家 + 1 个共享专家
- **Dense 等效规模**：如果将同等计算量用于 Dense 模型，相当于约 37B 参数的模型
- **相比 V2 的扩展**：总参数 671B vs 236B (2.8x)，激活参数 37B vs 21B (1.8x)

## 三、MoE 架构深度解析 ★★★

> 这是 DeepSeek-V3 最核心的架构设计。MoE（Mixture of Experts）允许模型拥有巨大的参数量，同时每次推理只激活一小部分，实现了"大参数、小计算"的平衡。

### 3.1 MoE 基础概念

对于没有接触过 MoE 的读者，先建立基本直觉：

```
传统 Dense 模型（如 Llama）：
  输入 → [同一个 FFN] → 输出
  每个 token 都经过完全相同的参数

MoE 模型（如 DeepSeek-V3）：
  输入 → [路由器选择] → [8 个专家并行处理] → [加权合并] → 输出
  不同 token 根据内容被路由到不同的专家组合

好处：
  - 模型容量大（671B 参数存储知识）
  - 计算成本低（每次只用 37B 参数计算）
  - 不同专家可以"专精"不同类型的知识
```

### 3.2 专家配置

DeepSeek-V3 的 MoE 层配置如下：

| 配置项 | 数值 | 说明 |
|--------|------|------|
| 路由专家总数 | 256 | 可被路由器选择的稀疏专家 |
| 共享专家数 | 1 | 所有 token 始终经过 |
| 每 token 激活路由专家 | 8 | Top-8 路由选择 |
| 每 token 实际使用专家 | 9 | 8 路由 + 1 共享 |
| MoE 层数 | 58 | 第 4-61 层 (前 3 层是 Dense) |
| Dense 层数 | 3 | 第 1-3 层 |

**为什么前 3 层使用 Dense？** 模型的前几层主要处理 token embedding 的初步转换，这种低级特征处理不需要专家特化，使用 Dense 层更稳定。

### 3.3 路由机制详解

#### 从 Softmax 到 Sigmoid

DeepSeek-V2 使用 Softmax 路由，即所有专家的亲和力分数互相竞争、归一化求和为 1。DeepSeek-V3 做了一个关键改变——使用 **Per-expert Sigmoid** 门控：

```
DeepSeek-V2 (Softmax 路由)：
  affinity_i = Softmax(x · W_g)_i          ← 所有专家分数之和 = 1
  问题：一个专家分数上升，其他专家分数必然下降（零和博弈）

DeepSeek-V3 (Sigmoid 路由)：
  affinity_i = Sigmoid(x · w_i)            ← 每个专家独立计算，范围 [0, 1]
  然后选 Top-K 个专家
  gate_i = affinity_i / Σ(selected affinity)  ← 在被选中的专家间归一化

好处：
  - 多个专家可以同时获得高亲和力（不再是零和博弈）
  - 允许更灵活的专家组合
  - 更好的梯度流动
```

#### 路由流程图

```
输入 token 表示 x (7168 维)
        │
        ▼
┌─────────────────────────┐
│ 256 个独立 Sigmoid 门控 │
│ s_i = σ(x · w_i + b_i) │  ← 每个专家有独立的门控向量 w_i 和偏置 b_i
└─────────────────────────┘
        │
        ▼
┌─────────────────────────┐
│  选择 Top-8 亲和力最高  │
│  的路由专家              │
└─────────────────────────┘
        │
        ▼
┌─────────────────────────────────────┐
│ 门控值归一化（仅用原始亲和力，不含偏置）│
│ g_i = σ(x · w_i) / Σ_top8 σ(x·w_j) │
└─────────────────────────────────────┘
        │
        ▼
┌──────────────────────────────────────┐
│ output = 共享专家(x) + Σ g_i · 路由专家_i(x) │
└──────────────────────────────────────┘
```

### 3.4 无辅助损失负载均衡 (Auxiliary-Loss-Free Load Balancing) ★★★

> 这是 DeepSeek-V3 最重要的工程创新之一，解决了困扰 MoE 研究多年的核心问题。

#### 问题背景

MoE 模型面临一个经典难题——**路由崩溃 (routing collapse)**：

```
路由崩溃示例：
  假设有 256 个专家，但训练中路由器"偏爱"其中少数几个
  → 被频繁选择的专家学到更好的表示
  → 更好的表示吸引更多路由
  → 恶性循环：大部分专家闲置，少数专家过载
  → 模型的有效容量大大降低
```

传统解决方案是添加**辅助损失 (auxiliary loss)**：

```
传统方法 (GShard, Switch Transformer, Mixtral 等)：

  total_loss = main_loss + α × auxiliary_balance_loss

  auxiliary_balance_loss 惩罚不均匀的专家利用率

  问题：
  1. α 太大 → 强制均匀分配，损害模型性能（某些 token 被送到"错误"的专家）
  2. α 太小 → 均衡效果不足，仍然路由崩溃
  3. α 的最优值因任务和训练阶段而变化，难以调优
  4. 根本矛盾：辅助损失与主任务损失方向不一致
```

#### DeepSeek-V3 的解决方案：动态偏置项

DeepSeek-V3 提出了一种优雅的替代方案——**不在损失函数中添加辅助项**，而是通过**动态调节路由偏置**来实现负载均衡：

```
核心机制：

  对每个专家 i，维护一个偏置项 b_i（初始值为 0）

  路由决策时（决定选哪个专家）：
    routing_score_i = σ(x · w_i) + b_i    ← 偏置影响选择
    选 Top-K 个 routing_score 最高的专家

  门控计算时（决定专家贡献多大权重）：
    gate_i = σ(x · w_i)                    ← 用原始亲和力，不含偏置！
    gate_i = gate_i / Σ_topK gate_j        ← 归一化

  关键设计："选哪个"和"权重多大"被解耦！
  偏置只影响选择，不影响加权 → 不会扭曲梯度
```

#### 偏置动态调整规则

```
每个训练步结束后，统计每个专家被选中的频率：
  f_i = 专家 i 在当前 batch 中被选中的次数

  如果 f_i > 平均频率 f_avg：
    b_i ← b_i - γ    ← 过载专家：降低偏置，未来少被选中

  如果 f_i < 平均频率 f_avg：
    b_i ← b_i + γ    ← 欠载专家：升高偏置，未来多被选中

  γ 是调节速度（缩放因子），控制偏置更新的幅度
```

#### 训练配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 偏置更新速度 γ | 0.001 | 前 14.3T tokens |
| 偏置更新速度 γ | 0 | 最后 500B tokens（冻结偏置） |
| 序列级平衡损失 | 是 | 辅以轻量的序列内平衡约束 |
| 补充平衡系数 α | 1e-5 | 极小的序列级辅助损失 |

**注意**：虽然名为"无辅助损失"，但实际上仍保留了一个系数极小 (α = 1e-5) 的序列级平衡损失，用于防止单个序列内的极端不均匀。关键区别是这个系数比传统方法小 2-3 个数量级，对模型质量的影响可忽略不计。

#### 效果对比

| 方法 | 负载均衡 | 模型质量 | 调参难度 |
|------|---------|---------|---------|
| 无负载均衡 | 极差（路由崩溃） | 差 | 无 |
| 辅助损失 (α=0.01) | 好 | 中（被牺牲） | 高（需调 α） |
| 辅助损失 (α=0.001) | 中 | 较好 | 高 |
| **DeepSeek-V3 动态偏置** | **好** | **最好** | **低（γ 不敏感）** |

#### 为什么这很重要

```
传统方法的根本矛盾：
  辅助损失说："把 token 均匀分配"
  主任务损失说："把 token 送到最合适的专家"
  → 两个目标方向冲突

DeepSeek-V3 的解决：
  偏置只影响"谁被选中"（离散决策）
  不影响"权重多大"（梯度流）
  → 主任务损失完全不受干扰
  → 专家可以更深度地特化，同时保持整体均衡
```

### 3.5 通信高效路由

在大规模分布式训练中，MoE 的一个关键瓶颈是 **All-to-All 通信**——每个 token 需要被发送到其路由专家所在的 GPU 节点。DeepSeek-V3 引入了**节点级路由约束**：

```
硬件布局：
  2048 个 H800 GPU，分布在多个节点上
  每个节点 8 个 GPU，通过 NVLink 高速互联
  节点间通过 InfiniBand 连接（带宽较低）

路由约束：
  每个 token 最多发送到 M = 4 个节点
  在这 M 个节点中选择 Top-8 专家

  好处：
  - 限制跨节点通信量
  - 保证 NVLink 内的高带宽通信为主
  - 跨节点通信量减少约 4x
```

### 3.6 共享专家的作用

```
共享专家 vs 路由专家的分工：

┌─────────────────────────────────────────────────────────┐
│                                                         │
│  共享专家（1 个，所有 token 必经）：                       │
│  ├── 通用语言理解（语法、语义）                            │
│  ├── 基础常识推理                                        │
│  └── "锚点"作用：提供稳定的基础表示                       │
│                                                         │
│  路由专家（256 个，每 token 选 8 个）：                    │
│  ├── 领域专业知识（医学、法律、编程等）                     │
│  ├── 语言特定处理（中文、英文、代码等）                     │
│  └── 长尾知识：低频但专业的信息                            │
│                                                         │
└─────────────────────────────────────────────────────────┘

类比：
  共享专家 = 通识课老师（所有学生都上）
  路由专家 = 专业课老师（按需选修）
```

### 3.7 核心要点

1. **256 路由 + 1 共享**：每 token 激活 9 个专家（8 路由 + 1 共享），仅用 5.5% 参数
2. **Sigmoid 独立门控**：解除专家间的分数竞争，允许更灵活的专家组合
3. **无辅助损失负载均衡**：通过动态偏置实现均衡，不牺牲模型质量
4. **节点级路由约束**：每 token 最多跨 4 个节点，大幅减少通信开销

## 四、MLA 机制详解 (Multi-head Latent Attention) ★★★

> MLA 是 DeepSeek-V2 首创、V3 沿用的注意力机制，通过将 KV 缓存压缩到低维潜在空间，实现了 93%+ 的内存压缩，是 DeepSeek 系列的另一核心创新。

### 4.1 问题：KV Cache 的内存瓶颈

在标准的 Multi-Head Attention (MHA) 中，推理时需要缓存所有历史 token 的 Key 和 Value 向量：

```
标准 MHA 的 KV Cache 大小：

  KV Cache = n_layers × n_heads × 2 × d_head × seq_len × dtype_size

  对于 DeepSeek-V3 如果使用标准 MHA：
  = 61 层 × 128 头 × 2(K+V) × 128 维 × 128K tokens × 2 bytes (FP16)
  = 61 × 128 × 2 × 128 × 131072 × 2
  ≈ 520 GB（单个请求！）

  问题：
  - 一个 128K 上下文的请求就需要 ~520 GB KV Cache
  - 无法在单个 GPU（80GB）上服务
  - 严重限制并发请求数
  - 成为推理成本的主要瓶颈
```

### 4.2 MLA 核心思想：联合 KV 压缩

MLA 的关键洞察是：**Key 和 Value 之间存在大量冗余信息，可以联合压缩到一个低维潜在向量中。**

```
MLA 的压缩过程：

输入 X (7168 维)
    │
    │  下投影矩阵 W^{DKV} (7168 → 512)
    ▼
 c^{KV} (512 维)  ◄── 这就是推理时唯一需要缓存的向量！
    │
    ├── W^{UK} (512 → 16384)  → 恢复 Key  (128 头 × 128 维)
    │
    └── W^{UV} (512 → 16384)  → 恢复 Value (128 头 × 128 维)

数学表达：
  c^{KV} = W^{DKV} · X          (压缩：7168 → 512)
  K = W^{UK} · c^{KV}           (解压：512 → 16384)
  V = W^{UV} · c^{KV}           (解压：512 → 16384)

关键：训练时 K 和 V 都从同一个 512 维向量 c^{KV} 恢复
     推理时只需缓存 c^{KV}，不需要存完整的 K 和 V
```

### 4.3 压缩效果量化

| 指标 | 标准 MHA | GQA (8 组) | MQA | **MLA** |
|------|---------|-----------|-----|---------|
| 每 Token KV 存储维度 | 2 × 128 × 128 = 32,768 | 2 × 8 × 128 = 2,048 | 2 × 128 = 256 | **512** |
| 128K 上下文 KV Cache | ~520 GB | ~32.5 GB | ~4.1 GB | **~8.1 GB** |
| 压缩比 (vs MHA) | 1x | 16x | 128x | **64x** |
| 模型质量 | 最好 | 轻微下降 | 明显下降 | **与 MHA 持平** |

**MLA 的独特优势**：在压缩比和模型质量之间取得了最佳平衡。GQA 和 MQA 通过减少注意力头数来减少 KV Cache，但不可避免地损失了多头注意力的表达能力。MLA 通过学习压缩变换保留了完整的多头注意力能力。

### 4.4 数学原理详解

#### 完整 MLA 计算流程

```
设：
  d_model = 7168     (模型隐藏维度)
  n_h = 128          (注意力头数)
  d_h = 128          (每头维度)
  d_c = 512          (KV 压缩维度)
  d_R = 64           (RoPE 解耦维度)

Step 1: Query 生成
  Q^{content} = W^{DQ}(X) → c^Q (1536 维) → W^{UQ}(c^Q)  (各头的内容 Query)
  Q^{rope}    = W^{QR}(c^Q)                                (位置 Query，d_R=64 维)

Step 2: Key-Value 压缩
  c^{KV} = W^{DKV}(X)      (7168 → 512，下投影)
  K^{content} = W^{UK}(c^{KV})  (512 → 16384，上投影恢复 Key)
  V = W^{UV}(c^{KV})            (512 → 16384，上投影恢复 Value)
  K^{rope} = W^{KR}(X)          (单独的位置 Key，d_R=64 维)

Step 3: 注意力计算
  Attention Score = Q^{content} · K^{content}ᵀ + Q^{rope} · K^{rope}ᵀ
  Output = Softmax(Score / √d_h) · V

缓存需求：
  推理时只需缓存 c^{KV} (512 维) + K^{rope} (64 维) = 576 维/token/层
  相比 MHA 的 32768 维/token/层，压缩了约 57 倍
```

#### 压缩维度选择

| 压缩维度 d_c | 压缩比 | 模型质量 | 选择 |
|-------------|--------|---------|------|
| 128 | 256x | 明显下降 | 不采用 |
| 256 | 128x | 轻微下降 | 不采用 |
| **512** | **64x** | **与 MHA 持平** | **采用** |
| 1024 | 32x | 略优于 MHA | 性价比不高 |

DeepSeek 选择 d_c = 512 是经过消融实验确定的最优点——在几乎不损失模型质量的前提下实现了最大压缩。

### 4.5 解耦 RoPE 处理

标准 Transformer 使用 RoPE (Rotary Position Embedding) 为 Key 和 Query 注入位置信息。但 RoPE 与 MLA 的压缩存在冲突：

```
问题：
  RoPE 需要作用在完整维度的 K 和 Q 上
  如果先压缩再加 RoPE → 位置信息与压缩后的表示混淆
  如果先加 RoPE 再压缩 → 压缩会破坏位置编码

DeepSeek-V3 的解决方案：解耦 RoPE

  将 Query 和 Key 各分成两部分：
  1. 内容部分 (content)：参与 KV 压缩，不加 RoPE
  2. 位置部分 (rope)：单独处理，加 RoPE，不参与压缩

  注意力分数 = 内容分数 + 位置分数

  推理缓存：c^{KV} (512 维, 内容) + K^{rope} (64 维, 位置) = 576 维
```

### 4.6 权重吸收技巧 (Weight Absorption)

这是 MLA 的另一个精妙设计——在推理时避免显式还原完整的 K 和 V：

```
朴素实现：
  c^{KV} → W^{UK} → K_full (16384 维) → Q · K_fullᵀ
  需要计算和存储完整的 K_full

权重吸收：
  Q · K_fullᵀ = Q · (W^{UK} · c^{KV})ᵀ
              = Q · c^{KV}ᵀ · W^{UK}ᵀ
              = (Q · W^{UK}ᵀ)ᵀ · c^{KV}ᵀ      (将 W^{UK} "吸收"到 Q 的变换中)

  实际操作：
  Q_absorbed = Q · W^{UK}ᵀ   (离线预计算，或融合到 Q 的投影中)
  Score = Q_absorbed · c^{KV}ᵀ  (直接用压缩向量计算，不需要还原 K_full)

  好处：
  - 不需要显式还原 16384 维的 K
  - 计算量从 O(n_h × d_h × seq_len) 降到 O(d_c × seq_len)
  - 进一步节省计算和内存
```

### 4.7 与其他注意力机制的综合对比

| 特性 | MHA | GQA | MQA | **MLA** |
|------|-----|-----|-----|---------|
| 注意力头数 | 全部独立 | 分组共享 KV | 全部共享 KV | 全部独立 |
| KV 缓存维度/层 | 2 × n_h × d_h | 2 × n_g × d_h | 2 × d_h | d_c + d_R |
| 以 DeepSeek-V3 配置计 | 32,768 | 2,048 (8组) | 256 | 576 |
| 模型质量 | 基线 | 轻微下降 | 明显下降 | **与基线持平** |
| 压缩比 | 1x | 16x | 128x | **57x** |
| 需要特殊训练 | 否 | 否 | 否 | 是 |
| 推理吞吐 | 基线 | 2-4x | 8-16x | **~20x** |

### 4.8 核心要点

1. **57x 压缩**：128K 上下文的 KV Cache 从 ~520 GB 降至 ~8.1 GB
2. **质量无损**：通过学习压缩变换，保留了完整多头注意力的表达能力
3. **解耦 RoPE**：巧妙解决了位置编码与压缩的冲突
4. **权重吸收**：推理时不需要显式还原 K/V，进一步加速
5. **~20x 推理加速**：综合 KV Cache 减少和计算优化

## 五、FP8 混合精度训练 ★★

> DeepSeek-V3 首次在超大规模模型 (671B) 上验证了 FP8 训练的可行性和有效性，将约 80-85% 的模型计算从 BF16 降到 FP8。

### 5.1 什么是 FP8

```
数值精度对比：

FP32 (32 位浮点)：1 位符号 + 8 位指数 + 23 位尾数
  范围：±3.4×10^38，精度：~7 位十进制
  ↑ 传统训练默认精度

BF16 (16 位浮点)：1 位符号 + 8 位指数 + 7 位尾数
  范围：±3.4×10^38，精度：~3 位十进制
  ↑ 当前主流训练精度（保留 FP32 的范围）

FP8 E4M3 (8 位浮点)：1 位符号 + 4 位指数 + 3 位尾数
  范围：±240，精度：~2 位十进制
  ↑ DeepSeek-V3 的创新——用于大部分计算

FP8 E5M2 (8 位浮点)：1 位符号 + 5 位指数 + 2 位尾数
  范围：±57344，精度：~1.5 位十进制
  ↑ 用于需要更大范围的场景

从 BF16 → FP8：
  计算吞吐翻倍（NVIDIA H800 的 FP8 Tensor Core 性能 = 2x BF16）
  内存占用减半
  通信带宽需求减半
```

### 5.2 量化范围

DeepSeek-V3 将约 **80-85% 的模型计算**使用 FP8 执行，包括：

| 组件 | 精度 | 说明 |
|------|------|------|
| 线性层 GEMM（前向） | **FP8** | 专家 FFN、注意力投影 |
| 线性层 GEMM（反向-权重梯度） | **FP8** | 大部分梯度计算 |
| 线性层 GEMM（反向-激活梯度） | **FP8** | 激活梯度计算 |

### 5.3 保持高精度的部分

以下组件始终使用 BF16 或 FP32，因为它们对数值精度敏感：

| 组件 | 精度 | 原因 |
|------|------|------|
| Embedding 层 | BF16 | 词嵌入精度影响所有下游计算 |
| 输出头 (LM Head) | BF16 | 直接影响 token 概率分布 |
| MoE 门控网络 | BF16 | 路由决策精度影响负载均衡 |
| Normalization (RMSNorm) | FP32 | 数值稳定性要求高 |
| 注意力 Softmax | FP32 | 指数运算对精度敏感 |
| 主权重 (Master Weights) | FP32 | 权重更新的累积精度 |
| 优化器状态 (Adam) | FP32 | 一阶和二阶动量需高精度 |

### 5.4 细粒度量化策略

FP8 只有 8 位精度，直接对整个张量统一量化会导致严重的精度损失（因为离群值会"占用"量化范围）。DeepSeek-V3 使用了**细粒度分组量化**：

```
细粒度量化示意：

激活矩阵 (tile-wise, 1×128)：
┌─────────────────────────────────────────────┐
│ [a1, a2, ..., a128] → scale_1, 独立量化     │  ← 每 128 个连续元素
│ [a129, a130, ..., a256] → scale_2, 独立量化  │     有自己的缩放因子
│ ...                                         │
└─────────────────────────────────────────────┘

权重矩阵 (block-wise, 128×128)：
┌──────────┬──────────┬──────────┐
│ 128×128  │ 128×128  │ 128×128  │  ← 每个 128×128 的块
│ block_1  │ block_2  │ block_3  │     有独立的缩放因子
│ scale_1  │ scale_2  │ scale_3  │
├──────────┼──────────┼──────────┤
│ block_4  │ block_5  │ block_6  │
│ scale_4  │ scale_5  │ scale_6  │
└──────────┴──────────┴──────────┘

好处：
- 离群值只影响其所在的小块，不会"污染"其他块
- 每个小块的有效精度更高
- 额外的缩放因子存储开销很小（<0.1% 参数量）
```

| 量化目标 | 分组方式 | 分组大小 | 缩放因子存储开销 |
|---------|---------|---------|----------------|
| 激活 (Activations) | tile-wise | 1×128 | 每 128 元素一个 FP32 |
| 权重 (Weights) | block-wise | 128×128 | 每 16384 元素一个 FP32 |

### 5.5 数值稳定性保障

为了确保 FP8 训练的数值稳定性，DeepSeek-V3 采取了多项措施：

```
WGMMA 累加策略：

  NVIDIA H800 的 WGMMA (Warp Group Matrix Multiply Accumulate) 指令
  在 FP8 模式下，内部累加器使用 FP16 → 可能溢出

  DeepSeek-V3 方案：
  每 4 次 WGMMA 操作后，将部分结果累加到 FP32 寄存器
  → 防止 FP16 累加器溢出
  → 额外计算开销约 2-3%

完整精度保障链：
  1. 细粒度量化：小块独立缩放
  2. 定期 FP32 累加：防止中间结果溢出
  3. 主权重 FP32：确保权重更新的长期精度
  4. 优化器 FP32：Adam 动量的数值稳定性
  5. 在线统计：动态计算缩放因子（非延迟统计）
```

### 5.6 精度验证

DeepSeek 进行了严格的精度验证实验：

| 验证指标 | 结果 |
|---------|------|
| 训练损失相对误差 | **始终低于 0.25%** |
| 下游任务性能差异 | 在训练随机性范围内 |
| 与 BF16 基线对比 | 无统计显著差异 |
| 训练稳定性 | 未出现 loss spike 或发散 |

### 5.7 FP8 训练的经济收益

```
FP8 vs BF16 训练的性能提升：

             BF16        FP8         提升
TFLOPS:     ~500        ~1000       2x 计算吞吐
显存:        基线        ~50%        2x 容量
通信:        基线        ~50%        2x 带宽效率

综合训练速度提升约 40-60%（受限于非 FP8 组件和通信）
```

### 5.8 核心要点

1. **80-85% 计算使用 FP8**：大幅降低计算成本
2. **细粒度量化**：激活 1×128、权重 128×128 独立缩放
3. **关键组件保持高精度**：Embedding、输出头、门控、Norm、Attention Softmax
4. **验证结果**：相对误差始终低于 0.25%，与 BF16 训练无统计显著差异
5. **首次大规模验证**：证明 FP8 可用于 671B 级别模型训练

## 六、多 Token 预测 (Multi-Token Prediction, MTP) ★★

> DeepSeek-V3 在标准的 Next-Token Prediction 基础上，增加了预测后续多个 token 的训练目标，既提升了训练效率，又为推理时的推测解码提供了天然支持。

### 6.1 传统 NTP vs MTP

```
标准 Next-Token Prediction (NTP)：
  输入:  [The, cat, sat, on]
  目标:  [cat, sat, on, the]     ← 每个位置只预测下一个 token

Multi-Token Prediction (MTP, D=2)：
  输入:  [The, cat, sat, on]
  目标1: [cat, sat, on, the]     ← 预测下一个 token (主目标)
  目标2: [sat, on, the, mat]     ← 预测第 2 个 token (辅助目标)

  D = 预测深度，DeepSeek-V3 使用 D = 1（即额外预测 1 个 token）
```

### 6.2 MTP 模块架构

DeepSeek-V3 的 MTP 实现与 Meta 的方案有重要区别：

```
Meta 的 MTP (Better & Faster LLMs, 2024)：
  主模型 → D 个独立输出头 → 并行预测 D 个 token
  各预测之间独立，缺少因果依赖

DeepSeek-V3 的 MTP：
  顺序预测，在每个预测深度保持因果链

  具体架构（D=1 时）：

  ┌─────────────────────────────┐
  │     主 Transformer (61 层)   │
  │     输出: hidden states h   │
  └──────────┬──────────────────┘
             │
             ▼
  ┌─────────────────────────────────┐
  │  MTP Module (预测第 2 个 token)   │
  │                                  │
  │  1. 将 h 与下一个 token 的        │
  │     embedding 拼接                │
  │  2. 投影矩阵 M_1 降维            │
  │  3. 一个 Transformer 块 T_1      │
  │  4. 共享输出头 Out(.) 预测       │
  └─────────────────────────────────┘

  关键设计：
  - 共享 embedding 层 E(.)：不额外增加参数
  - 共享输出头 Out(.)：不额外增加参数
  - 仅 T_1 和 M_1 是新增参数（很小）
  - 拼接操作提供了因果链上下文
```

### 6.3 MTP 的训练收益

| 收益 | 说明 |
|------|------|
| **更丰富的梯度信号** | 每个位置不仅要预测下一个 token，还要为后续预测提供有用的表示 |
| **更好的规划能力** | 模型学会"提前规划"，使隐藏状态编码更远的未来信息 |
| **更高效的数据利用** | 同样的数据提供了约 2x 的训练信号 |
| **训练损失** | MTP 损失作为辅助目标，权重 λ = 0.3 |

MTP 损失定义：
```
总损失 = L_main + λ × L_MTP
       = L_NTP + 0.3 × L_{depth=1}

其中：
  L_main = 标准 Next-Token Prediction 损失
  L_MTP  = 第 2 个 token 预测的交叉熵损失
  λ = 0.3（通过消融实验确定）
```

### 6.4 推理时的灵活性

MTP 模块在推理时有两种使用方式：

```
方式 1：直接丢弃 MTP 模块（零开销推理）

  训练完成后，MTP 模块可以完全移除
  主模型独立运行，与标准 NTP 模型完全相同
  → 不增加任何推理开销
  → 但仍享受训练阶段 MTP 带来的质量提升

方式 2：复用为推测解码的 draft model

  ┌──────────────────────────────────────────┐
  │  推测解码 (Speculative Decoding) 流程：    │
  │                                           │
  │  1. Draft (MTP 模块) 快速生成 D 个候选      │
  │  2. Verify (主模型) 一次前向验证所有候选     │
  │  3. 接受正确的前缀，拒绝错误的              │
  │                                           │
  │  MTP 模块天然适合做 draft model：           │
  │  - 与主模型分布匹配度高（共享训练）          │
  │  - 参数很少，速度快                         │
  │  - 不需要额外训练 draft model              │
  └──────────────────────────────────────────┘
```

### 6.5 推测解码性能

| 指标 | 数值 |
|------|------|
| 第 2 token 接受率 | **85-90%** |
| 推理吞吐加速 | **约 1.8x** |
| 额外参数量 | < 1B（与 671B 相比可忽略） |
| 额外延迟 | 几乎为零（与验证并行） |

### 6.6 核心要点

1. **D=1**：额外预测 1 个 token，辅助损失权重 λ=0.3
2. **顺序因果**：与 Meta 并行 MTP 不同，保持因果链
3. **共享参数**：embedding 和输出头共享，新增参数极少
4. **双模式推理**：训练后可丢弃（零开销）或复用为推测解码（1.8x 加速）
5. **接受率 >85%**：MTP 模块作为 draft model 的预测准确率很高

## 七、DualPipe 流水线并行 ★★

> DualPipe 是 DeepSeek-V3 的训练基础设施创新，通过双向流水线调度实现了计算与通信的深度重叠，将 GPU 利用率推至 85%+。

### 7.1 问题：通信瓶颈

在大规模 MoE 训练中，跨节点的 All-to-All 通信是主要瓶颈：

```
MoE 的通信模式：

  每个 MoE 层需要两次 All-to-All 通信：
  1. Dispatch（分发）：将 token 发送到对应专家所在的 GPU
  2. Combine（合并）：将专家处理结果收集回来

  通信量：
  每个 token 的路由专家可能分布在不同节点上
  → 大量跨节点数据传输

  DeepSeek-V3 的情况：
  - 2048 个 H800 GPU
  - 256 个节点，每节点 8 个 GPU
  - 跨节点通信与计算的比例约 1:1（对半分）

  如果顺序执行：一半时间在计算，一半时间在通信
  → GPU 利用率只有约 50%
```

### 7.2 DualPipe 设计

DualPipe 的核心思想是**将前向和反向传播拆分为多个可重叠的组件**：

```
Transformer 块的 4 个组件：

  前向传播：
  ┌───────────┐    ┌──────────────┐    ┌───────────┐    ┌──────────────┐
  │ Attention  │ →  │ All-to-All   │ →  │    MLP     │ →  │ All-to-All   │
  │  (计算)    │    │  Dispatch    │    │  (计算)    │    │  Combine     │
  │            │    │  (通信)      │    │            │    │  (通信)      │
  └───────────┘    └──────────────┘    └───────────┘    └──────────────┘
       ATTN              DISP              MLP              COMB

  反向传播同理，也拆为 4 个对应组件

DualPipe 的重叠策略：

  时间 →
  ┌──────┬──────┬──────┬──────┬──────┬──────┐
  │ ATTN │ DISP │ MLP  │ COMB │ ATTN │ ...  │  流 A（正向）
  │      │ MLP  │ COMB │ ATTN │ DISP │ ...  │  流 B（反向）
  └──────┴──────┴──────┴──────┴──────┴──────┘

  关键：一个流中的计算操作与另一个流中的通信操作在时间上重叠
  - 流 A 做 Attention 计算时，流 B 在做 All-to-All 通信
  - 流 A 做 All-to-All 通信时，流 B 在做 MLP 计算
  → 计算和通信几乎完全重叠！
```

### 7.3 双向流水线调度

DualPipe 的"Dual"指的是**双向流水线**：

```
传统流水线（单向）：

  GPU 0: [F1][F2][F3][F4][ idle ][B4][B3][B2][B1]
  GPU 1:     [F1][F2][F3][F4][ idle ][B4][B3][B2][B1]
  GPU 2:         [F1][F2][F3][F4][ idle ][B4][B3][B2][B1]
  ...
  问题：大量 "bubble"（空闲时间）

DualPipe（双向）：

  GPU 0: [F→1][F→2][B←4][F→3][B←3][F→4][B←2][B←1]
  GPU 1: [F→1][B←4][F→2][B←3][F→3][B←2][F→4][B←1]
  ...

  从流水线两端同时注入微批次：
  - 正向流 (Forward →): 从 GPU 0 → GPU N
  - 反向流 (Backward ←): 从 GPU N → GPU 0

  结果：
  - Pipeline bubble 大幅减少
  - GPU 空闲时间最小化
```

### 7.4 硬件配置与并行策略

| 配置项 | 值 | 说明 |
|--------|-----|------|
| GPU 总数 | 2048 | NVIDIA H800 80GB |
| 节点数 | 256 | 每节点 8 GPU |
| 节点内互联 | NVLink | ~900 GB/s 双向带宽 |
| 节点间互联 | InfiniBand | ~400 Gbps |
| 流水线并行 (PP) | 16 路 | 61 层分配到 16 个 PP stage |
| 专家并行 (EP) | 64 路 | 256 个专家分布到 64 个 EP group |
| 数据并行 (DP) | ZeRO-1 | 优化器状态分片 |
| 总并行度 | 16 × 64 × 2 = 2048 | PP × EP × DP |

### 7.5 GPU 利用率

| 项目 | DeepSeek-V3 | 行业平均 |
|------|------------|---------|
| GPU 计算利用率 | **85%+** | 55-65% |
| 通信重叠率 | **>90%** | 30-50% |
| Pipeline bubble | **<5%** | 15-30% |
| MFU (Model FLOPS Utilization) | **~45%** | 30-40% |

### 7.6 核心要点

1. **4 组件拆分**：Attention/Dispatch/MLP/Combine，计算与通信交错执行
2. **双向流水线**：从两端注入微批次，大幅减少 pipeline bubble
3. **>90% 通信重叠**：通信几乎完全被计算"隐藏"
4. **85%+ GPU 利用率**：远超行业平均 55-65%
5. **2048 H800 GPU**：16-way PP × 64-way EP × ZeRO-1 DP

### 8.3 Fill-in-Middle (FIM) 策略

DeepSeek-V3 在预训练中使用了 Fill-in-Middle 策略来增强代码补全能力：

| FIM 配置 | 值 |
|----------|-----|
| 框架 | PSM (Prefix-Suffix-Middle) |
| FIM 比例 | 0.1（10% 的训练样本使用 FIM 格式） |
| 应用范围 | 主要用于代码数据 |

```
PSM 格式示例：

原始代码：
  def add(a, b):
      return a + b

FIM 变换后：
  [PREFIX] def add(a, b): [SUFFIX] [MIDDLE] return a + b

训练目标：给定前缀和后缀，预测中间部分
→ 增强模型的代码补全和填充能力
```

### 8.4 文档打包策略

```
文档打包 (Document Packing)：

  将多个短文档拼接成一个训练序列，提高 GPU 利用率

  ┌─────────┬─────────┬─────────┬───────┐
  │ Doc A   │ Doc B   │ Doc C   │ Pad   │  ← 一个训练序列 (4K tokens)
  └─────────┴─────────┴─────────┴───────┘

  关键设计：不使用跨文档注意力掩码

  传统做法：使用 attention mask 阻止不同文档间的注意力
  DeepSeek-V3：不使用跨样本注意力掩码
  → 实现更简单，且实验表明对模型质量影响可忽略
```

### 8.5 上下文长度扩展

```
两阶段 YaRN 扩展：

阶段 1: 4K → 32K
  - 训练步数：1000 步
  - 序列长度：32K tokens
  - 学习率：7.3e-6（主训练的 1/10）

阶段 2: 32K → 128K
  - 训练步数：1000 步
  - 序列长度：128K tokens
  - 学习率：7.3e-6

总共仅需 2000 步即可将上下文从 4K 扩展到 128K
→ 成本极低，约占总训练的 0.02%
```

### 8.6 核心要点

1. **14.8T tokens**：相比 V2 (8.1T) 增加 83%

## 九、后训练

### 9.1 SFT 阶段

| 配置项 | 值 |
|--------|-----|
| SFT 样本总量 | **1.5M** (150 万条) |
| 推理任务样本来源 | DeepSeek-R1 生成 |
| 通用任务样本来源 | DeepSeek-V2.5 生成 + 人工验证 |
| 训练轮次 | 2 epochs |
| 学习率 | 5e-6 |

#### 推理数据的特殊处理

```
推理数据生成流程：

1. 使用 DeepSeek-R1 对推理任务（数学、代码、科学）生成回答
2. R1 生成的回答包含详细的思维链 (Chain-of-Thought)
3. 筛选正确的回答作为 SFT 数据

过度思考问题：
  R1 生成的推理链往往过长（数千 token）
  对于简单问题也会进行深度推理
  → 浪费计算资源
  → 用户体验差（等待时间长）

解决方案：
  - 对简单任务使用 V2.5 生成的简洁回答
  - 只对困难推理任务使用 R1 生成的详细推理链
  - 人工审核和过滤过度冗长的回答
```

#### 数据分布

| 任务类型 | 数据来源 | 占比（估算） |
|---------|---------|------------|
| 数学推理 | R1 生成 | ~15% |
| 代码推理 | R1 生成 | ~15% |
| 科学推理 | R1 生成 | ~5% |

| 指令遵循 | 人工标注 | ~10% |

### 9.2 RL 阶段

DeepSeek-V3 的 RL 阶段借鉴了 DeepSeek-R1 的经验：

| 配置项 | 值 |
|--------|-----|
| RL 算法 | GRPO (Group Relative Policy Optimization) |
| 奖励来源 | 规则奖励 + 奖励模型 |
| 推理任务奖励 | 规则验证（数学答案正确性、代码执行通过率） |
| 通用任务奖励 | 奖励模型打分 |

```
RL 训练流程：

  SFT 模型
      │
      ▼
  ┌─────────────────────────────────────────┐
  │  GRPO 训练                               │
  │                                          │
  │  推理任务（数学、代码）：                   │
  │  → 规则奖励（答案正确 = +1，错误 = -1）    │
  │  → 不需要奖励模型                         │
  │                                          │
  │  通用任务（对话、写作）：                   │
  │  → 奖励模型评分                           │
  │  → 评估有用性和安全性                      │
  └─────────────────────────────────────────┘
      │
      ▼
  最终 DeepSeek-V3 Chat 模型
```

### 9.3 从 R1 蒸馏推理能力

这是 DeepSeek-V3 后训练的一个关键创新——利用 R1 的推理数据来增强 V3 的推理能力：

```
蒸馏路径：

  DeepSeek-R1 (强推理模型)
      │
      │ 生成 ~800K 推理样本
      ▼
  DeepSeek-V3-Base
      │
      │ SFT on R1 生成的推理数据
      ▼
  DeepSeek-V3 (增强推理能力)

关键发现：
  - 小模型（<70B）只需 SFT 就能获得显著的推理提升
  - 大模型（V3 级别）需要 SFT + RL 才能充分发挥潜力
  - 蒸馏比从头 RL 训练更高效
```

### 9.4 核心要点

1. **1.5M SFT 样本**：推理用 R1 生成，通用用 V2.5 + 人工
2. **R1 蒸馏**：约 800K 推理样本从 R1 蒸馏到 V3
3. **过度思考问题**：R1 生成的推理链过长，需要筛选和控制
4. **GRPO 算法**：推理任务用规则奖励，通用任务用奖励模型
5. **发现**：大模型需要 SFT + RL，小模型仅 SFT 即可

### 10.2 成本拆解：五大降本因素

#### 因素 1：MoE 架构——只激活 5.6% 参数

```
Dense 模型 vs MoE 模型：

  假设目标：达到"671B 参数容量"的模型质量

  Dense 路线：
  - 训练 671B Dense 模型
  - 每 token 计算量 ≈ 2 × 671B = 1.342T FLOPS
  - 14.8T tokens 总计 ≈ 1.99 × 10^25 FLOPS

  MoE 路线 (DeepSeek-V3)：
  - 训练 671B MoE 模型，每 token 激活 37B 参数
  - 每 token 计算量 ≈ 2 × 37B = 74B FLOPS（仅约 Dense 的 5.5%）
  - 但实际总计算量高于上式（考虑路由、通信等开销）
  - 实际计算量 ≈ Dense 等效 37B 模型水平

  节省：约 18x 计算量
  代价：更大的内存占用（存储 671B 参数），更多的通信

  关键洞察：DeepSeek-V3 证明 MoE 的"有效质量"远高于激活参数量
  37B 激活的 MoE 质量 >> 37B Dense 模型，接近 ~200B+ Dense 模型
```

#### 因素 2：FP8 训练——计算效率翻倍

```
BF16 训练 vs FP8 训练：

  H800 GPU 理论峰值：
  - BF16 Tensor Core: ~990 TFLOPS
  - FP8 Tensor Core:  ~1979 TFLOPS (2x)

  实际训练中：
  - 80-85% 计算使用 FP8
  - 综合加速约 40-60%（受限于非 FP8 组件和通信）

  等效效果：
  如果全部用 BF16 训练，需要约 1.5x 的 GPU 小时
  即 2,788,000 × 1.5 ≈ 4,182,000 GPU 小时
  节省约 1,394,000 GPU 小时 ≈ $2.79M
```

#### 因素 3：DualPipe + 通信重叠——85% GPU 利用率

```
GPU 利用率对成本的影响：

  行业平均 GPU 利用率：55-65%
  DeepSeek-V3 GPU 利用率：85%+

  假设：同样的计算量，利用率从 60% 提升到 85%
  需要的 wall-clock 时间减少约 30%
  → 节省约 30% 的 GPU 租赁成本

  关键因素：
  - DualPipe 双向流水线减少 pipeline bubble
  - 计算与通信深度重叠（>90% 通信被隐藏）
  - 精心设计的并行策略 (PP×EP×DP)
```

#### 因素 4：数据效率

```
更好的数据效率意味着更少的训练 tokens：

  DeepSeek-V3 (14.8T tokens) vs Llama 3.1 (15T tokens)
  虽然 token 数相近，但 DeepSeek-V3 的每 token 信息量更高：

  1. 更好的 tokenizer：128K 词表 vs 128K 词表
     → 中文每 token 覆盖更多字符
  2. 更高质量的数据筛选
  3. MTP 训练目标提供额外训练信号
     → 等效约 1.3-1.5x 的数据利用效率
```

#### 因素 5：MLA 内存优化

```
MLA 对训练的贡献：

  更小的 KV Cache → 更大的微批次 → 更高的 GPU 利用率

  标准 MHA：KV Cache 占大量 GPU 内存 → 限制批次大小
  MLA：KV Cache 压缩 57x → 释放内存 → 增大批次
  → 间接提升训练吞吐量
```

## 附录 A：关键公式速查

### MLA 压缩

```
压缩: c^{KV} = W^{DKV} · X        (W^{DKV} ∈ R^{d_c × d_model}, d_c=512, d_model=7168)
还原: K = W^{UK} · c^{KV}          (W^{UK} ∈ R^{(n_h·d_h) × d_c})
      V = W^{UV} · c^{KV}          (W^{UV} ∈ R^{(n_h·d_h) × d_c})
```

### MoE 路由

```
亲和力: s_i = σ(x · w_i)           (Sigmoid, 独立于其他专家)
选择:   T = TopK({s_i + b_i}, K=8)  (偏置影响选择)
门控:   g_i = s_i / Σ_{j∈T} s_j    (用原始亲和力归一化)
输出:   y = FFN_shared(x) + Σ_{i∈T} g_i · FFN_i(x)
```

### MTP 损失

```
总损失: L = L_NTP + λ · L_MTP       (λ = 0.3)
L_NTP = -Σ log P(t_{i+1} | t_1...t_i)          (标准下一 token 预测)
L_MTP = -Σ log P(t_{i+2} | t_1...t_i, MTP_1)   (第 2 token 预测)
```

### FP8 量化

```
量化:   x_q = clamp(round(x / scale), -448, 448)  (E4M3 范围)
反量化: x_dq = x_q × scale
scale = max(|x_block|) / 448                       (每块独立缩放)
```
