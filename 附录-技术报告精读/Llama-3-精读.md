## 二、模型家族概览

Llama 3 发布了三个规模的密集（Dense）Transformer 模型，核心设计理念是：**在标准 Dense 架构上把规模和数据量推到极致，而非追求 MoE 等架构创新**。

### 模型参数表

| 参数 | 8B | 70B | 405B |
|------|-----|------|------|
| Transformer 层数 | 32 | 80 | 126 |
| 隐藏维度 (d_model) | 4,096 | 8,192 | 16,384 |
| 注意力头数 (n_heads) | 32 | 64 | 128 |
| KV 头数 (GQA) | 8 | 8 | 8 |
| FFN 中间维度 | 14,336 | 28,672 | 53,248 |
| 词表大小 | 128,256 | 128,256 | 128,256 |
| 最大上下文长度 | 128K | 128K | 128K |
| 位置编码 | RoPE (base=500,000) | RoPE (base=500,000) | RoPE (base=500,000) |
| 激活函数 | SwiGLU | SwiGLU | SwiGLU |
| 归一化 | RMSNorm | RMSNorm | RMSNorm |

### 设计哲学

Meta 的核心信念：**"旗舰模型应该是 Dense Transformer，以最大化训练稳定性，且不需要定制推理内核。"** 这与 DeepSeek 走 MoE 路线的策略形成了鲜明对比——Meta 选择通过充足的数据量（15.6T tokens）来释放标准 Dense 架构的全部潜力。

## 三、预训练数据工程 ★★★

> 这是整篇论文中工程细节最丰富、对实际工作参考价值最高的章节。

### 3.1 数据规模与组成

Llama 3 的预训练数据量达到 **15.6T tokens**，相比 Llama 2 的 2T tokens 增长了约 **7 倍**。

| 数据类别 | 占比 | 说明 |
|---------|------|------|
| 英文网页文本 | ~50% | 主要来源，经过多层过滤 |
| 代码数据 | ~17% | 比 Llama 2 增加约 4 倍 |
| 多语言数据 | ~8% | 覆盖 30+ 种语言 |
| 数学与推理数据 | ~5% | 数学论坛、证明、习题 |
| 其他高质量文本 | ~20% | 书籍、学术论文、百科等 |

**语言分布**：
- **英文**：约 95% 的主要训练数据
- **非英文**：5%+ 覆盖超过 30 种语言（德语、法语、意大利语、葡萄牙语、印地语、西班牙语、泰语等）
- 多语言数据的加入使 Llama 3 在非英文基准上有显著提升

### 3.2 多阶段过滤流水线

Llama 3 的数据清洗是一个包含多个阶段的精密工程流水线。以下按执行顺序详细展开：

```
原始网页数据 (数十万亿 tokens)
  │
  ├─ 阶段 1: URL 级去重
  │   └── 基于完整 URL 的精确去重，去除重复抓取
  │
  ├─ 阶段 2: 文档级去重
  │   ├── 全局 MinHash 去重 (跨所有数据源)
  │   ├── 使用 5-gram + 84 个哈希函数
  │   └── 去除约 38% 的近似重复文档
  │
  ├─ 阶段 3: 行级去重
  │   ├── 对全语料中的每一行计算哈希
  │   ├── 出现次数 > 阈值的行被移除
  │   └── 有效去除模板文本、导航栏、Cookie 声明等
  │
  ├─ 阶段 4: 脏词 (dirty word) 过滤
  │   └── 基于关键词黑名单过滤成人/有害内容
  │
  ├─ 阶段 5: KL 散度域过滤
  │   ├── 计算每个域的 token 分布与高质量参考集的 KL 散度
  │   └── KL 散度过大的整个域被移除
  │
  ├─ 阶段 6: 质量分类器过滤 ★
  │   ├── FastText 快速分类器 (第一道筛选)
  │   ├── DistilRoBERTa 精细分类器 (第二道筛选)
  │   └── 详见下文 "质量分类器" 小节
  │
  └─ 最终输出: 15.6T 高质量 tokens
```

### 3.3 质量分类器 ★★

质量分类器是整个数据流水线中最关键的组件，Llama 3 采用了**两级分类**策略：

**第一级：FastText 快速分类器**
- 模型：基于 n-gram 的 FastText 分类器
- 训练数据：正样本来自维基百科、书籍等高质量来源；负样本来自随机网页
- 速度：极快，可以在合理时间内处理数万亿 tokens
- 作用：快速过滤掉明显的低质量数据，粗筛

**第二级：DistilRoBERTa 精细分类器**
- 模型：DistilRoBERTa（约 82M 参数）
- 训练数据标注：使用 **Llama 2** 模型作为标注器
  - 给 Llama 2 展示网页文本片段
  - 让 Llama 2 判断 "这段文本是否可能出现在维基百科/高质量文献中？"
  - 将 Llama 2 的判断作为伪标签训练 DistilRoBERTa
- 作用：精细过滤，保留高质量数据

**为什么用 Llama 2 标注而非人工标注？**
- 需要标注的数据量级达到数十亿条，人工不可行
- Llama 2 的质量判断与人类判断高度相关
- 成本可忽略（Llama 2 推理成本 vs 人工标注成本）
- 这是一个"用上一代模型帮助训练下一代模型"的典型范式

### 3.4 代码数据处理

| 处理步骤 | 详情 |
|---------|------|
| 来源 | GitHub 公开仓库 |
| 语言覆盖 | Python, C++, Java, JavaScript, TypeScript, Go, Rust 等 |
| 去重 | 文件级去重 + 基于 AST 的语义去重 |
| 质量过滤 | 基于仓库星标数、代码复杂度、注释比例等 |
| 数据量 | 约占总数据的 17%，是 Llama 2 代码数据的 **4 倍** |

### 3.5 数学数据增强

数学能力是 Llama 3 重点提升的方向。主要数据来源包括：
- 数学论坛（如 Art of Problem Solving、StackExchange Math）
- LaTeX 格式的数学论文和教材
- 合成数学问题（用模型生成的带解答的数学习题）
- 代码化数学推理数据（将数学问题转化为可执行代码验证的形式）

### 3.6 对比 Llama 2 的数据改进

| 维度 | Llama 2 | Llama 3 | 提升倍数 |
|------|---------|---------|---------|
| 总数据量 | 2T tokens | 15.6T tokens | **7.8x** |
| 代码数据量 | ~3% | ~17% | **约 4x** |
| 多语言覆盖 | 有限 | 30+ 语言 | 显著扩展 |
| 去重方法 | 基础去重 | 三级去重（URL + 文档 + 行级） | 大幅增强 |
| 质量分类 | 单一分类器 | 双级分类（FastText + DistilRoBERTa） | 更精细 |
| 知识截止日期 | 2022 年底 | 2024 年初 | 更新 |

### 3.7 数据混合与退火 (Annealing)

训练末期使用了**数据退火 (data annealing)** 策略：
- 在预训练的最后阶段，将学习率线性衰减至 0
- 同时调整数据混合比例，**增大高质量数据**（如代码、数学、精选网页）的占比
- 具体操作：在最后 40M tokens 中，高质量数据的上采样倍率显著提高
- 效果：在不增加总训练量的前提下，显著提升基准性能（尤其是 MATH 和 HumanEval）

### 核心要点

1. **数据量是王道**：15.6T tokens，比 Llama 2 多 7 倍以上，是模型性能提升的最大贡献因素
2. **两级质量分类器**：FastText 粗筛 + DistilRoBERTa 精筛，用上一代模型（Llama 2）标注训练数据
3. **三级去重**：URL → 文档 → 行级，去除约 38% 的冗余数据
4. **代码和数学数据大幅加码**：代码量 4 倍增长，数学数据专项增强
5. **数据退火**：训练末期增大高质量数据占比，是低成本提升性能的有效手段

## 四、模型架构

Llama 3 的架构是一个**相对保守但高度优化**的标准 Transformer。Meta 在这里的哲学很明确——不追求架构创新，而是在久经验证的架构上做精细调优。

### 4.1 GQA 分组查询注意力

**Grouped-Query Attention (GQA)** 是 Llama 3 最重要的架构选择之一，平衡了推理效率和模型质量。

```
标准 Multi-Head Attention (MHA)：
  Query heads:  128 个 (以 405B 为例)
  Key heads:    128 个
  Value heads:  128 个
  → KV Cache 巨大

Multi-Query Attention (MQA)：
  Query heads:  128 个
  Key heads:    1 个  (所有 Query 共享)
  Value heads:  1 个  (所有 Query 共享)
  → KV Cache 最小，但质量下降

Grouped-Query Attention (GQA) — Llama 3 的选择：
  Query heads:  128 个
  Key heads:    8 个  (每 16 个 Query 共享一组 KV)
  Value heads:  8 个
  → KV Cache 缩减到 MHA 的 1/16，质量几乎无损
```

**GQA 配置**（所有三个规模统一使用 8 个 KV 头）：

| 模型 | Query 头数 | KV 头数 | 共享比 | KV Cache 缩减 |
|------|-----------|---------|--------|-------------|
| 8B | 32 | 8 | 4:1 | 4x |
| 70B | 64 | 8 | 8:1 | 8x |
| 405B | 128 | 8 | 16:1 | 16x |

### 4.2 RoPE 旋转位置编码

Llama 3 使用 **Rotary Position Embedding (RoPE)**，但将 base frequency 从 Llama 2 的 10,000 提升至 **500,000**。

```
RoPE 公式：
  θ_i = base^(-2i/d)

Llama 2: base = 10,000   → 适合 4K 上下文
Llama 3: base = 500,000  → 适合 8K 初始上下文，可扩展至 128K

base 频率越大，位置编码的旋转频率越低，对远距离位置差异更敏感
→ 更容易扩展到更长的上下文
```

**上下文长度扩展路线**：
1. 使用 base=500,000 预训练到 8K 上下文
2. 在后续阶段通过渐进式训练扩展到 128K
3. 扩展过程分为 6 个阶段，序列长度逐步增加

### 4.3 SwiGLU 激活函数

FFN 层使用 SwiGLU：`FFN(x) = SiLU(xW_gate) * (xW_up) · W_down`。引入门控机制，需要三个权重矩阵（gate、up、down），FFN 中间维度约为隐藏维度的 3.5 倍。在同 FLOP 下性能优于 ReLU/GELU。

### 4.4 128K BPE 词表

词表设计是 Llama 3 架构中一个容易被忽视但非常重要的改进。

| 属性 | Llama 2 | Llama 3 |
|------|---------|---------|
| 词表大小 | 32,000 | **128,256** |
| 编码算法 | SentencePiece BPE | **tiktoken BPE** |
| 编码效率提升 | 基线 | **+15%**（同等文本需要更少 tokens） |
| 多语言支持 | 有限 | 显著改善 |

128K 词表使得常见中文词语可以用 1-2 个 token 表示（而非 3-5 个），编码效率提升 15%。词表基于 tiktoken（OpenAI 的 BPE 实现），比 SentencePiece 更高效。代价是 embedding 层参数量增加约 4 倍，但相对于总参数量占比很小。

### 4.5 文档注意力掩码

标准做法将多个文档打包到同一个训练序列中，所有 tokens 互相可见，模型可能学到跨文档的虚假关联。Llama 3 使用**注意力掩码**确保跨文档 tokens 不可见——每个文档内部正常注意力，跨文档无注意力。报告指出这对性能有"小但一致"的正面影响。

### 核心要点

1. **保守但有效的架构**：不追求 MoE、MLA 等创新，而是在标准 Dense Transformer 上精细调优
2. **GQA 统一 8 KV 头**：在三个规模上统一配置，405B 的 KV Cache 缩减 16 倍
3. **RoPE base=500,000**：为长上下文扩展做预留，从 8K 可扩展至 128K
4. **128K 词表**：编码效率提升 15%，显著改善多语言支持
5. **文档注意力掩码**：消除多文档打包带来的跨文档信息泄露

## 五、训练基础设施与并行策略 ★★★

> 这是目前公开文献中对万卡级 GPU 集群训练方案描述最详细的章节。

### 5.1 硬件配置

| 项目 | 配置 |
|------|------|
| GPU 型号 | NVIDIA H100 80GB SXM5 |
| GPU 总数 | **16,384 块** |
| 每节点 GPU 数 | 8 |
| 节点总数 | 2,048 |
| 节点内互联 | NVSwitch + NVLink (900 GB/s 双向) |
| 节点间互联 | RoCE (RDMA over Converged Ethernet) |
| 网络拓扑 | 三层 CLOS 网络 |
| 存储 | 分布式并行文件系统 (Tectonic) + 240 PB 存储 |

### 5.2 4D 并行策略 ★★★

Llama 3 405B 的训练采用了 **4D 并行**，即同时使用 4 个维度的并行策略。这是整篇论文中对工程实践贡献最大的内容之一。

```
4D 并行 = TP × PP × DP(FSDP) × CP

总 GPU 数 = TP_size × PP_size × DP_size × CP_size
16,384    = 8       × 16      × ???     × ???
```

#### 张量并行 (Tensor Parallelism, TP)

| 属性 | 详情 |
|------|------|
| TP 大小 | 8 (一个节点内的 8 块 GPU) |
| 通信模式 | AllReduce |
| 通信带宽需求 | 极高，使用 NVLink 900 GB/s |
| 作用 | 将每一层的矩阵运算拆分到 8 块 GPU 上 |

**为什么 TP=8**：
- TP 通信需要极低延迟和极高带宽
- NVLink 只存在于同一节点内的 8 块 GPU 之间
- 跨节点做 TP 会因网络延迟导致严重性能损失

#### 流水线并行 (Pipeline Parallelism, PP)

| 属性 | 详情 |
|------|------|
| PP 大小 | 16 |
| PP 调度 | 交错式 (Interleaved Schedule) |
| 微批次数 | 通过微批次填充流水线气泡 |
| 作用 | 将 126 层分配到 16 个流水线阶段 |

**Llama 3 的流水线设计细节**：
- 126 层分为 16 个阶段，每阶段约 7-8 层
- 使用交错式调度 (1F1B interleaved) 减少流水线气泡 (bubble)
- 流水线气泡比例控制在约 **5%** 以内

#### 数据并行 (Data Parallelism, DP with FSDP)

| 属性 | 详情 |
|------|------|
| 方法 | FSDP (Fully Sharded Data Parallelism) |
| 分片策略 | 将优化器状态、梯度和权重全部分片 |
| 通信模式 | AllGather (前向) + ReduceScatter (反向) |
| 作用 | 减少每块 GPU 的内存占用 |

**FSDP 的内存优化**：标准 DDP 每块 GPU 保存完整参数 + 优化器状态（405B 需约 4.8TB，单块 H100 80GB 完全无法容纳）。FSDP 将参数和优化器状态分片到所有 DP 节点，前向/反向时通过 AllGather 临时收集需要的参数。

#### 上下文并行 (Context Parallelism, CP)

| 属性 | 详情 |
|------|------|
| CP 大小 | 视上下文长度动态调整 |
| 方法 | Ring Attention 变体 |
| 适用场景 | 长序列训练（从 8K 扩展到 128K） |
| 作用 | 将长序列拆分到多块 GPU 上计算注意力 |

**CP 是 Llama 3 引入的新并行维度**：
- 标准训练（8K 序列）：CP=1（不使用 CP）
- 长上下文训练（128K 序列）：CP>1
- CP 将输入序列沿序列维度切分，每块 GPU 负责序列的一部分
- 通过 Ring Attention 机制在 GPU 之间传递 KV 来完成完整注意力计算

### 5.3 性能指标

| 指标 | 数值 |
|------|------|
| 每 GPU 计算效率 | **380-430 TFLOPS** |
| Model FLOPs Utilization (MFU) | **38%-43%** |
| H100 理论峰值 (BF16) | 989 TFLOPS |
| 实际利用率 | 约 39-43% |
| 405B 训练总计算量 | 约 **3.8 × 10^25 FLOPs** |
| 训练时间 | 约 54 天 |

38%-43% 的 MFU 在万卡级训练中属于**非常优秀**的水平（损失主要来自通信等待、流水线气泡和内存管理开销）。

### 5.4 通信优化

- **计算-通信重叠**：反向传播时将 ReduceScatter 与后续层反向计算重叠，前向传播时将 AllGather 与前一层计算重叠，隐藏大部分通信延迟
- **网络拓扑感知调度**：同机架节点优先高带宽链路，跨机架用 ECMP 负载均衡，RoCE 使用 Priority Flow Control 避免拥塞
- **异步检查点**：写入不阻塞训练，用于故障恢复

### 5.5 训练稳定性

在 16,384 GPU 上持续训练 54 天，稳定性是最大的工程挑战之一：

**硬件故障统计**：
- 在 405B 训练期间，共经历 **466 次作业中断**
- 其中约 **78% 是计划外中断**（硬件故障、网络故障等）
- 约 **22% 是计划内中断**（维护、代码更新等）
- 平均每 **2-3 小时**就会发生一次非预期中断

**容错策略**：快速自动检测故障 GPU 并移出集群 → 热备节点替换 → 从最近检查点恢复。训练中出现约 **3 次较大的损失尖峰**，每次通过回滚检查点并跳过有问题数据批次来恢复。

### 核心要点

1. **16,384 块 H100 GPU**，2,048 个节点，三层 CLOS 网络互联
2. **4D 并行 = TP(8) × PP(16) × DP(FSDP) × CP**，CP 是新引入的长序列并行维度
3. **MFU 38%-43%**，每 GPU 380-430 TFLOPS，在万卡级训练中属于顶级水平
4. **466 次作业中断**，平均每 2-3 小时一次，对容错工程要求极高
5. **异步检查点 + 快速恢复**是保证训练连续性的关键

## 六、Scaling Laws ★★

> Meta 用一套系统化的方法从小规模实验预测了 405B 模型的最终性能。

### 6.1 IsoFLOPs 实验方法

核心目标：**在投入巨额计算资源之前，用小实验准确预测旗舰模型性能**。

Llama 3 采用 **IsoFLOPs 曲线方法**：固定计算预算 C → 训练不同大小的模型 → 画出"模型大小 vs 损失"曲线 → 曲线最低点对应该预算下的最优模型大小 → 对多个预算重复，从规律性中外推大模型性能。

### 6.2 实验规模

| 维度 | 范围 |
|------|------|
| 模型参数量 | **40M 到 16B** |
| 计算预算 | **6 × 10^18 到 10^22 FLOPs** |
| 实验次数 | 数百次训练运行 |
| 预测目标 | 405B 模型在 3.8 × 10^25 FLOPs 下的性能 |
| 外推跨度 | 跨越 **4 个数量级** 的计算规模 |

### 6.3 两步预测法 ★

Llama 3 的 Scaling Laws 采用了一个精心设计的**两步预测法**：

**第一步：预测负对数似然 (NLL)**
```
从小模型实验中拟合 Scaling Law：
  NLL(C) = A * C^(-α) + ε

其中：
  C = 计算预算 (FLOPs)
  A, α, ε = 拟合参数
  NLL = 验证集上的负对数似然损失

结果：
  对 405B 模型的 NLL 预测误差 < 2%
```

**第二步：从 NLL 映射到下游任务性能**
```
关键洞察：
  下游基准分数与 NLL 之间存在近似线性（或 sigmoid）关系

方法：
  1. 在不同规模模型上测量 (NLL, benchmark_score) 对
  2. 拟合映射函数：score = f(NLL)
  3. 将第一步预测的 405B NLL 代入映射函数
  4. 得到 405B 的下游基准预测分数
```

**为什么两步**：直接预测下游分数不可靠（离散、非平滑），NLL 是连续且平滑的，容易用 power law 拟合。

### 6.4 预测准确度与 Compute-Optimal 分析

**405B 的实际性能仅略微超过 Scaling Laws 预测**（NLL 预测误差 < 2%），说明该方法在跨 4 个数量级外推时仍然有效。

Meta 还发现 Llama 3 405B 实际使用了比 compute-optimal 更多的数据（15.6T tokens），这是**有意的过训练**：目标不是优化单个模型，而是让较小的模型（8B、70B）从过训练中获益更大——8B 模型推理效率更高，推理阶段节省的成本远超训练阶段的额外开支。

### 核心要点

1. **IsoFLOPs 方法**：在 40M-16B 范围内做实验，跨 4 个数量级外推到 405B
2. **两步预测法**：先预测 NLL（连续且平滑），再映射到下游任务分数
3. **预测误差极小**：405B 实际性能仅略超 Scaling Laws 预测
4. **有意过训练**：用超过 compute-optimal 的数据量训练，让小模型获益更多
5. **方法论意义**：为业界提供了一套可靠的大模型性能预测框架

## 七、后训练与对齐 ★★★

> Llama 3 的后训练流水线简洁而有效，核心思路是：用 DPO 替代 PPO，通过多轮迭代逐步提升。

### 7.1 整体流程

```
预训练模型 (Base Model)
  │
  ├── 第 1 轮 ──┐
  │              ├── Rejection Sampling → 生成高质量回答
  │              ├── SFT → 在高质量数据上监督微调
  │              └── DPO → 基于偏好数据做直接偏好优化
  │              ↓
  ├── 第 2 轮 ──┐
  │              ├── 用上一轮模型做 Rejection Sampling（质量更高）
  │              ├── SFT → 在更新的数据上微调
  │              └── DPO → 在更新的偏好数据上优化
  │              ↓
  ├── ... 重复 ...
  │              ↓
  └── 第 6 轮 ── 最终模型
```

**总共迭代 6 轮**，每一轮都使用上一轮的模型来生成更高质量的训练数据，形成正反馈循环。

### 7.2 SFT (Supervised Fine-Tuning)

**数据规模**：

| 维度 | 详情 |
|------|------|
| 总标注样本数 | **超过 1,000 万条** |
| 标注来源 | 人类标注 + 合成数据 |
| 涵盖领域 | 通用对话、代码、数学、推理、创意写作、多语言等 |
| 质量控制 | 多级人工审核 + 自动化质量评分 |

**SFT 数据构成**：人类标注的高质量 prompt-response 对 + 合成数据（用上一轮模型 Rejection Sampling 生成）。学习率约 10^-5，2-3 个 epoch，只在 response 部分计算损失。

### 7.3 Reward Model (奖励模型)

奖励模型是整个后训练流水线的核心评判标准。

**模型架构**：
- 基于 Llama 3 预训练模型初始化
- 移除最后的语言模型头 (LM head)，替换为一个标量输出头
- 输入一个 (prompt, response) 对，输出一个标量分数

**训练方法：Bradley-Terry 模型**
```
Bradley-Terry 偏好模型：
  P(response_w > response_l) = σ(r(response_w) - r(response_l))

其中：
  response_w = 人类偏好的回答 (winner)
  response_l = 人类不偏好的回答 (loser)
  r(·) = 奖励模型的打分函数
  σ(·) = sigmoid 函数

训练损失：
  L = -E[log σ(r(response_w) - r(response_l))]
```

**偏好数据收集**：
- 对同一个 prompt，让模型生成多个候选回答
- 人类标注员对候选回答进行**排序**（而非简单的二选一）
- 支持 2-7 个候选回答的排序标注
- 从排序中可以构造多个 (winner, loser) 对

### 7.4 DPO 取代 PPO ★★

Llama 3 在对齐阶段选择了 **DPO (Direct Preference Optimization)** 而非传统的 **PPO (Proximal Policy Optimization)**。这是一个具有重大意义的工程决策。

**PPO vs DPO 对比**：

| 维度 | PPO | DPO |
|------|-----|-----|
| 同时运行模型数 | 4 个（策略、参考、奖励、价值） | 2 个（策略、参考） |
| 内存开销 | 极大（405B × 4） | 减半 |
| 训练稳定性 | 超参数敏感 | 更可预测 |
| 奖励 hacking 风险 | 存在 | 无 |
| 可迭代性 | 复杂 | 可直接在偏好数据上迭代 |

**DPO 损失函数**：
```
L_DPO = -E[log σ(β * (log π(y_w|x)/π_ref(y_w|x) - log π(y_l|x)/π_ref(y_l|x)))]

π = 当前策略模型, π_ref = 参考模型（SFT 后）
y_w = 偏好回答, y_l = 非偏好回答, β = 温度参数
```

Meta 选择 DPO 的核心原因：在 405B 规模上，PPO 需要同时运行 4 个超大模型，工程难度极高；DPO 更简单、更稳定、可迭代，且性能不逊于 PPO。

### 7.5 Rejection Sampling ★

拒绝采样是 Llama 3 后训练流水线中提升数据质量的核心手段。

```
Rejection Sampling 流程：
  1. 对每个 prompt，用当前模型生成 K 个候选回答（K 通常为 10-30）
  2. 用奖励模型对所有候选回答打分
  3. 只保留分数最高的 1-3 个回答
  4. 丢弃其余低分回答
  5. 将保留的高分回答用于下一轮 SFT 训练

关键参数：
  - 采样温度：通常 > 1.0（增加多样性）
  - 采样数量 K：10-30（越大，能找到更好回答的概率越高，但计算成本也越高）
  - 筛选比例：保留 Top 10%-30%
```

同一模型在不同采样路径下输出质量差异巨大，通过多次采样 + 奖励筛选可以"发现"模型已有但不总是表达的高质量回答，再反馈给 SFT 形成正循环。

### 7.6 合成数据在后训练中的角色

Llama 3 大量使用合成数据来增强后训练效果：

| 合成数据类型 | 生成方法 | 用途 |
|-------------|---------|------|
| 代码质量数据 | 让模型生成代码 → 执行 → 验证正确性 | 代码能力 SFT |
| 数学推理数据 | 让模型生成解题过程 → 验证答案 | 数学能力 SFT |
| 对话改写数据 | 让模型改写低质量回答为高质量版本 | 通用能力 SFT |
| 安全拒绝数据 | 让模型对有害 prompt 生成安全回答 | 安全对齐 |
| 多语言翻译数据 | 英文高质量数据 → 翻译到其他语言 | 多语言能力 |

### 7.7 关键训练细节

- **学习率**：SFT 约 10^-5（Cosine schedule），DPO 约 5 × 10^-7
- **DPO 正则化**：β=0.1 + NLL 正则化损失（防止遗忘 SFT 学到的格式）
- **数据混合**：每轮 SFT 数据包含当前轮新数据 + 之前轮精选数据，新数据占比逐轮增大

### 核心要点

1. **6 轮迭代**的 Rejection Sampling → SFT → DPO 流水线
2. **超 1000 万标注样本**，人类标注 + 合成数据混合
3. **DPO 取代 PPO**：工程更简单、训练更稳定、可迭代、405B 规模下尤其有优势
4. **Bradley-Terry 奖励模型**：从偏好排序中学习打分
5. **Rejection Sampling**：每轮用 10-30 次采样 + 奖励筛选获取高质量训练数据
6. **合成数据**：代码验证、数学验证、改写、翻译等多种方式生成高质量训练数据

## 八、安全体系

> Llama 3 构建了一个多层级的安全防护系统，而非依赖单一的安全过滤器。

### 8.1 系统级安全哲学

Meta 的安全哲学是**"系统级安全" (system-level safety)**：

```
传统做法：
  在模型层面做所有安全防护
  → 模型要么过度拒绝（影响有用性），要么不够安全

Meta 的做法：
  分层防护，每一层负责不同的安全任务

  ┌────────────────────────────────────────┐
  │  第 1 层: Prompt Guard (前端防护)       │  ← 检测恶意 prompt
  ├────────────────────────────────────────┤
  │  第 2 层: Llama Guard 3 (输入审核)     │  ← 分类输入风险
  ├────────────────────────────────────────┤
  │  第 3 层: 模型内置安全对齐              │  ← 模型自身的安全行为
  ├────────────────────────────────────────┤
  │  第 4 层: Llama Guard 3 (输出审核)     │  ← 分类输出风险
  ├────────────────────────────────────────┤
  │  第 5 层: Code Shield (代码审核)       │  ← 检测不安全代码
  └────────────────────────────────────────┘
```

### 8.2 Llama Guard 3

Llama Guard 3 是一个独立的安全分类器模型，负责对输入和输出进行风险分类。

**13 类风险分类表**：

| 编号 | 风险类别 | 说明 |
|------|---------|------|
| S1 | 暴力犯罪 | 煽动、计划或美化暴力行为 |
| S2 | 非暴力犯罪 | 欺诈、盗窃、网络犯罪等 |
| S3 | 性相关内容 | 色情或性暗示内容 |
| S4 | 儿童安全 | 任何涉及儿童性虐待的内容 |
| S5 | 诽谤 | 针对个人的虚假和有害声明 |
| S6 | 专业建议 | 未经验证的法律、医疗、金融建议 |
| S7 | 隐私 | 泄露个人可识别信息 (PII) |
| S8 | 知识产权 | 侵犯版权、商标等 |
| S9 | 自杀自伤 | 煽动或美化自杀/自伤 |
| S10 | 仇恨言论 | 基于身份的歧视和仇恨 |
| S11 | 选举干预 | 虚假选举信息、投票压制 |
| S12 | 代码安全 | 恶意代码、漏洞利用 |
| S13 | 违禁物品 | 涉及毒品、武器等管制物品 |

**Llama Guard 3 的技术实现**：
- 基于 Llama 3 8B 微调
- 输入格式：`[SYSTEM_PROMPT] + [USER_MESSAGE] + [ASSISTANT_MESSAGE]`
- 输出格式：`safe` 或 `unsafe\nS{类别编号}`
- 支持自定义安全策略（可添加/移除风险类别）

### 8.3 Prompt Guard 与 Code Shield

**Prompt Guard**：基于多语言 BERT 变体的前端防护，毫秒级延迟，检测越狱 (Jailbreak)、间接注入 (Injection)、角色扮演攻击等恶意 prompt。

**Code Shield**：使用静态分析检测模型生成代码中的安全问题（不安全系统调用、SQL 注入、硬编码密钥等），支持 Python、JavaScript、C++ 等。

### 8.4 红队测试

内部安全团队 + 外部专家进行多轮迭代红队测试，覆盖 13 类风险 × 多种攻击手法 × 多语言，并使用 AI 自动生成攻击 prompt。

### 核心要点

1. **系统级安全**：五层纵深防护，不依赖单一模型
2. **Llama Guard 3**：13 类风险分类，基于 Llama 3 8B 微调，支持自定义策略
3. **Prompt Guard + Code Shield**：前端恶意检测 + 代码静态分析
4. **大规模红队测试**：人工 + AI 自动化多轮攻击测试

## 九、多模态扩展

> Llama 3 系列通过组合式架构逐步扩展到视觉和语音模态，核心思想是保持语言模型不变。

### 9.1 Llama 3.2 视觉模型

Llama 3.2 将视觉能力引入了 Llama 3 系列，发布了 11B 和 90B 两个支持图像输入的模型。

**模型配置**：

| 模型 | 语言参数 | 视觉编码器 | 总参数 | 图像分辨率 |
|------|---------|-----------|--------|-----------|
| Llama 3.2 11B-Vision | 8B | ViT-H/14 | ~11B | 最高 1120×1120 |
| Llama 3.2 90B-Vision | 70B | ViT-H/14 | ~90B | 最高 1120×1120 |

### 9.2 组合式架构

```
Llama 3.2 视觉架构：

  图像输入                    文本输入
    │                          │
    ▼                          │
  ViT 视觉编码器               │
  (预训练，冻结或微调)          │
    │                          │
    ▼                          │
  Cross-Attention 适配层        │
  (新增，训练)                 │
    │                          │
    ▼                          ▼
  ┌──────────────────────────────────┐
  │     Llama 3 语言模型             │
  │  (大部分参数冻结，保持语言能力)    │
  └──────────────────────────────────┘
    │
    ▼
  文本输出
```

**关键设计决策**：
- **语言模型核心冻结**：视觉训练不改变大部分语言参数
- **Cross-Attention 适配层**：视觉特征作为 Key/Value，语言特征作为 Query，只有这些新增层需要从头训练
- **ViT 编码器**：预训练 Vision Transformer，高分辨率图像通过 tile 机制分割后独立编码

### 9.3 视觉能力

| 能力 | 说明 | 性能 |
|------|------|------|
| 图像理解 | 描述图像内容、回答图像相关问题 | 与 GPT-4V 同级别 |
| 文档理解 | 解析文档、表格、图表 | DocVQA 上表现优异 |
| 图表分析 | 理解和解释图表数据 | ChartQA 上表现优异 |
| VQA (视觉问答) | 基于图像回答开放式问题 | VQAv2 上表现优异 |
| OCR | 读取图像中的文字 | 支持多语言 OCR |
| 空间推理 | 理解物体的位置和空间关系 | 需改善 |

### 9.4 语音探索

报告还提到了语音模态的初步探索（语音编码器 + 语言模型，支持语音输入 → 文本输出），仍处于实验阶段，为后续 Llama 4 多模态统一模型奠定基础。

### 核心要点

1. **组合式架构**：ViT + Cross-Attention 适配层 + 冻结语言模型，保持语言能力不退化
2. **两个视觉模型**：11B（基于 8B）和 90B（基于 70B），覆盖图像理解、文档/图表分析、VQA、OCR
3. **语音探索**：为 Llama 4 多模态统一模型做准备

## 十、工具调用

> Llama 3 系统性地增强了函数调用能力，使其可以作为 Agent 系统的核心模型。

### 10.1 支持的调用模式

| 调用模式 | 说明 | 示例 |
|---------|------|------|
| 单次调用 (Single Call) | 调用一个函数，一次完成 | `get_weather("Beijing")` |

### 核心要点

1. **405B 在 ARC-Challenge、GSM8K、BIG-Bench Hard 上是最佳**
2. **总体与 GPT-4o 和 Claude 3.5 Sonnet 在同一水平**，但在 MATH 和 GPQA 上略有差距
3. **8B 和 70B 在同规模开源模型中大幅领先**（发布时）
4. **Llama 3.3 70B** 通过后训练优化，部分指标接近甚至超过 405B
5. **多语言能力有限**：因训练数据以英文为主（95%），中文等非英文语言的能力仍有提升空间

## 十二、核心贡献与启示

### 十大核心贡献

1. **Scaling Law 预测方法论**：可跨 4 个数量级外推的两步预测法，为业界提供了规划超大模型训练的方法论基础
2. **数据质量为王**：15.6T tokens + 两级质量分类器 + 三级去重，数据工程是最核心竞争力
3. **4D 并行训练工程**：TP + PP + FSDP + CP，目前公开最完整的万卡级训练方案
4. **简洁后训练流水线**：Rejection Sampling → SFT → DPO 迭代 6 轮，DPO 有效替代 PPO
5. **128K 词表扩展**：tiktoken BPE，编码效率 +15%，显著改善多语言支持
6. **系统级安全架构**：五层纵深防御（Prompt Guard → Llama Guard → Code Shield）
7. **组合式多模态**：Cross-Attention 适配层嫁接视觉能力，语言能力不退化
8. **工具调用增强**：四种调用模式，并行调用超越 GPT-4o 7%
9. **数据退火策略**：训练末期增大高质量数据占比，低成本高回报
10. **Dense 架构极限探索**：证明 Dense 在充足数据下不输 MoE，且部署更简单

### 对行业的启示

**数据 vs 架构**：
```
Llama 3 的核心信条：
  "在标准架构上推大数据量" > "在新架构上推参数量"

对比：
  DeepSeek-V3: 架构创新 (MoE + MLA + FP8) + 14.8T tokens
  Llama 3:     标准架构 (Dense Transformer)  + 15.6T tokens

两种路线都达到了 GPT-4 级别的性能
→ 数据质量和数量是更稳定的性能提升路径
→ 架构创新可以显著降低成本（DeepSeek-V3 仅 $5.6M vs Llama 3 估算 $60M+）
```

**开源 vs 闭源**：
- Llama 3 405B 证明开源模型可以达到闭源前沿水平
- 但 Llama 3 仍然是"仅开源权重"，不公开训练数据和代码
- 真正的"完全开源"参见 OLMo 3
- Llama 3 的 Community License 对月活 7 亿以下的企业完全免费

**后训练的杠杆效应**：
- Llama 3.3 70B 通过更好的后训练，在部分基准上超过了 405B
- 这说明后训练的投入产出比可能远高于预训练
- DPO + 迭代式 Rejection Sampling 是一条可行且高效的路线

### 与其他重要工作的关系

| 维度 | Llama 3 | DeepSeek-V3 | OLMo 3 |
|------|---------|-------------|--------|
| **核心策略** | 数据量 + Dense 架构 | 架构创新 + 成本优化 | 完全开源 + 三阶段训练 |
| **训练成本** | ~$60M+（估算） | ~$5.6M | 未公开 |
| **架构** | Dense Transformer | MoE + MLA | Dense Transformer |
| **数据量** | 15.6T tokens | 14.8T tokens | ~6T tokens |
| **开源程度** | 仅权重 | 仅权重 + 论文 | 完全开源 |
| **最大亮点** | 全栈工程文档 | 成本效率 | 数据透明度 |
| **最大局限** | 训练成本高 | Dense 推理效率低 | 数据量受限 |

### 值得深入研究的方向

1. **数据退火的最优配比**：在什么时间点开始退火？高质量数据的上采样倍率应该是多少？
2. **DPO vs PPO 的规模边界**：在什么规模下 DPO 的优势最大？是否存在 PPO 更好的场景？
3. **Context Parallelism 的实现细节**：如何在 Ring Attention 中高效处理因果掩码？
4. **两级质量分类器的阈值选择**：FastText 和 DistilRoBERTa 的阈值如何互相配合？
5. **Scaling Laws 在多模态中的适用性**：两步预测法能否扩展到视觉-语言模型？
