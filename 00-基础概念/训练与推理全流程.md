## Part A: 训练的本质 — 从猜测到学习

### 1. 神经网络 = 一个巨大的函数

你可以把一个大语言模型（LLM）理解成一个**超级复杂的函数**：

```
输入：一段文字（"今天天气"）
  ↓
  [ 巨大的函数 f(x) ]
  ↓
输出：下一个词的概率分布
  "真好" → 0.35
  "不错" → 0.25
  "很冷" → 0.15
  "怎样" → 0.10
  ...（词表中每个词都有一个概率）
```

这个函数里面有大量的**参数**（也叫权重），参数就是模型的"知识"：

| 模型 | 参数量 | 类比 |
|------|--------|------|
| GPT-2 | 1.5 亿 | 一本百科全书的目录 |
| Llama 3 8B | 80 亿 | 一座小型图书馆 |
| Llama 3 70B | 700 亿 | 一座大型图书馆 |
| DeepSeek-V3 | 6710 亿（总） | 一座超级图书馆，但每次只用 370 亿 |

**训练 = 调整这些参数，让模型的预测越来越准。**

一开始这些参数是随机初始化的（相当于一个婴儿，什么都不知道），然后通过海量数据的学习，一步步把参数调到合适的值。

---

### 2. 损失函数 — 衡量"猜得多离谱"

训练的核心问题是：**模型猜得对不对？差多少？**

我们用**损失函数（Loss Function）**来衡量。大语言模型用的是**交叉熵损失（Cross-Entropy Loss）**。

**直觉理解**：

假设正确答案是"猫"：

```
场景 1：模型说 "猫" 的概率 = 0.9（很自信，而且猜对了）
  损失 = -log(0.9) = 0.105    ← 很小，奖励！

场景 2：模型说 "猫" 的概率 = 0.1（不自信，虽然答案在选项里但排很后面）
  损失 = -log(0.1) = 2.303    ← 很大，惩罚！

场景 3：模型说 "猫" 的概率 = 0.01（几乎没猜到）
  损失 = -log(0.01) = 4.605   ← 非常大，严厉惩罚！
```

**公式**（人话版）：

```
损失 = -log(模型给正确答案的概率)
```

- 概率越高 → log 值越大 → 加负号后越小 → 损失越小 → 好！
- 概率越低 → log 值越小（负得更多）→ 加负号后越大 → 损失越大 → 差！

**训练目标：让所有训练数据上的总损失越来越小。**

一个形象的类比：损失函数就像考试分数的"反面"—— 考得越好，损失越低；考得越差，损失越高。训练就是不断刷题，让"考试失分"越来越少。

---

### 3. 反向传播 — "谁该负责" ★

现在我们知道了"猜得多差"（损失），下一个问题是：**80 亿个参数中，每个参数该怎么调？**

这就是**反向传播（Backpropagation）**的工作。

**直觉**：
考试考砸了（损失很大），你需要从最后一道错题开始，一步步追溯是哪里学错了：
- 最后一道题错了 → 是因为某个公式记错了 → 是因为某个概念没理解 → 是因为基础定义没学好

反向传播做的就是同样的事情：从输出层（最终预测）开始，沿着网络逐层往回算，计算每个参数对损失的影响程度（这个影响程度叫做**梯度**）。

**数学基础：链式法则**

```
如果 y = f(g(x))，那么 dy/dx = f'(g(x)) × g'(x)
```

人话版：如果一个结果是经过多步计算得到的，那么每一步的影响可以用"乘法"串起来。

**完整流程图**：

```
前向传播（Forward Pass）                      反向传播（Backward Pass）
========================                      ========================

输入文字                                       损失 = 2.3
  │                                              │
  ▼                                              ▼
[第1层] ──参数 w1──→ 中间结果1          梯度 ∂L/∂w3 ←── [第3层]
  │                                              │
  ▼                                              ▼
[第2层] ──参数 w2──→ 中间结果2          梯度 ∂L/∂w2 ←── [第2层]
  │                                              │
  ▼                                              ▼
[第3层] ──参数 w3──→ 预测结果            梯度 ∂L/∂w1 ←── [第1层]
  │
  ▼
计算损失 ─────────────────────────────→ 开始反向传播

方向：从左到右 →                         方向：从右到左 ←
目的：用当前参数做预测                   目的：算出每个参数该怎么调
```

每个参数都会得到一个**梯度值**：
- 梯度为正 → 这个参数增大会让损失增大 → 应该减小它
- 梯度为负 → 这个参数增大会让损失减小 → 应该增大它
- 梯度绝对值大 → 这个参数影响大 → 需要大幅调整
- 梯度绝对值小 → 这个参数影响小 → 微调即可

---

### 4. 梯度下降 — 调整参数 ★

有了梯度，我们就知道该怎么调参数了。

**核心公式**：

```
参数_new = 参数_old - 学习率 × 梯度
```

**直觉：下山**

想象你站在一座山上（山的高度 = 损失），你的目标是到达山谷（损失最小）。

- **梯度** = 脚下最陡的上坡方向
- **负梯度** = 最陡的下坡方向
- **学习率** = 你每步迈多大

```
                    *  ← 起始位置（随机初始化）
                   / \
                  /   \
                 /     * ← 第 1 步
                /       \
               /         * ← 第 2 步
              /           \
             /             * ← 第 3 步
            /               \
           /        *        \
          /      ← 最低点      \
         /       (目标!)        \
```

**学习率的重要性**：

```
学习率太大（步子太大）：                学习率太小（步子太小）：

    *         *                             *
     \       /                              *
      \     /                               *
       \   /                                *
        \ /                                 *
    ─────X───── ← 在最低点来回震荡            *
        / \                                 *
       /   \                                * ← 100步了还在半山腰
                                            ...
                                            （训练时间太长）

学习率刚好：

    *
     \
      \
       \
        \
         * ← 稳稳到达最低点
```

**典型学习率**：大语言模型通常用 1e-4 到 3e-4（即 0.0001 到 0.0003），非常小。

---

### 5. 优化器的演化

实际训练中，我们不直接用最朴素的梯度下降，而是用更聪明的**优化器（Optimizer）**。

| 优化器 | 核心想法 | 类比 | 现状 |
|--------|---------|------|------|
| SGD | 最基础的梯度下降 | 每次看一个方向就走 | 简单但慢 |
| SGD + Momentum | 加"惯性" | 滚雪球，越滚越快，不容易卡在小坑 | 经典方法 |
| Adam | 自适应学习率 + 动量 | 给每个参数配一个"私人教练"，单独调节步幅 | 最常用 |
| AdamW | Adam + 权重衰减 | Adam + 防止参数长太大的"弹簧" | 大模型标配 |
| Muon/MuonClip | 牛顿法近似，更精确的方向 | 不只看坡度，还看地形曲率 | 2025 前沿 |

**为什么 Adam 这么流行？**

SGD 对所有参数用同一个学习率，但不同参数的"地形"差异很大。Adam 给每个参数维护两个统计量：
1. **一阶矩（动量）**：记住之前的梯度方向，避免来回震荡
2. **二阶矩（自适应学习率）**：如果一个参数的梯度一直很大，就自动减小它的学习率；反之增大

类比：不同科目用不同的学习策略。数学梯度大（难度大），就放慢节奏仔细学；语文梯度小（相对简单），就加快进度。

---

### 6. 学习率调度 ★

学习率不是固定的，而是按照一个精心设计的**调度策略（Schedule）**变化。

**最主流的策略：Warmup + Cosine Decay**

```
学习率
  ^
  |          peak (如 3e-4)
  |         /‾‾‾‾‾‾\
  |        /         \
  |       /            \
  |      /               \
  |     /                  \
  |    /                     \
  |   /                        \_____ min (如 3e-5)
  |  /
  | /
  |/ ← 从接近 0 开始
  +──────────────────────────────────→ 训练步数
  |<--->|<------------- --------->|
  warmup     cosine decay
  (2000步)   (剩余所有步)
```

**为什么要 Warmup？**

训练刚开始时，参数是随机的，梯度非常不稳定。如果一上来就用大学习率，模型容易"发疯"（loss 飙升）。所以先用很小的学习率"热身"，等模型稳定了再加速。

类比：开车先慢慢加速，等上了高速再全速前进，到了目的地附近再减速。

**为什么要 Cosine Decay？**

训练后期，模型已经学到了大部分知识，需要"微调细节"。学习率太大会破坏已经学好的部分，所以慢慢降下来。

**典型设置**：
- Warmup: 2000 步（从接近 0 线性增长到 peak）
- Peak 学习率: 3e-4
- Cosine 衰减到: peak 的 1/10（即 3e-5）

## Part B: 预训练 — 教模型"阅读" ★★★

预训练（Pre-training）是大模型训练中最消耗资源的阶段，通常占总训练算力的 95% 以上。

### 1. 目标：Next Token Prediction

预训练的目标惊人地简单：

**给前 N 个 token，预测第 N+1 个 token。**

```
输入:  "今天 天气 真"
目标:  预测下一个 token → "好" (概率最高)

输入:  "The capital of France is"
目标:  预测下一个 token → "Paris" (概率最高)
```

**Token 是什么？**

Token 不完全等于"词"，而是模型处理文本的最小单位。中文大约 1-2 个字 = 1 个 token，英文大约 1 个词 = 1-2 个 token。

就这么简单的目标，在**万亿级 token** 上训练，模型竟然涌现出了各种能力：

```
训练目标：预测下一个词
   ↓ 在海量数据上训练
涌现能力：
  ├── 语法理解（"他跑得很___" → "快"）
  ├── 常识知识（"水在100度会___" → "沸腾"）
  ├── 逻辑推理（"如果A>B, B>C, 那么A___C" → ">"）
  ├── 翻译能力（看了大量双语文本后）
  ├── 编程能力（看了大量代码后）
  └── 数学能力（看了大量数学推导后）
```

为什么预测下一个词能涌现这些能力？因为要准确预测下一个词，模型必须**真正理解**前面的内容。要预测"水在100度会沸腾"，模型必须学会物理常识；要预测代码的下一行，模型必须理解编程逻辑。

---

### 2. 数据从哪来

预训练需要海量文本数据，这些数据的来源和处理方式直接决定模型质量。

**数据来源**：

```
互联网爬虫 (Common Crawl)
  │
  ├── 网页文本 → FineWeb / Dolma / RedPajama
  ├── 书籍 → Books3 / Project Gutenberg
  ├── 学术论文 → arXiv / S2ORC
  ├── 代码 → GitHub / The Stack
  ├── 百科 → Wikipedia (多语种)
  ├── 论坛 → StackOverflow / Reddit
  └── 其他 → 专利、法律文档、新闻...
```

**数据清洗流程**（非常关键！）：

```
原始爬虫数据 (数百 TB)
  │
  ├── 1. 语言过滤 → 去掉非目标语言的页面
  ├── 2. 质量过滤 → 去掉广告、垃圾、乱码
  ├── 3. 去重 → MinHash / SimHash 去除重复内容
  ├── 4. 敏感内容过滤 → 去除有害/违法内容
  ├── 5. PII 过滤 → 去除个人身份信息
  └── 6. 数据配比 → 按类别混合（如：网页 60%、代码 15%、学术 10%...）
  │
  ▼
清洗后数据 (5-15T tokens)
```

**关键洞察：数据质量 >> 数据数量**

一条高质量的维基百科文章对模型的帮助，远超一百条低质量的广告文本。现代模型在数据清洗上投入的工程量，甚至超过模型架构设计本身。

**现代模型的数据规模**：

| 模型 | 训练 Token 数 | 类比 |
|------|--------------|------|
| GPT-2 (2019) | 40B | 约 300 亿字 |
| Llama 2 (2023) | 2T | 约 1.5 万亿字 |
| Llama 3 (2024) | 15T | 约 11 万亿字 |
| DeepSeek-V3 (2024) | 14.8T | 约 11 万亿字 |

### 3. 一个训练步骤（Step）的完整流程 ★★

让我们放大到**一个训练步骤**，看看到底发生了什么：

```
步骤 1: 采样数据
  └── 从数据集随机抽一个 batch（如 4M tokens，约 300 万字）

步骤 2: 前向传播 (Forward Pass)
  └── 模型读取 batch 中的每段文本
  └── 对每个位置，预测下一个 token 的概率分布
  └── 例："今天天气" → 模型预测所有可能的下一个词的概率

步骤 3: 计算损失 (Compute Loss)
  └── 把模型的预测和真实的下一个 token 对比
  └── 用交叉熵损失衡量差距
  └── 得到一个数字，如 loss = 3.2

步骤 4: 反向传播 (Backward Pass)
  └── 从 loss 出发，逐层往回算梯度
  └── 每个参数都得到一个 "该怎么调" 的信号

步骤 5: 优化器更新 (Optimizer Step)
  └── 用 AdamW 优化器，根据梯度调整所有参数
  └── 参数_new = 参数_old - 学习率 × 优化后的梯度

步骤 6: 记录指标 (Logging)
  └── 记录 loss, 学习率, 梯度范数 (gradient norm)
  └── 发送到监控系统（如 Weights & Biases）

重复以上步骤... 数十万到数百万次
```

**一些具体数字**：

| 指标 | 典型值 |
|------|--------|
| Batch Size (token数) | 4M - 60M tokens |
| 总训练步数 | 20 万 - 300 万步 |
| 每步时间 | 几秒到几十秒（取决于模型大小和GPU数量） |
| 总训练时间 | 几周到几个月 |
| 训练开始的 loss | ~10-11（接近随机猜测） |
| 训练结束的 loss | ~1.5-2.5（已经很准了） |

**Loss 下降曲线长什么样？**

```
Loss
  ^
11|*
  |*
  | *
  | *
 8|  *
  |   *
  |    **
 5|      ***
  |         ****
  |             ******
 2|                   ****************************
  |
  +──────────────────────────────────────────────→ 训练步数
  0           100k        200k        300k

  特点：开始下降很快，后面越来越慢（收益递减）
```

### 4. 为什么需要并行化 ★★

**问题：一块 GPU 装不下大模型！**

让我们算一笔账：

```
模型参数: 70B (700 亿个)
每个参数存储 (FP16): 2 字节
参数内存: 70B × 2 = 140 GB

但是！训练时还需要额外存储：
  ├── 梯度 (和参数一样大): 140 GB
  ├── 优化器状态 (Adam 需要 2 份): 280 GB
  └── 中间激活值: ~200 GB (取决于 batch size)

总内存需求: 约 760 GB

而一块 H100 GPU 只有: 80 GB
```

**差了快 10 倍！** 所以必须把模型和计算"切开"，分到多块 GPU 上。

**并行化策略对比**：

```
┌──────────────────────────────────────────────────────────────┐
│                    并行化策略总览                              │
├──────────────┬──────────────────────┬───────────────────────┤
│   策略        │    做什么             │    切什么             │
├──────────────┼──────────────────────┼───────────────────────┤
│ 数据并行(DP) │ 每块GPU一份完整模型  │ 切数据                 │
│              │ 不同GPU处理不同数据  │ (每块GPU看不同的batch) │
├──────────────┼──────────────────────┼───────────────────────┤
│ 张量并行(TP) │ 把一层内的大矩阵切开│ 切矩阵(行或列)         │
│              │ 分到多块GPU上        │ (需要高速互联)         │
├──────────────┼──────────────────────┼───────────────────────┤
│ 流水线并行(PP)│ 不同层放不同GPU     │ 切层                   │
│              │ 像工厂流水线        │ (第1-10层在GPU0,       │
│              │                     │  第11-20层在GPU1...)   │
├──────────────┼──────────────────────┼───────────────────────┤
│ 序列并行(CP) │ 长序列切到多块GPU   │ 切序列                  │
│              │ 处理超长上下文       │ (前半段GPU0,后半段GPU1)│
├──────────────┼──────────────────────┼───────────────────────┤
│ 专家并行(EP) │ MoE中不同专家放不同 │ 切专家                  │
│              │ GPU                  │ (专家0-63在GPU0...)    │
└──────────────┴──────────────────────┴───────────────────────┘
```

**类比：搬家**

- 数据并行：请 4 个搬家队，每队搬不同楼层的东西（同样的方法，不同的数据）
- 张量并行：一个超大沙发切成 4 块，4 个人各搬一块（拆分单个大物件）
- 流水线并行：一个人打包 → 一个人搬下楼 → 一个人装车 → 一个人运走（接力）
- 序列并行：一本超长的书，4 个人各读 1/4（拆分长内容）

**实际使用：同时混合多种并行策略**

```
DeepSeek-V3 的并行配置（2048 块 H800 GPU）：
├── 16-way 流水线并行 (PP)  → 模型的 61 层分到 16 组
├── 64-way 专家并行 (EP)    → 256 个专家分到 64 块 GPU
├── 数据并行 (DP)           → 多份模型副本并行处理数据
└── 总 GPU: 2048 块 H800

Llama 3 405B 的并行配置（16,384 块 H100 GPU）：
├── 8-way 张量并行 (TP)
├── 16-way 流水线并行 (PP)
├── 数据并行 (DP)
└── 总 GPU: 16,384 块 H100
```

### 5. Scaling Laws — 多大的模型需要多少数据 ★

**Scaling Laws（缩放定律）** 是大模型领域最重要的发现之一：模型的性能可以被预测！

**Kaplan Scaling Laws (2020, OpenAI)**：

```
模型性能（Loss）与三个因素存在幂律关系：
  1. 模型参数量 N
  2. 训练数据量 D
  3. 训练算力 C

Loss ∝ N^(-0.076)    （参数越多，loss 越低）
Loss ∝ D^(-0.095)    （数据越多，loss 越低）
Loss ∝ C^(-0.050)    （算力越多，loss 越低）
```

**人话版**：模型越大、数据越多、训练越久，模型就越好。而且这个关系是可以用数学公式预测的。

**Chinchilla 定律 (2022, DeepMind)**：

Chinchilla 的关键发现：之前的模型都"训练不充分"了！

```
旧观念：尽可能把模型做大
  → GPT-3 (175B 参数) 只训了 300B tokens

Chinchilla 定律：在给定算力下，存在最优的模型大小和数据量组合
  → 最优比例：约 1 个参数对应 20 个 token
  → 70B 参数模型最优需要 1.4T tokens

例：
  同样的算力预算 →
    方案A: 175B 模型 × 300B tokens  = 效果一般
    方案B:  70B 模型 × 1.4T tokens  = 效果更好!
```

**但实际操作中，现代模型远超 Chinchilla 最优比例**：

| 模型 | 参数量 | Chinchilla 最优 Token | 实际训练 Token | 超标倍数 |
|------|--------|----------------------|---------------|---------|
| Llama 3 8B | 8B | 160B | 15T | 约 94x |
| DeepSeek-V3 | 671B | 13.4T | 14.8T | 约 1.1x |

为什么要远超 Chinchilla 最优？因为**推理成本比训练成本更重要**。多花一次训练的钱，换一个更小但更聪明的模型，长期推理时可以省大量的钱。

**核心洞察**：Scaling Laws 让大模型训练从"炼丹"（碰运气）变成了"工程"（可预测）。你可以在小规模上实验，然后推算大规模的效果。

### 6. 混合精度训练 ★

**精度（Precision）** 指的是用多少位（bit）来表示一个数字。

| 精度 | 位数 | 每参数字节 | 能表示的精度 | 谁在用 |
|------|------|-----------|-------------|--------|
| FP32 | 32 位 | 4 字节 | 最高 | 优化器状态（Adam 的动量等） |
| BF16 | 16 位 | 2 字节 | 高（范围大） | 主流训练精度 |
| FP16 | 16 位 | 2 字节 | 高（精度细） | 部分训练场景 |
| FP8  | 8 位  | 1 字节 | 中 | DeepSeek-V3 首创用于训练 |
| INT4 | 4 位  | 0.5 字节 | 低 | 仅推理（量化后） |

**为什么要降低精度？**

```
FP32 训练 70B 模型:
  参数: 70B × 4 bytes = 280 GB
  速度: 慢（每次计算处理 32 位）

BF16 训练 70B 模型:
  参数: 70B × 2 bytes = 140 GB   ← 省一半内存!
  速度: 快约 2x（每次计算处理 16 位）
  精度损失: 几乎可以忽略

FP8 训练 (DeepSeek-V3 创新):
  参数: 70B × 1 byte = 70 GB     ← 又省一半!
  速度: 快约 4x
  精度损失: 需要特殊技巧来保证质量
```

**混合精度的"混合"体现在哪？**

不是所有地方都用低精度：
- **前向传播和反向传播**：用 BF16/FP8（快！省内存！）
- **参数的 master copy**：用 FP32（保证精度不丢失）
- **优化器状态**：用 FP32（Adam 需要高精度的动量统计）

类比：你可以用草稿纸（低精度）快速演算，但最终结果要工整地抄到答题卡（高精度）上。

## Part C: 训练中的常见问题与解决 ★★

训练大模型就像开一辆在悬崖边行驶的赛车——速度要快，但随时可能翻车。

### 1. 梯度消失 / 梯度爆炸

**梯度消失（Vanishing Gradient）**：

```
反向传播时，梯度经过每一层都会乘以一个系数。
如果这个系数 < 1，经过几十层后：

0.9 × 0.9 × 0.9 × ... (60 层) = 0.9^60 ≈ 0.002

梯度变得极其微小 → 浅层参数几乎不更新 → 模型前面的层"学不到东西"
```

**梯度爆炸（Exploding Gradient）**：

```
反过来，如果系数 > 1：

1.1 × 1.1 × 1.1 × ... (60 层) = 1.1^60 ≈ 304

梯度变得极其巨大 → 参数更新过猛 → 模型"飞了"（loss 变成 NaN）
```

**解决方案**：

| 方法 | 原理 | 效果 |
|------|------|------|
| **Pre-Norm** | 在每层计算前先做归一化 | 稳定梯度，现代 Transformer 标配 |
| **残差连接** | 每层的输出 = 该层计算 + 直接跳过（shortcut） | 梯度可以"走捷径"直达浅层 |
| **梯度裁剪** | 如果梯度超过阈值就截断 | 防止梯度爆炸，典型阈值 = 1.0 |

**残差连接的直觉**：

```
没有残差连接:
  输入 → [第1层] → [第2层] → ... → [第60层] → 输出
  梯度必须经过所有层，容易消失或爆炸

有残差连接:
  输入 ─────────────────────────────────────→ (+) → 输出
    └─→ [第1层] → [第2层] → ... → [第60层] ─→↗

  梯度有两条路：一条经过所有层，一条直接"跳"到浅层
  即使经过所有层的梯度消失了，"跳"过去的梯度还在！
```

---

### 2. Loss Spike — 训练突然崩溃 ★

**现象**：训练得好好的，loss 突然飙升。

```
Loss
  ^
  |                    *
  |                   * * ← Loss Spike!
  |                  *   *
  |                 *     *
 2|*****************       **************
  |
  +──────────────────────────────────────→ 步数
                    ^
                    这里突然崩了!
```

**为什么会发生？**

- **极端数据**：某些 batch 中的数据分布异常（比如全是乱码或极端长度）
- **数值溢出**：某些中间计算结果超出 FP16/BF16 能表示的范围
- **注意力分数爆炸**：在注意力计算中，QK 点积可能变得极大
- **学习率过大**：在训练的某个阶段，当前学习率对当前"地形"来说太大了

**解决方案**：

```
方案 1: Checkpoint 回退 + 跳过问题数据
  └── 回到 spike 之前的 checkpoint
  └── 跳过导致问题的 batch
  └── 继续训练

方案 2: QK-Norm / QK-Clip
  └── 对注意力的 Query 和 Key 做归一化
  └── 防止 QK 点积变得过大
  └── DeepSeek-V3、GLM-5 等都采用了这个技巧

方案 3: 更稳定的优化器
  └── MuonClip 自带梯度裁剪机制
  └── 比 AdamW 更不容易出现 spike
```

**Llama 3 的训练日志**：Meta 在训练 Llama 3 405B 时记录了 **466 次训练中断**。每次中断都需要从 checkpoint 恢复。这就是为什么大模型训练是一项"工程"而不仅仅是"科学"。

---

### 3. 过拟合 vs 欠拟合

**过拟合（Overfitting）**："背答案"

```
训练集 Loss: 0.5 (很低，好像学得很好)
测试集 Loss: 3.0 (很高，实际表现很差)

类比：学生把所有练习题的答案都背下来了，但考试换一道新题就不会了。
```

**欠拟合（Underfitting）**："没学会"

```
训练集 Loss: 4.0 (还很高)
测试集 Loss: 4.2 (也很高)

类比：学生上课没认真听，练习题和考试都考不好。
```

**大模型时代的好消息**：

过拟合在大模型训练中**不是主要问题**！原因：
- 数据量足够大（万亿级 token），模型很难"背下来"
- 模型只看数据一遍（或最多几遍），不会反复刷同一道题
- 真正需要担心的是**欠拟合**：数据质量不够好、模型架构有缺陷、训练不充分

---

### 4. MoE 负载不均衡

MoE（Mixture of Experts，混合专家）架构中，每个 token 只激活少量"专家"。但如果所有 token 都去找同一个"热门专家"呢？

```
理想情况（负载均衡）:          实际可能（负载不均）:

专家1: ████████ (25%)          专家1: ██████████████████ (80%)  ← 累死!
专家2: ████████ (25%)          专家2: ██ (8%)
专家3: ████████ (25%)          专家3: █ (5%)
专家4: ████████ (25%)          专家4: █ (7%)                   ← 闲着!
```

**后果**：热门专家过载，其他专家浪费，模型容量没有充分利用。

**解决方案**：

| 方案 | 思路 | 缺点 |
|------|------|------|
| 辅助损失（传统方法） | 添加额外的损失项惩罚不均衡 | 辅助损失和主损失冲突，降低模型质量 |
| 动态 bias（DeepSeek-V3 创新） | 给冷门专家加分，热门专家减分 | 无明显缺点，已成为新标准 |

DeepSeek-V3 的动态 bias 方案类比：
想象一个食堂有 256 个窗口。如果某个窗口排队太长，就在其他窗口贴"优惠券"吸引人分流。这个优惠力度（bias）会根据实时排队情况自动调整。

---

### 5. Checkpoint 与恢复 ★

**Checkpoint = 训练的"存档点"**。

```
训练过程:
  步骤 0    步骤 1000   步骤 2000   步骤 3000   ...
  │           │            │            │
  ▼           ▼            ▼            ▼
  [保存]     [保存]       [保存]       [保存]     ← 每 1000 步存一次

每次保存的内容:
  ├── 所有参数的当前值
  ├── 优化器状态（Adam 的动量等）
  ├── 学习率调度器的状态
  ├── 当前步数
  └── 数据读取的位置（读到哪了）
```

**为什么重要？**

大模型训练需要数周甚至数月，期间各种意外都可能发生：
- GPU 硬件故障
- 网络通信错误
- 电力中断
- Loss spike 需要回退

没有 checkpoint，一次故障就得从头开始训练，浪费数百万美元的算力。

**一些数字**：
- Llama 3 405B: 记录了 466 次训练中断
- 一个 70B 模型的 checkpoint: 约 140-560 GB（取决于精度和是否包含优化器状态）
- 保存间隔: 通常每 500-2000 步保存一次

## Part D: 后训练 — 教模型"听话" ★★

### 1. 为什么需要后训练

预训练完的模型只学会了"续写"——给任何文字，它都会继续写下去。但它不会"对话"。

```
你输入: "中国的首都是哪里？"

预训练模型可能回答:
  "这是一道小学地理题，考试中常考。以下是相关练习题：
   1. 日本的首都是？ 2. 法国的首都是？..."
  → 它在"续写"一份练习题，而不是回答你的问题！

你真正想要的回答:
  "中国的首都是北京。"
  → 简洁、直接、有帮助
```

后训练的目标就是把一个"续写机器"变成一个"聊天助手"。

```
整个训练流程:

预训练 (Pre-training)     → 后训练 (Post-training)
━━━━━━━━━━━━━━━━━━━━━     ━━━━━━━━━━━━━━━━━━━━━━
                          ┌──────────────────────┐
万亿级 token              │  SFT     RL (RLHF/   │
的互联网文本   ───────→   │ (微调) → DPO/GRPO)   │
                          └──────────────────────┘
学到：语言知识             学到：对话格式、人类偏好
产出：续写机器             产出：聊天助手
成本：95% 算力             成本：5% 算力
```

---

### 2. SFT (Supervised Fine-Tuning, 监督微调) ★

**做什么**：用人类标注的高质量对话数据微调模型。

**数据格式**：

```json
{
  "messages": [
    {"role": "user", "content": "解释一下什么是光合作用？"},
    {"role": "assistant", "content": "光合作用是植物利用阳光将二氧化碳和水转化为葡萄糖和氧气的过程..."}
  ]
}
```

**关键数字**：
- 数据量：几千到几万条高质量对话（远少于预训练的万亿 token）
- 训练时间：几小时到几天（远短于预训练的数月）
- 学习率：比预训练小约 10 倍（如 1e-5 到 2e-5，防止"忘记"预训练学到的知识）

**效果**：

```
SFT 前 (预训练模型):
  用户: 帮我写一首关于春天的诗
  模型: 关于春天的诗歌有很多，唐代诗人白居易曾写过..."
        → 在讲"关于诗"，而不是在写诗

SFT 后:
  用户: 帮我写一首关于春天的诗
  模型: 春风拂面暖阳照，
        万物复苏嫩芽冒。
        桃花朵朵映碧水，
        燕子归来筑新巢。
        → 直接写诗，这就是 SFT 教会的"对话格式"
```

**SFT 的局限**：SFT 教会了模型"如何回答"，但没有教会它"什么是好的回答"。模型可能给出格式正确但质量平庸的回答。这就需要下一步——强化学习。

---

### 3. RLHF (基于人类反馈的强化学习) ★★

RLHF = Reinforcement Learning from Human Feedback

**核心思想**：让人类告诉模型什么回答更好，然后训练模型生成人类更喜欢的回答。

**三个步骤**：

```
步骤 1: 收集人类偏好数据
━━━━━━━━━━━━━━━━━━━━━━

给同一个问题，模型生成两个回答，人类标注员选更好的：

问题: "如何学好数学？"

回答A: "多做题就行了。"
回答B: "建议从基础概念入手，先理解定义和定理的含义，
        然后通过适量练习巩固，遇到不会的题目要及时
        分析错因，而不是单纯刷题。"

人类选择: B > A  ✓

收集数千到数万这样的偏好对
```

```
步骤 2: 训练奖励模型 (Reward Model, RM)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

用人类的偏好数据训练一个"打分器"：

输入: 一个问题 + 一个回答
输出: 一个分数（越高越好）

训练目标: 让人类偏好的回答得分更高
  RM(问题, 回答B) > RM(问题, 回答A)

这个奖励模型学会了"人类觉得什么是好回答"
```

```
步骤 3: 用 PPO 优化模型
━━━━━━━━━━━━━━━━━━━━━━

PPO (Proximal Policy Optimization) 是一种强化学习算法：

循环:
  1. 模型生成回答
  2. 奖励模型打分
  3. 如果分数高 → 强化这种回答方式
     如果分数低 → 削弱这种回答方式
  4. 重复

  同时加一个约束：不能偏离 SFT 模型太远（防止模型"走歪"）
```

**直觉类比**：

```
驯动物:
  狗做出"握手"动作 → 给零食（奖励）→ 狗学会更多"握手"
  狗做出"乱咬"动作 → 不给零食（惩罚）→ 狗减少"乱咬"

RLHF:
  模型给出详细有帮助的回答 → 奖励模型打高分 → 模型学会更多这类回答
  模型给出敷衍或有害的回答 → 奖励模型打低分 → 模型减少这类回答
```

---

### 4. DPO (Direct Preference Optimization, 直接偏好优化)

**DPO 是 RLHF 的简化版本**。

```
RLHF 的流程（复杂）:
  人类偏好数据 → 训练奖励模型 → 用 PPO 优化模型
  需要: 3 个模型（主模型、奖励模型、参考模型）
  问题: PPO 训练不稳定，超参数多，工程复杂

DPO 的流程（简洁）:
  人类偏好数据 → 直接优化模型
  需要: 2 个模型（主模型、参考模型）
  优点: 更简单、更稳定、效果接近 RLHF
```

**DPO 的核心思想**：

数学上可以证明，RLHF 的优化目标可以被改写成一个不需要奖励模型的形式。直接用偏好数据优化：
- **增大**模型给"好回答"的概率
- **减小**模型给"坏回答"的概率

**人话公式**：

```
目标: 让 log(模型给好回答的概率) - log(模型给坏回答的概率) 越大越好
      同时不能偏离原始模型太远
```

---

### 5. GRPO — 推理能力的 RL ★★

GRPO = Group Relative Policy Optimization，由 DeepSeek-R1 首创。

**为什么需要 GRPO？**

RLHF/DPO 适合"偏好对齐"（哪个回答更好），但对于**数学、代码这类有明确正确答案的任务**，我们可以做得更好。

```
传统 RLHF:
  "1+1=?"
  回答A: "1+1=2"          人类标注: A 更好
  回答B: "1+1=3"          这需要人来标注!

GRPO:
  "1+1=?"
  回答A: "1+1=2"  → 自动验证: 正确! → 奖励 +1
  回答B: "1+1=3"  → 自动验证: 错误! → 奖励 -1
  不需要人类标注，也不需要奖励模型!
```

**GRPO 的完整流程**：

```
步骤 1: 给模型一个问题（如数学题）

步骤 2: 模型同时生成一组回答（如 16 个）
  回答1: "先算...结果是 42"  ← 正确
  回答2: "直接得到 35"       ← 错误
  回答3: "分步计算...是 42"  ← 正确
  ...
  回答16: "答案是 40"        ← 错误

步骤 3: 用规则自动判断对错（可验证奖励, Verifiable Reward）
  对数学题：答案 = 42？
  对代码题：运行代码，测试通过？

步骤 4: 在这一组内计算相对优势
  正确的回答 → 正奖励（相对于组内平均）
  错误的回答 → 负奖励（相对于组内平均）

步骤 5: 更新模型，强化正确的推理路径
```

**GRPO 的关键创新**：
- **无需奖励模型**：用规则验证代替学习型奖励模型
- **无需 Critic 模型**：用组内相对排名代替价值网络
- **简化训练**：比 PPO 简单很多，效果更好

**GRPO 催生了"推理模型"**：DeepSeek-R1 使用 GRPO 训练后，模型学会了"自己思考"——在回答前先进行长链推理（Chain of Thought），数学能力大幅提升。

                    模型训练全流程

阶段 1: 预训练 (Pre-training)

## Part E: 推理 — 模型怎么生成回答 ★★★

训练好了，模型是怎么一个字一个字吐出回答的？

### 1. 自回归生成 (Autoregressive Generation) ★★

**核心机制**：一个 token 一个 token 地生成，每次用前面**所有 token** 作为输入，预测下一个。

```
用户输入: "你好，请介绍一下"
（这是 Prompt，模型先处理这些 token）

生成过程:

第 1 步: 输入 = "你好，请介绍一下"
        → 模型预测下一个 token → "自"

第 2 步: 输入 = "你好，请介绍一下自"
        → 模型预测下一个 token → "己"

第 3 步: 输入 = "你好，请介绍一下自己"
        → 模型预测下一个 token → "。"

第 4 步: 输入 = "你好，请介绍一下自己。"
        → 模型预测下一个 token → "我"

第 5 步: 输入 = "你好，请介绍一下自己。我"
        → 模型预测下一个 token → "是"

... 继续 ...

第 N 步: → 模型预测 → <EOS>（结束标记）→ 停止生成
```

**关键点**：
- 生成 100 个 token = 做 **100 次**前向传播
- 每一步都依赖前面所有步的结果 → **无法并行**，必须串行生成
- 这就是为什么推理比训练慢得多的根本原因

---

### 2. 采样策略 — 下一个词怎么选

模型每一步输出的是一个**概率分布**（词表中每个 token 的概率），如何从中选出下一个 token？

```
模型输出: "天气" 后面的概率分布:
  "真好" → 0.35
  "不错" → 0.25
  "很好" → 0.15
  "晴朗" → 0.10
  "糟糕" → 0.05
  ...（几万个词，概率越来越小）
```

**常见策略**：

| 策略 | 做法 | 效果 |
|------|------|------|
| Greedy（贪心） | 每次选概率最高的 | 确定性，但可能无聊重复 |
| Temperature | 调整概率分布的"尖锐度" | T=0 等同贪心，T=1 正常，T>1 更随机 |
| Top-K | 只从概率最高的 K 个中选 | 避免选到极低概率的离谱词 |
| Top-P (Nucleus) | 选概率累加到 P 的最小词集 | 更灵活的截断方式 |

**Temperature 直觉**：

```
Temperature = 0.1 (很低, 确定性强):
  "真好" → 0.95    几乎一定选"真好"
  "不错" → 0.04
  "很好" → 0.01

Temperature = 1.0 (正常):
  "真好" → 0.35    按原始概率选
  "不错" → 0.25
  "很好" → 0.15

Temperature = 2.0 (高, 很随机):
  "真好" → 0.22    各选项概率更接近
  "不错" → 0.20    什么都可能被选到
  "很好" → 0.16
  "糟糕" → 0.12
```

类比：Temperature 就像一个"创造力旋钮"。写代码时调低（要确定性），写诗时调高（要创造性）。

---

### 3. KV Cache — 核心优化 ★★

**问题**：自回归生成中，每生成一个新 token 都要"看"前面所有 token。如果不做优化，越到后面计算量越大。

```
无优化的情况（每步重新算所有）:
  第 1 步: 处理 1 个 token 的注意力    计算量: 1
  第 2 步: 处理 2 个 token 的注意力    计算量: 2
  第 3 步: 处理 3 个 token 的注意力    计算量: 3
  ...
  第 1000 步: 处理 1000 个 token       计算量: 1000

  总计算量: 1 + 2 + 3 + ... + 1000 = 500,500
```

**KV Cache 的核心思想**：前面 token 的 K（Key）和 V（Value）已经算过了，不需要重复计算，缓存下来复用。

```
有 KV Cache 的情况:
  第 1 步: 算 token 1 的 K,V → 缓存 [K1, V1]
  第 2 步: 算 token 2 的 K,V → 缓存 [K1,K2, V1,V2]
           新 token 的 Q 只需要和缓存的 K 做注意力
  第 3 步: 算 token 3 的 K,V → 缓存 [K1,K2,K3, V1,V2,V3]
           只需计算新 token 的注意力
  ...
  第 1000 步: 只算 1 个新 token 的 Q/K/V
              然后和缓存的 999 个 K,V 做注意力

  每步计算量: 基本恒定!
```

```
对比图:

无 Cache:                      有 Cache:
┌─────────┐                    ┌─────────┐
│ 步骤  计算│                    │ 步骤  计算│
│  1    █  │                    │  1    █  │
│  2    ██ │                    │  2    █  │
│  3    ███│                    │  3    █  │
│  ...     │                    │  ...     │
│  N   ████████│               │  N    █  │
└─────────┘                    └─────────┘
每步递增 O(N)                   每步近似恒定 O(1)*

*注: 内存消耗仍然线性增长，但计算量大幅减少
```

**代价**：KV Cache 需要大量内存。

```
一个 70B 模型, 128K 上下文:
  KV Cache 大小 ≈ 2 × 层数 × 隐藏维度 × 序列长度 × 精度
              ≈ 2 × 80 × 8192 × 128000 × 2 bytes (FP16)
              ≈ 约 320 GB  (比模型本身还大!)
```

这就是为什么 GQA（Grouped Query Attention）和 MLA（Multi-Head Latent Attention）这类注意力优化如此重要——它们可以把 KV Cache 压缩 4-16 倍。

### 4. Prefill vs Decode — 两个阶段 ★★

推理过程分为两个截然不同的阶段：

```
┌──────────────────────────────────────────────────────────────┐
│                      推理的两个阶段                           │
├────────────────────────┬─────────────────────────────────────┤
│   Prefill（填充阶段）   │   Decode（解码阶段）                │
├────────────────────────┼─────────────────────────────────────┤
│ 做什么:                 │ 做什么:                              │
│  处理用户的全部输入      │  一个一个生成回答的 token            │
│                         │                                     │
│ 特点:                   │ 特点:                                │
│  所有输入 token 可以     │  每次只处理 1 个新 token             │
│  一起并行处理            │  必须等前一个生成完                  │
│                         │                                     │
│ 瓶颈:                   │ 瓶颈:                                │
│  GPU 算力               │  GPU 内存带宽                       │
│  (大量矩阵乘法)         │  (每步要读全部参数 + KV Cache)       │
│                         │                                     │
│ 时间感受:               │ 时间感受:                             │
│  "等一下..." (首字延迟)  │  "一个字一个字蹦出来" (逐字输出)    │
│                         │                                     │
│ 优化目标:               │ 优化目标:                             │
│  减少首字延迟 (TTFT)     │  提高每秒生成 token 数 (TPS)        │
└────────────────────────┴─────────────────────────────────────┘
```

**为什么 Decode 阶段是"内存带宽受限"？**

```
生成 1 个 token 需要:
  1. 从 GPU 显存读取所有模型参数 (~140GB for 70B, FP16)
  2. 从 GPU 显存读取 KV Cache
  3. 做很少量的计算（只有 1 个 token 的矩阵乘法）

H100 GPU 的带宽: 3.35 TB/s
读 140GB 参数: 140 / 3350 ≈ 0.042 秒 ≈ 42ms

意味着: 70B 模型理论上每秒最多生成约 24 个 token (单卡)
瓶颈完全在"读数据"而不是"算数据"!
```

这就像你有一个超级快的计算器，但每次计算前都需要从一本超厚的参考书里查数据。翻书的时间远超计算时间。

### 5. 为什么推理这么慢 & 怎么加速

**瓶颈分析**：

```
生成每个 token 的耗时:
  ├── 读取模型参数:     ████████████████████ (80%)  ← 主要瓶颈
  ├── 读取 KV Cache:    ████████ (15%)
  └── 实际计算:         █ (5%)

大部分时间花在"搬运数据"而不是"计算"!
```

**加速方案汇总**：

| 方法 | 思路 | 效果 | 代价 |
|------|------|------|------|
| **量化** | 降低精度 (FP16 → INT4) | 速度 2-4x，内存减半到 1/4 | 质量微降 |
| **KV Cache 压缩** | GQA/MLA/PagedAttention | 内存减 2-16x | 工程复杂度 |
| **推测解码** | 小模型猜 + 大模型验 | 速度 2-3x | 需要额外小模型 |
| **连续批处理** | 动态合并多个请求 | 吞吐提升数倍 | 略增延迟 |
| **模型蒸馏** | 大模型教小模型 | 小模型接近大模型效果 | 需要训练 |

### 6. 推测解码 (Speculative Decoding) ★

这是一个非常巧妙的加速技术。

**核心思想**：

```
正常解码（慢）:
  大模型生成 token 1 → 大模型生成 token 2 → ... → 大模型生成 token K
  每步都要读一遍大模型全部参数，串行执行

推测解码（快）:
  步骤 1: 小模型快速猜 K 个 token （非常快，因为小模型参数少）
  步骤 2: 大模型一次性验证这 K 个 token （只需一次前向传播！）
  步骤 3: 如果全猜对 → 相当于 K 倍加速
          如果猜错 → 从第一个错误位置重新开始
```

**具体例子**：

```
问题: "法国的首都是什么？"

小模型快速猜 5 个 token: "法国 首都 是 巴黎 。"

大模型一次性验证:
  "法国" ← 对!
  "首都" ← 对!
  "是"   ← 对!
  "巴黎" ← 对!
  "。"   ← 对!

全部正确! 一次前向传播生成了 5 个 token!
相当于 5 倍加速!
```

```
另一种情况:

小模型快速猜 5 个 token: "法国 首都 是 里昂 。"

大模型验证:
  "法国" ← 对!
  "首都" ← 对!
  "是"   ← 对!
  "里昂" ← 错! 应该是"巴黎"

接受前 3 个 + 大模型自己生成第 4 个 ("巴黎")
一次前向传播生成了 4 个 token! 还是比逐个快!
```

**关键定理**：推测解码在数学上可以证明**质量完全不变**——最终输出的分布和大模型独自生成的分布完全一样。加速是"免费的"（不牺牲质量）。

**实际加速效果**取决于小模型猜测的准确率。简单任务（如翻译、续写）加速明显，复杂推理任务加速较少。

### 7. 量化 — 压缩模型 ★

量化（Quantization）是让模型在更少的内存中运行的核心技术。

**基本原理**：

```
原始 (FP16, 16位):
  一个参数: 0.123456789...  占 2 字节，精度高

量化到 INT8 (8位):
  一个参数: 0.123           占 1 字节，精度略降
  内存减半!

量化到 INT4 (4位):
  一个参数: 0.12            占 0.5 字节，精度再降
  内存降到 1/4!
```

**常见量化方案**：

| 量化方案 | 精度 | 7B 模型大小 | 质量损失 | 适用场景 |
|---------|------|------------|---------|---------|
| FP16 | 16-bit | ~14 GB | 无 | GPU 训练和推理 |
| INT8 | 8-bit | ~7 GB | 极小 | GPU 推理，速度快 |
| INT4 (GPTQ/AWQ) | 4-bit | ~4 GB | 小 | 消费级 GPU (如 RTX 4090) |
| Q4_K_M (GGUF) | ~4.5-bit | ~5 GB | 小 | Ollama / llama.cpp (CPU + GPU) |
| Q2_K (GGUF) | ~2.5-bit | ~3 GB | 明显 | 极端内存限制 |

**实际体验**：

```
Llama 3 8B 在不同量化下:

FP16 (14GB):  需要 16GB+ 显存的 GPU → 质量 100%
INT8 (7GB):   普通 GPU 可以跑       → 质量约 99%
INT4 (4GB):   RTX 3060 就能跑       → 质量约 96%
Q4_K_M (5GB): Ollama 一键部署       → 质量约 97%
```

**量化的两大流派**：

```
训练后量化 (Post-Training Quantization, PTQ):
  训练完的模型 → 直接转换精度
  方法: GPTQ, AWQ, GGUF
  优点: 简单，不需要重新训练
  缺点: 极低精度时质量下降明显

量化感知训练 (Quantization-Aware Training, QAT):
  在训练时就模拟低精度
  优点: 质量更好
  缺点: 需要重新训练
```

## Part F: 模型家族是怎么来的 ★★

> Qwen3 一出就是 0.6B / 1.7B / 4B / 8B / 14B / 32B / 235B 七个尺寸，它们是怎么产出的？

### 1. 架构设计 — 定好每个尺寸的"骨架"

同一个家族的模型共享相同的架构设计理念（比如都用 GQA、SwiGLU、RoPE），但每个尺寸的具体参数不同：

| 参数 | 小模型 (0.6B) | 中模型 (8B) | 大模型 (72B) |
|------|-------------|------------|-------------|
| 层数 | ~28 | ~32 | ~80 |
| 隐藏维度 | ~1024 | ~4096 | ~8192 |
| 注意力头数 | ~16 | ~32 | ~64 |
| FFN 维度 | ~2816 | ~14336 | ~28672 |

怎么决定这些数字？用 **Scaling Laws** 指导：

```
给定目标参数量（比如 8B），Scaling Laws 告诉你：
  - 层数和宽度怎么分配最优
  - 需要多少训练数据
  - 需要多少算力
```

### 2. 独立预训练 — 各自从零开始

**最关键的事实：每个尺寸是独立训练的，不是从大模型"砍"出来的。**

```
                    共享部分                    独立部分
                 ┌──────────┐
                 │ 同一个    │
                 │ 分词器    │──→ 0.6B ──→ 在 ~3T tokens 上训练 ──→ 完成
                 │ (词表)    │
                 │          │──→ 8B   ──→ 在 ~15T tokens 上训练 ──→ 完成
                 │ 同一套    │
                 │ 数据管线  │──→ 32B  ──→ 在 ~15T tokens 上训练 ──→ 完成
                 │          │
                 │ 同一套    │──→ 72B  ──→ 在 ~15T tokens 上训练 ──→ 完成
                 │ 清洗流程  │
                 └──────────┘
                                各自独立的随机初始化
                                各自独立的训练过程
                                可以同时在不同 GPU 集群上并行
```

**共享的是**：
- 分词器（所有尺寸用同一个词表，比如 128K tokens）
- 预训练数据（同一份清洗好的数据）
- 训练方法论（同一套优化器配置、学习率策略）

**不共享的是**：
- 模型权重（各自从随机数开始）
- GPU 资源（大模型用更多卡，小模型用更少）
- 训练时长（大模型训更久）

### 3. 知识蒸馏 — 大模型教小模型 ★★

除了独立训练，还有一种方式：**让训好的大模型当"老师"**。

```
蒸馏流程：

  大模型（老师）                      小模型（学生）
  ┌─────────────┐                   ┌─────────────┐
  │ Qwen3-235B  │                   │ Qwen3-8B    │
  │ 或           │                   │ 已完成预训练  │
  │ DeepSeek-R1 │                   │             │
  └──────┬──────┘                   └──────┬──────┘
         │                                 │
         │  第一步：大模型生成数据             │
         │  "请解决这道数学题"                │
         │  → 大模型输出详细解题过程            │
         │                                 │
         │  收集 10 万 ~ 100 万条              │
         │  高质量推理数据                    │
         │                                 │
         └────────────→ 第二步：用这些数据   │
                        微调(SFT)小模型 ────┘
                              │
                              ↓
                        小模型获得了大模型
                        的部分推理能力
```

**最成功的蒸馏案例 — DeepSeek-R1**：

| 模型 | 做法 | MATH 分数 | 对比 |
|------|------|----------|------|
| DeepSeek-R1 671B | 完整 RL 训练 | 97.3% | 老师 |
| Qwen2.5-32B（原版） | 标准训练 | ~70% | 蒸馏前 |
| DeepSeek-R1-Distill-Qwen-32B | 用 R1 数据蒸馏 | 94.3% | **蒸馏后** |
| DeepSeek-R1-Distill-Qwen-7B | 用 R1 数据蒸馏 | 83.9% | 7B 也能推理 |

> 蒸馏的本质：**小模型学不会的东西，让大模型"示范"给它看，再用示范数据训练**。

### 4. 后训练 — 共享对齐流程

预训练完成后，所有尺寸通常使用同一套后训练流程：

```
所有尺寸的 Base 模型
    │
    ├──→ 同一套 SFT 数据微调 ──→ Instruct 版本（能对话）
    │
    ├──→ 同一套 RLHF/DPO 对齐 ──→ 质量提升
    │
    └──→ 大模型蒸馏数据 + GRPO ──→ Think 版本（能推理）
```

发布时每个尺寸通常有多个变体：

| 变体 | 含义 | 适用场景 |
|------|------|---------|
| **Base** | 只做了预训练 | 研究、继续训练 |
| **Instruct** | 预训练 + SFT + RLHF | 日常对话、任务 |
| **Think** | Instruct + 推理 RL | 数学、编程、复杂推理 |
| **Chat** | Instruct 的别名 | 同 Instruct |

### 5. 量化发布 — 同一个模型压缩成不同大小

这不是"训练"，是发布后的压缩：

```
Qwen3-8B-Instruct (FP16, ~16GB)
    │
    ├── GPTQ INT8  → ~8GB    （GPU 推理，几乎无损）
    ├── GPTQ INT4  → ~4GB    （GPU 推理，略有损失）
    ├── AWQ INT4   → ~4GB    （GPU 推理，优化版）
    ├── GGUF Q4_K_M → ~5GB   （CPU/GPU 推理，Ollama 用这个）
    └── GGUF Q2_K  → ~3GB    （极限压缩，质量明显下降）
```

### 6. 完整产出流程图

```
┌─────────────────────────────────────────────────────┐
│               模型家族产出全流程                       │
│                                                     │
│  设计阶段：                                          │
│    Scaling Laws → 确定 0.6B/8B/32B/72B 架构参数       │
│    设计共享分词器（128K 词表）                         │
│                                                     │
│  预训练阶段（并行进行）：                              │
│    集群 A (64 GPU)   → 训练 0.6B  ─┐                │
│    集群 B (256 GPU)  → 训练 8B   ──┤                │
│    集群 C (1024 GPU) → 训练 32B  ──┤ 各自独立        │
│    集群 D (4096 GPU) → 训练 72B  ──┘                │
│                                                     │
│  蒸馏阶段（可选）：                                   │
│    72B 生成推理数据 → 蒸馏到 8B/32B                   │
│                                                     │
│  后训练阶段（共享流程）：                              │
│    所有尺寸 → SFT → RLHF/DPO → 产出 Instruct 版     │
│    部分尺寸 → GRPO/推理 RL → 产出 Think 版           │
│                                                     │
│  发布阶段：                                          │
│    每个尺寸 × 每个变体 × 多种量化                     │
│    0.6B-Instruct, 8B-Instruct, 8B-Think, ...        │
│    社区量化：GGUF, GPTQ, AWQ                         │
│    集成到 Ollama, LM Studio, vLLM                   │
└─────────────────────────────────────────────────────┘
```

### 核心要点

- **大小模型是独立训练的**，不是从大模型"裁剪"出来的
- 共享的是数据管线、分词器和训练方法论
- **蒸馏**是让小模型获得大模型能力的捷径（尤其是推理能力）
- 量化是部署时的压缩手段，不涉及训练
- 一个家族发布的模型数 = 尺寸数 × 变体数 × 量化数，可以非常多
