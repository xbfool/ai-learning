## 深入解析：华为昇腾 910B 训练

### 昇腾 910B 芯片概述

华为昇腾 (Ascend) 910B 是由华为海思 (HiSilicon) 设计、中芯国际 (SMIC) 使用 N+1 工艺（相当于 7nm）制造的 AI 训练芯片。它是当前中国国产 AI 芯片中性能最强的训练级芯片之一，也是美国出口管制背景下中国 AI 算力自主化的核心产品。

**核心规格：**

| 指标 | 昇腾 910B | NVIDIA H100 | NVIDIA A100 |
|------|----------|-------------|-------------|
| FP16 算力 | ~320 TFLOPS | ~989 TFLOPS (密集) | ~312 TFLOPS |
| BF16 算力 | ~256 TFLOPS | ~989 TFLOPS | ~312 TFLOPS |
| INT8 算力 | ~640 TOPS | ~1,979 TOPS | ~624 TOPS |
| 显存 | 64GB HBM2e | 80GB HBM3 | 80GB HBM2e |
| 显存带宽 | ~1,200 GB/s | ~3,350 GB/s | ~2,039 GB/s |
| TDP 功耗 | ~400W | ~700W | ~400W |
| 制程 | SMIC 7nm (N+1) | TSMC 4nm | TSMC 7nm |
| AI 核心 | 25 个 DaVinci 核 | 132 个 SM | 108 个 SM |
| 芯片面积 | ~665 mm² | ~814 mm² | ~826 mm² |

**关键分析：**

1. **算力差距**：昇腾 910B 的 FP16 算力约为 H100 的 1/3，但与 A100 基本持平（320 vs 312 TFLOPS）。这意味着用 3 张 910B 才能匹配 1 张 H100 的理论算力
2. **显存带宽瓶颈**：910B 的 HBM2e 带宽（~1,200 GB/s）远低于 H100 的 HBM3（~3,350 GB/s），在大模型训练中显存带宽往往是关键瓶颈
3. **能效比优势**：在 400W TDP 下实现 ~1.5 TFLOPS/W，略优于 H100 的 ~1.4 TFLOPS/W，但这是因为绝对算力更低
4. **DaVinci 架构**：每个 DaVinci AI 核包含 1 个矩阵执行单元和 2 个向量执行单元（比前代多 1 个向量单元），针对矩阵乘法和张量运算优化

### CANN 软件栈

CANN (Compute Architecture for Neural Networks) 是华为专为昇腾芯片开发的异构计算架构软件栈，相当于 NVIDIA 的 CUDA 生态。

**CANN 的组成：**
- **算子库**：提供常用神经网络算子的高性能实现
- **图编译器**：将计算图编译为昇腾芯片可执行的指令
- **运行时环境**：管理设备内存、任务调度、流控制等
- **调试工具**：性能分析、精度对比、日志诊断

**与 CUDA 的差距：**
- CUDA 经过 17 年以上的生态积累，拥有海量第三方库和社区支持
- CANN 的算子库优化覆盖面不够全面，许多非标准算子需要手动适配
- 开发者普遍反映 CANN 的调试体验不如 CUDA 成熟，存在"难用且不稳定"的问题
- Model FLOPS Utilization (MFU) 的差距不仅来自硬件，也来自 CANN 内核执行效率的差距

**MindSpore 框架：**

GLM-5 使用的是华为的 MindSpore 深度学习框架，而非 PyTorch。MindSpore 专为昇腾硬件生态设计，提供：
- 编译器级别的算子融合和内存优化
- 针对昇腾互连 (HCCS) 的分布式训练原语
- 自动并行策略搜索
- 大规模集群的容错和检查点机制

### 非 NVIDIA 硬件训练的挑战

在 10 万张昇腾 910B 上完成 GLM-5 的训练，本质上是一个巨大的**系统集成工程挑战**：

1. **芯片互连瓶颈**：昇腾使用 HCCS (Huawei Cache Coherence System) 进行芯片间通信，带宽低于 NVIDIA 的 NVLink/NVSwitch，导致分布式训练中通信开销更大
2. **软件生态适配**：大量基于 CUDA/PyTorch 的训练代码需要重写或迁移到 CANN/MindSpore，华为为此派驻工程团队到百度、科大讯飞、腾讯等公司协助迁移
3. **故障率管理**：10 万张芯片同时运行时，单芯片故障率被放大，需要极强的容错机制和快速检查点恢复能力。据报道，昇腾 910C 的良率仅约 20%（行业标准通常超过 70%），910B 的情况类似
4. **MFU 差距**：由于 HCCS 带宽限制（通信时间更长）和 CANN 软件栈优化水平较低（内核执行效率差），实际训练中的 MFU 通常低于同等规模的 NVIDIA 集群
5. **全栈适配**：GLM-5 实现了 7 个国产芯片平台的深度适配，包括华为昇腾、摩尔线程、海光、寒武纪、昆仑芯、沐曦、燧原，展示了国产 AI 算力生态的多元化

## 深入解析：28.5T Token 预训练数据

### 数据规模对比

GLM-5 的 28.5T token 预训练数据量是目前公开报告中最大的之一：

| 模型 | 预训练数据量 | 发布时间 |
|------|------------|---------|
| **GLM-5** | **28.5T tokens** | 2026.02 |
| Qwen 3 (235B) | 36T tokens | 2025 |
| GLM-4.5 | 23T tokens | 2025 |
| Kimi K2 | 15.5T tokens | 2025 |
| Llama 3 (405B) | 15T tokens | 2024 |
| DeepSeek V3 | 14.8T tokens | 2024 |
| Llama 2 | 2T tokens | 2023 |

**趋势观察**：前沿模型的预训练数据量从 2023 年的 2T 级别快速增长到 2025-2026 年的 15-36T 级别。这一趋势被称为"过度训练" (overtraining)——即超出 Chinchilla 缩放定律建议的训练时长，以在固定参数量下获得更好的性能。

### 两阶段预训练策略

GLM-5 的预训练分为两个阶段：

**第一阶段：通用语言能力（~18T tokens）**
- 通用网络数据（经过严格去重和质量过滤）
- 书籍、学术论文、科学文献
- 多语言数据（中文、英文为主）
- 使用"世界知识分类器" (World Knowledge Classifier) 从中低质量数据中提取有价值信息——该分类器基于维基百科条目和 LLM 标注数据训练

**第二阶段：代码与推理能力（~9T tokens）**
- GitHub 代码数据（相比前代版本增加了 28% 的独特 token）
- 数学文本和推理数据
- 高质量指令数据

**关键设计原则：**
- **严格避免合成数据**：过滤流水线严格排除合成的、AI 生成的或模板化的数据，确保数据的真实性和多样性
- **代码数据占比提升**：第二阶段专注于代码和推理，这直接支撑了 GLM-5 在 SWE-bench 上的优异表现
- **质量优先于数量**：虽然数据量庞大，但通过多层过滤（去重、质量评估、知识分类）确保高信号比

### 数据处理流水线的工程挑战

处理 28.5T token 的数据本身就是一个巨大的工程挑战：
- 原始数据的存储和传输需要 PB 级基础设施
- 去重算法需要在全局范围内运行（跨数万亿 token 的近似去重）
- 数据质量过滤需要高效的分类器推理管线
- 数据混合比例的调优需要大量消融实验

## 深入解析：SWE-bench Verified 77.8%

### SWE-bench 是什么

SWE-bench 是一个评估 AI 模型解决真实软件工程问题的基准测试。它从真实的 GitHub 仓库中提取 Bug 修复任务，要求模型：

1. **理解大型代码库**：需要导航数千行甚至数万行代码
2. **定位问题**：从 Issue 描述出发，找到有问题的代码位置
3. **生成补丁**：编写正确的代码修改并确保测试通过
4. **跨文件推理**：通常需要修改多个文件、理解模块间依赖

SWE-bench Verified 是经过人工验证的子集，确保每个任务都有明确的解决方案且测试用例可靠。

### GLM-5 的成绩在行业中的定位

| 模型 | SWE-bench Verified |
|------|-------------------|
| Claude Opus 4.6 | 80.8% |
| GPT-5.2 | 80.0% |
| Claude Sonnet 4.6 | 79.6% |
| **GLM-5** | **77.8%** |
| DeepSeek V3.2-Speciale | 73.1% |
| 其他开源模型 | <70% |

GLM-5 是**开源模型中 SWE-bench Verified 分数最高的**，与顶级闭源模型的差距已经缩小到 3 个百分点以内。

### 什么造就了优秀的编码 Agent

GLM-5 在 SWE-bench 上的成功反映了其"从 Vibe Coding 到 Agentic Engineering"的理念转变：

**1. 强大的长上下文理解**
- 200K 输入上下文使模型能够一次性加载大量代码文件
- DSA 稀疏注意力确保长上下文下的计算效率

**2. 工具使用能力**
- SWE-bench 评估使用 OpenHands 框架，模型需要调用文件浏览、搜索、编辑等工具
- GLM-5 经过专门的 Agentic RL（智能体强化学习）训练，增强了多步骤工具调用的连贯性

**3. 三阶段强化学习流水线**
- **Reasoning RL**：增强推理能力
- **Agentic RL**：增强 Agent 工具使用和长期规划
- **General RL**：通用能力校准
- 使用 On-Policy Cross-Stage Distillation 防止灾难性遗忘

**4. 异步强化学习基础设施**
- GLM-5 提出了新的异步 RL 基础设施，将生成 (generation) 和训练 (training) 解耦
- 这使得模型能从复杂的长期交互 (long-horizon interactions) 中更有效地学习
- 对于 SWE-bench 这类需要多步推理和工具调用的任务尤为关键

### 实际使用中的注意事项

尽管 GLM-5 在 SWE-bench 上表现优异，独立测试揭示了基准分数与实际体验的差距：
- 在 Terminal-Bench 2.0 上得分 56.2%，显示出在终端操作场景中的局限
- 使用 Claude Code + GLM-5 时，约 27% 的任务超时（88 个任务中有 24 个）
- 30% 的 KIRO 试验产生完全无效输出
- 这提醒我们 SWE-bench 成绩并不能完全代表真实编程场景的表现

## 深入解析：DeepSeek 稀疏注意力 (DSA)

### 从 NSA 到 DSA

GLM-5 采用的是 DeepSeek Sparse Attention (DSA)，这是基于 DeepSeek 提出的 Native Sparse Attention (NSA) 发展而来的稀疏注意力机制。NSA 论文获得了 ACL 2025 最佳论文奖，是长上下文建模领域的重要突破。

### NSA 的核心设计：三分支架构

NSA 将传统的全注意力 (Full Attention) 替换为三个互补的注意力分支：

**1. 压缩注意力 (Compressed Attention)**
- 将连续的 token 块（块大小 = 32）压缩为单一的摘要表示
- 通过一个小型神经网络层将整个块的内容总结为压缩的 key 和 value
- 这意味着大量的全局上下文被保留在更少的高层级"token"中，大幅减少注意力计算量

**2. 选择注意力 (Selected Attention)**
- 识别出最相关的 token 块，保留其原始的、未压缩的表示
- 复用压缩步骤中的注意力分数来判断哪些块对当前 query 最重要
- 选择块大小 = 64，选择数量 = 16（包含固定激活的 1 个初始块和 2 个局部块）

**3. 滑动窗口注意力 (Sliding Window Attention)**
- 在当前位置附近取一个固定大小的窗口（窗口大小 = 512）
- 确保模型始终能捕获局部上下文信息
- 防止压缩和选择分支被短程模式所干扰，让三个分支各司其职

### DSA 在 GLM-5 中的实现

DSA 在 NSA 基础上进一步演进，引入了 **Lightning Indexer**：

**Lightning Indexer 的工作原理：**
1. 使用类似 Multi-Query Attention 的架构：多个 query head 共享一个 key
2. query 和 key 都被投影到低维空间（128 维）的 FP8 表示
3. 通过高效的批量矩阵乘法 (batched MatMul) 在 FP8 精度下计算近似注意力分数
4. 使用 GPU 的部分排序算法 (partial sort) 执行 top-k 选择
5. 对选中的 token 子集执行精确的注意力计算

**性能收益：**
- 将全注意力的 O(L²) 复杂度降低为接近 O(L) 的稀疏计算
- 在 64K 长度序列上，解码、前向传播和反向传播都实现了显著加速
- 使得 200K 输入上下文在 MoE 架构下具有实际可行性
- 训练和推理成本降低约 2 倍

**与 MLA 的配合：**
- GLM-5 同时采用了 Multi-head Latent Attention (MLA) 来压缩 KV 对到潜在空间
- DSA 在 MLA 的基础上进一步引入 indexer 机制，检索 top-k 最相关的 KV 条目
- 两者配合实现了 KV Cache 的高效管理和稀疏计算

**RL 训练中的挑战：**
- top-k 操作本身是确定性的不可微操作，在强化学习中可能影响梯度传播
- GLM-5 团队专门研究了 DSA indexer 中确定性 top-k 算子对 RL 稳定性和性能的影响

## 深入解析：中国 AI 主权的战略意义

### 地缘政治背景

自 2022 年美国对华实施芯片出口管制以来，NVIDIA 的高端 AI 芯片（A100、H100、H200、Blackwell 系列）被禁止出口到中国。这一禁令直接威胁了中国 AI 产业的算力供给。

**关键时间线：**
- 2022.10：美国首次实施对华 AI 芯片出口管制
- 2023.10：管制升级，封堵降速版芯片（如 H800、A800）
- 2024：中国开始大规模推广昇腾 910B 替代
- 2025：中国政府要求国资背景数据中心禁用外国 AI 芯片
- 2026.02：GLM-5 发布，证明 10 万张国产芯片可训练前沿模型

### NVIDIA 在中国市场的衰落

NVIDIA 在中国 AI 芯片市场的份额从 2024 年底的 66% 暴跌至 2026 年初的约 8%。中国政府禁止政府资助的数据中心使用外国 AI 芯片，并指示国资项目只使用国产芯片。

### GLM-5 的里程碑意义

GLM-5 的发布具有多重战略意义：

**1. 技术验证**
- 首次证明 10 万张国产芯片可以稳定完成前沿大模型的全流程训练
- 训练出的模型性能接近 GPT-5.2 级别（差距在个位数百分比内）
- 在 Humanity's Last Exam 上得分 50.4%，超过 Claude Opus 4.5

**2. 产业信心**
- GLM-5 发布后 24 小时内，智谱 AI 港股股价上涨 28.7%
- 向全行业传递了"国产芯片可用"的强烈信号
- 为更多中国 AI 公司迁移到国产算力提供了路径参考

**3. 生态建设**
- MindSpore + CANN + 昇腾的全栈组合逐渐成熟
- 华为向百度、科大讯飞、腾讯等公司派驻工程团队协助 CUDA 代码迁移
- GLM-5 适配了 7 个国产芯片平台，推动了多元化的国产 AI 芯片生态

**4. 技术自主可控**
- 从芯片设计（海思）、制造（中芯国际 7nm）、软件栈（CANN/MindSpore）到模型训练（GLM-5）的完整国产化链条
- 减少了对 NVIDIA/CUDA 生态的依赖，降低了供应链风险

### 仍存在的差距与挑战

尽管 GLM-5 是一个重要里程碑，差距仍然客观存在：

1. **单芯片性能差距**：910B 的 FP16 算力仅为 H100 的约 1/3，需要更多芯片来弥补
2. **良率问题**：SMIC 7nm 工艺的良率（~20%）远低于 TSMC（>70%），制造成本高
3. **软件生态差距**：CANN 的成熟度与 CUDA 17 年的积累相比仍有明显差距
4. **互连带宽**：HCCS 带宽低于 NVLink/NVSwitch，在超大规模集群中通信开销更大
5. **全球化壁垒**：技术生态的分裂可能导致两个平行的 AI 技术栈长期共存

### 对全球 AI 格局的影响

GLM-5 的成功标志着全球 AI 算力格局正在发生根本性转变：

- **技术脱钩加速**：中美 AI 技术栈正在走向分裂，形成"NVIDIA/CUDA 阵营"和"昇腾/CANN 阵营"两个平行生态
